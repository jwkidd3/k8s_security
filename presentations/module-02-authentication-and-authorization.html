🛡️ Section 1

Authentication Fundamentals

🔐 How the API Server Authenticates Requests

Every request to the API server passes through an authentication chain. The first authenticator to succeed determines the identity.

Client Request
➡
Authenticator 1
➡
Authenticator 2
➡
Authenticator N
➡
Authenticated Identity
Key concept: Kubernetes does not have a built-in user database. It relies on external identity sources (certificates, tokens, OIDC providers).
Emphasize that Kubernetes has no "user" object in the API. Users are represented by strings extracted from authentication credentials. Multiple authenticators run in parallel; the first to succeed wins.

🔑 Authentication Methods

Method Credential Source Use Case
X.509 Client Certificates TLS handshake Cluster components, admin access
Bearer Tokens Authorization header Service accounts, static tokens
OIDC Tokens JWT in Authorization header Human users via IdP
Webhook Token Auth External webhook Custom authentication backends
Bootstrap Tokens Secret in kube-system Node bootstrapping (kubeadm)
Anonymous Requests No credentials Health checks (restricted)

📜 X.509 Client Certificates

The API server validates client certificates against the cluster CA specified by --client-ca-file.

Subject mapping:
  • CN (Common Name) → Username
  • O (Organization) → Group(s)
Caution: Client certificates cannot be revoked without rotating the CA. Keep certificate lifetimes short.

Generating a Client Certificate

# Generate private key
openssl genrsa -out admin.key 2048

# Create CSR with CN=admin, O=system:masters
openssl req -new -key admin.key \
  -subj "/CN=admin/O=system:masters" \
  -out admin.csr

# Sign with cluster CA
openssl x509 -req -in admin.csr \
  -CA /etc/kubernetes/pki/ca.crt \
  -CAkey /etc/kubernetes/pki/ca.key \
  -CAcreateserial \
  -out admin.crt -days 365
Stress that system:masters bypasses RBAC entirely. Never use it for regular operations. Show how the O field maps to groups, which is critical for RBAC bindings.

🎫 Bearer Tokens & OIDC Tokens

Static Token File

Configured via --token-auth-file. CSV format:

token,user,uid,"group1,group2"
Avoid in production: Static tokens are stored in plaintext, cannot be rotated without restart, and are deprecated.

OIDC Tokens (Recommended)

JWT tokens issued by an identity provider (e.g., Keycloak, Azure AD, Okta).

  • Short-lived tokens with automatic refresh
  • Group claims for RBAC mapping
  • Centralized user management
  • MFA support through the IdP

🌐 Webhook Token Authentication

The API server sends a TokenReview request to an external webhook service for validation.

Client + Token
➡
API Server
➡
Webhook Endpoint
➡
TokenReview Response
# Webhook config file (--authentication-token-webhook-config-file)
apiVersion: v1
kind: Config
clusters:
  - name: authn-webhook
    cluster:
      server: https://authn.example.com/authenticate
      certificate-authority: /etc/kubernetes/pki/webhook-ca.crt
users:
  - name: kube-apiserver
    user:
      client-certificate: /etc/kubernetes/pki/apiserver-webhook.crt
      client-key: /etc/kubernetes/pki/apiserver-webhook.key
Webhook authentication is commonly used with cloud provider IAM (e.g., AWS IAM Authenticator for EKS). The webhook must return a TokenReview response indicating whether the token is valid and the associated user info.

🚫 Anonymous Requests & Bootstrap Tokens

Anonymous Requests

When no credentials are provided, the request is assigned:

  • Username: system:anonymous
  • Group: system:unauthenticated

Controlled by --anonymous-auth (default: true).

Security: Never grant significant permissions to system:unauthenticated. Restrict to health endpoints only.

Bootstrap Tokens

Used during cluster initialization for node joining.

  • Format: [a-z0-9]{6}.[a-z0-9]{16}
  • Stored as Secrets in kube-system
  • Authenticated as system:bootstrap:<token-id>
  • Should be deleted after bootstrap completes

🛡️ Section 2

Service Accounts

⚙️ Service Account Fundamentals

Service accounts provide identity for processes running in pods. Unlike user accounts, they are Kubernetes API objects.

Key Properties

  • Namespaced resources
  • A default SA exists in every namespace
  • Tokens are projected as volumes into pods
  • Used by controllers, operators, and workloads
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app
  namespace: production
  annotations:
    description: "SA for my-app workload"
automountServiceAccountToken: false
Explain that in Kubernetes 1.24+, secrets are no longer auto-created for service accounts. Bound service account tokens (projected volumes) are the default mechanism.

📦 Bound Service Account Tokens

Since Kubernetes 1.22+, tokens are bound, time-limited, and audience-scoped.

Improvements over legacy tokens: Bound tokens expire, are tied to a specific pod, and cannot be reused after the pod is deleted. 
# Token projection in a pod spec
spec:
  serviceAccountName: my-app
  containers:
    - name: app
      volumeMounts:
        - name: token
          mountPath: /var/run/secrets/kubernetes.io/serviceaccount
          readOnly: true
  volumes:
    - name: token
      projected:
        sources:
          - serviceAccountToken:
              expirationSeconds: 3600
              audience: api
              path: token
          - configMap:
              name: kube-root-ca.crt
              items:
                - key: ca.crt
                  path: ca.crt
          - downwardAPI:
              items:
                - path: namespace
                  fieldRef:
                    fieldPath: metadata.namespace

🛠️ Service Account Best Practices

Opt Out of Auto-Mount

apiVersion: v1
kind: ServiceAccount
metadata:
  name: no-api-access
automountServiceAccountToken: false

Most workloads do not need API access. Disable token mounting by default.

Dedicated SAs

Create dedicated service accounts for each workload instead of using default.

  • Enables granular RBAC
  • Improves audit trail
  • Limits blast radius

Avoid Token Secrets

Legacy long-lived tokens (pre-1.24) are a security risk.

  • Never share tokens across pods
  • Delete unused token secrets
  • Use bound tokens instead
This is a critical security hardening step. Many breaches begin with a compromised pod that has unnecessary API access. Walk through a real scenario of lateral movement via a default service account token.

🛡️ Section 3

RBAC Architecture

🏗️ RBAC Building Blocks

Role-Based Access Control (RBAC) consists of four key resources:

Namespace-Scoped

Role
↔
RoleBinding

Grants permissions within a single namespace.

Cluster-Scoped

ClusterRole
↔
ClusterRoleBinding

Grants permissions cluster-wide or on non-namespaced resources.

Important: A RoleBinding can reference a ClusterRole to grant its permissions within a single namespace. This is a powerful pattern for reusing common roles across namespaces.
Draw the distinction clearly: a ClusterRole + RoleBinding gives namespaced access, while a ClusterRole + ClusterRoleBinding gives cluster-wide access. This is a common source of confusion.

💬 RBAC Verbs

Verb HTTP Method Description
get GET (single) Retrieve a specific resource
list GET (collection) List resources in a namespace
watch GET (watch) Stream changes to resources
create POST Create a new resource
update PUT Replace an entire resource
patch PATCH Partially modify a resource
delete DELETE Remove a resource
deletecollection DELETE (collection) Delete multiple resources at once
Special verbs: bind, escalate, and impersonate are privilege-escalation-sensitive and require careful handling.

📝 Role & ClusterRole Examples

Namespace Role

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: pod-reader
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods/log"]
    verbs: ["get"]

ClusterRole

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: node-reader
rules:
  - apiGroups: [""]
    resources: ["nodes"]
    verbs: ["get", "list", "watch"]
  - nonResourceURLs: ["/healthz", "/readyz"]
    verbs: ["get"]

Note: nonResourceURLs can only be used in ClusterRoles and match URL paths, not API resources.

🔗 Bindings: Connecting Roles to Subjects

RoleBinding

apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: production
subjects:
  - kind: User
    name: jane
    apiGroup: rbac.authorization.k8s.io
  - kind: ServiceAccount
    name: monitoring-agent
    namespace: monitoring
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

ClusterRoleBinding

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: read-nodes-global
subjects:
  - kind: Group
    name: platform-team
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: node-reader
  apiGroup: rbac.authorization.k8s.io
Subject kinds: User, Group, and ServiceAccount. Groups are extracted from authentication credentials (certificate O field, OIDC group claims).

🔄 Aggregated ClusterRoles

Aggregated ClusterRoles automatically combine rules from other ClusterRoles using label selectors.

Aggregation Target

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: monitoring-aggregate
aggregationRule:
  clusterRoleSelectors:
    - matchLabels:
        rbac.example.com/aggregate-to-monitoring:
          "true"
rules: [] # Rules are auto-filled

Contributing Role

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: monitoring-pods
  labels:
    rbac.example.com/aggregate-to-monitoring:
      "true"
rules:
  - apiGroups: [""]
    resources: ["pods", "services"]
    verbs: ["get", "list", "watch"]
Built-in example: The admin, edit, and view ClusterRoles use aggregation, allowing CRD controllers to automatically extend them.
Aggregated ClusterRoles are how built-in roles like admin and edit automatically gain permissions for new CRDs. Operators should add labels to their CRD ClusterRoles to integrate with the standard roles.

🛡️ Section 4

Designing Least-Privilege Roles

🎯 The Principle of Least Privilege

"Grant only the minimum permissions required for the task, for the shortest duration necessary, and in the narrowest scope possible."

Narrow Resources

Specify exact resources and subresources, avoid wildcards.

Narrow Verbs

Use read-only verbs when write access is unnecessary.

Narrow Scope

Prefer namespace Roles over ClusterRoles whenever possible.

🔧 Pattern: Namespace-Scoped Operator

An operator that manages custom resources and deployments within a single namespace.

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: app-system
  name: app-operator
rules:
  # Manage the custom resource
  - apiGroups: ["apps.example.com"]
    resources: ["myapps", "myapps/status"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  # Create and manage deployments
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
  # Read-only on services and configmaps
  - apiGroups: [""]
    resources: ["services", "configmaps"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  # Create events for status reporting
  - apiGroups: [""]
    resources: ["events"]
    verbs: ["create", "patch"]
Walk through each rule and explain why the operator needs each permission. Contrast this with giving the operator cluster-admin, which is far too broad.

📊 Pattern: Read-Only Dashboard

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: dashboard-viewer
rules:
  - apiGroups: [""]
    resources: ["pods", "services", "endpoints", "namespaces"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["apps"]
    resources: ["deployments", "replicasets", "statefulsets", "daemonsets"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["batch"]
    resources: ["jobs", "cronjobs"]
    verbs: ["get", "list", "watch"]
  # Explicitly exclude secrets, configmaps with sensitive data
  # Do NOT add: resources: ["secrets"]
Important: Never grant get on secrets for dashboard roles. Even read-only access to secrets exposes credentials and tokens.

🚀 Pattern: CI/CD Deployer

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: cicd-deployer
rules:
  - apiGroups: ["apps"]
    resources: ["deployments"]
    verbs: ["get", "list", "watch", "update", "patch"]
    resourceNames: ["frontend", "backend", "api-gateway"]
  - apiGroups: [""]
    resources: ["configmaps"]
    verbs: ["get", "list", "watch", "create", "update", "patch"]
  - apiGroups: [""]
    resources: ["services"]
    verbs: ["get", "list"]
Notice: The resourceNames field restricts the deployer to only the specific deployments it manages. This prevents accidental or malicious modification of other workloads.
Emphasize resourceNames as a powerful but underused feature. Also note that resourceNames cannot be used with create, list, watch, or deletecollection verbs since those operate on collections rather than named resources.

📈 Pattern: Monitoring Agent

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: monitoring-agent
rules:
  - apiGroups: [""]
    resources: ["pods", "nodes", "services", "endpoints"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods/log"]
    verbs: ["get"]
  - apiGroups: ["metrics.k8s.io"]
    resources: ["pods", "nodes"]
    verbs: ["get", "list"]
  - nonResourceURLs: ["/metrics", "/healthz"]
    verbs: ["get"]
Read-only: No write verbs needed for a monitoring agent.
Cluster-scoped: Monitoring typically needs visibility across all namespaces.

RBAC Anti-Patterns

⚠️ Anti-Pattern: cluster-admin for Everyone

Never do this: 
# The most dangerous ClusterRoleBinding in existence
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: everyone-is-admin
subjects:
  - kind: Group
    name: system:authenticated
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: cluster-admin
  apiGroup: rbac.authorization.k8s.io

Risks

  • Any authenticated user can delete any resource
  • Can read all secrets in all namespaces
  • Can create privileged pods for node escape
  • Can modify RBAC to persist access

Instead

  • Grant cluster-admin only to break-glass accounts
  • Use namespace-scoped admin roles
  • Implement just-in-time access elevation
  • Audit all cluster-admin bindings regularly
This exact pattern has been found in real production clusters. Ask the class: "How would you detect this in your own cluster?" This leads into the auditing section.

⚠️ Anti-Pattern: Wildcard Permissions

Dangerous

rules:
  - apiGroups: ["*"]
    resources: ["*"]
    verbs: ["*"]

Equivalent to cluster-admin. Grants full access to every resource and every action.

# Also dangerous:
rules:
  - apiGroups: [""]
    resources: ["*"]
    verbs: ["get", "list", "watch"]

Includes secrets, even if not intended.

Safe Alternative

rules:
  - apiGroups: [""]
    resources:
      - "pods"
      - "services"
      - "configmaps"
    verbs:
      - "get"
      - "list"
      - "watch"
  - apiGroups: ["apps"]
    resources:
      - "deployments"
    verbs:
      - "get"
      - "list"
      - "watch"

⚠️ Anti-Pattern: Bind & Escalate Permissions

Granting bind or escalate verbs allows a subject to give themselves higher privileges.

Privilege Escalation via Bind

# If a user can bind roles...
rules:
  - apiGroups:
      ["rbac.authorization.k8s.io"]
    resources: ["rolebindings"]
    verbs: ["create"]
  - apiGroups:
      ["rbac.authorization.k8s.io"]
    resources: ["roles"]
    verbs: ["bind"]

They can bind any role to themselves, including cluster-admin.

Privilege Escalation via Escalate

# If a user can escalate...
rules:
  - apiGroups:
      ["rbac.authorization.k8s.io"]
    resources: ["roles"]
    verbs: ["escalate"]

They can modify a role to add permissions beyond their own, then use that role.

Rule: Never grant bind, escalate, or impersonate verbs unless absolutely required and thoroughly documented.

⚠️ More Common Anti-Patterns

ClusterRoles When Roles Suffice

Using ClusterRoles + ClusterRoleBindings for namespace-scoped workloads unnecessarily broadens access.

Fix: If the workload operates in a single namespace, use a namespace-scoped Role and RoleBinding.

Not Using resourceNames

Granting access to all resources of a type when only specific resources are needed.

Fix: Use resourceNames to restrict access to specific named resources. 
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    resourceNames: ["app-tls-cert"]
    verbs: ["get"]
Walk through a scenario: "A CI/CD pipeline only deploys three microservices. Should it have access to modify ALL deployments in the namespace? No - use resourceNames to scope it down."

🛡️ Section 6

Auditing RBAC

🔍 Auditing with kubectl auth can-i

The built-in tool for checking authorization decisions.

# Can I create deployments in production?
kubectl auth can-i create deployments -n production
# yes

# Can a specific user list pods?
kubectl auth can-i list pods --as jane -n default
# no

# Can a service account delete secrets?
kubectl auth can-i delete secrets \
  --as system:serviceaccount:monitoring:prom-sa -n monitoring
# no

# List ALL permissions for the current user in a namespace
kubectl auth can-i --list -n production
# Resources                          Non-Resource URLs   Verbs
# pods                               []                  [get list watch]
# deployments.apps                   []                  [get list watch]
# ...
Tip: Use kubectl auth can-i --list during security audits to discover the full permission set of any identity.
Live demo opportunity: run these commands against the lab cluster and show the output. Demonstrate both user impersonation (--as) and service account impersonation.

🔎 Identifying Over-Privileged Service Accounts

A systematic approach to finding and fixing excessive permissions:

Audit Checklist

  1. List all ClusterRoleBindings: kubectl get clusterrolebindings -o wide
  2. Find bindings to cluster-admin: Filter for roleRef: cluster-admin
  3. Check default service accounts: Ensure no extra permissions on default SAs
  4. Review wildcard rules: Search for "*" in verbs or resources
  5. Check secret access: Identify who can get or list secrets
# Find all subjects with cluster-admin
kubectl get clusterrolebindings -o json | \
  jq -r '.items[] | select(.roleRef.name=="cluster-admin") |
    .subjects[]? | "\(.kind)/\(.name) in \(.namespace // "cluster-wide")"'

# Find roles with wildcard access
kubectl get roles,clusterroles -A -o json | \
  jq -r '.items[] | select(.rules[]? | .verbs[]? == "*" or .resources[]? == "*") |
    "\(.kind)/\(.metadata.name)"'
Run these commands live. The jq queries are powerful for bulk auditing. Recommend incorporating these checks into CI/CD pipelines and periodic security scans.

🛡️ Section 7

OIDC Integration

🌐 Why External Identity Providers?

Centralized Management

  • Single source of truth for identities
  • Onboarding / offboarding in one place
  • Consistent identity across systems

Enhanced Security

  • MFA enforcement at the IdP level
  • Short-lived tokens with refresh
  • Session management and revocation

Compliance

  • Audit trail for authentication events
  • Password policies and rotation
  • Regulatory requirement alignment
"Kubernetes should verify identity, not manage it. Delegate user management to purpose-built identity platforms."

🔄 OIDC Authentication Flow

1. User Authenticates
➡
2. IdP Issues JWT
➡
3. kubectl + Token
➡
4. API Server Validates

Step-by-Step

  1. User authenticates with the IdP (login page, SSO)
  2. IdP returns an id_token (JWT) and optionally a refresh_token
  3. kubectl sends the id_token in the Authorization: Bearer header
  4. API server validates the JWT signature against the IdP's public keys (JWKS)
  5. Username and groups are extracted from configured claims
  6. RBAC evaluates based on the extracted identity
Key point: The API server never contacts the IdP at request time. It validates the JWT signature using cached JWKS keys. This means token revocation only takes effect after the token expires.
This is a critical architectural detail. Because the API server validates tokens locally, revoked tokens remain valid until they expire. This is why short token lifetimes (5-15 minutes) are important.

⚙️ Configuring the API Server for OIDC

# API server flags for OIDC authentication
kube-apiserver \
  --oidc-issuer-url=https://keycloak.example.com/realms/kubernetes \
  --oidc-client-id=kubernetes \
  --oidc-username-claim=email \
  --oidc-username-prefix="oidc:" \
  --oidc-groups-claim=groups \
  --oidc-groups-prefix="oidc:" \
  --oidc-ca-file=/etc/kubernetes/pki/oidc-ca.crt \
  --oidc-required-claim="aud=kubernetes"
Flag Purpose
--oidc-issuer-url URL of the OIDC provider (must use HTTPS)
--oidc-client-id Client ID registered with the provider
--oidc-username-claim JWT claim to use as the username (default: sub)
--oidc-groups-claim JWT claim containing group memberships
--oidc-username-prefix Prefix to prevent collisions with other auth methods
Walk through each flag and explain its significance. The prefix flags are important to prevent naming collisions between OIDC users and certificate-based users.

👥 Group Claims for RBAC

Map IdP groups to Kubernetes RBAC for scalable access management.

JWT Token Claims

{
  "iss": "https://keycloak.example.com/realms/kubernetes",
  "sub": "a1b2c3d4",
  "email": "jane@example.com",
  "groups": [
    "platform-team",
    "production-deployers"
  ],
  "aud": "kubernetes",
  "exp": 1711900800
}

RBAC Binding to Group

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: platform-team-admin
subjects:
  - kind: Group
    name: "oidc:platform-team"
    apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: admin
  apiGroup: rbac.authorization.k8s.io
Scalable pattern: When a new team member is added to platform-team in the IdP, they automatically receive the bound Kubernetes permissions. No cluster configuration changes needed.

📝 Module 2 Key Takeaways

  • Kubernetes relies on external sources for authentication — it has no user database
  • Service accounts should use bound tokens and disable auto-mount when API access is unnecessary
  • RBAC follows a Role + Binding model — prefer namespace-scoped Roles over ClusterRoles
  • Apply least-privilege: specify exact resources, verbs, and use resourceNames when possible
  • Avoid wildcards, cluster-admin bindings, and bind/escalate permissions
  • Audit RBAC regularly with kubectl auth can-i and jq-based queries against role definitions
  • Integrate OIDC for human users to leverage centralized identity management and MFA
