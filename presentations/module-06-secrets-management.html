🔑 Section 1

Secrets Fundamentals

We start with what Kubernetes Secrets actually are, how they are stored, and why the default behavior is not as secure as many assume.

What Are Kubernetes Secrets?

📋 Definition

Secrets are Kubernetes objects that hold small amounts of sensitive data such as passwords, tokens, and keys — separate from Pod specs and container images.

  • Created independently of Pods that consume them
  • Less risk of exposure during Pod creation, viewing, and editing
  • Kubelet and API server can apply access controls
  • Maximum size: 1 MiB per Secret
Secrets were introduced to keep sensitive data out of Pod specs and container images. They provide a layer of indirection, but they are NOT encrypted by default. This is a critical distinction that many practitioners overlook.

Built-in Secret Types

Type Description Use Case
Opaque Arbitrary user-defined data Passwords, API keys
kubernetes.io/tls TLS cert & key pair Ingress TLS termination
kubernetes.io/dockerconfigjson Docker registry credentials Image pull secrets
kubernetes.io/basic-auth Basic authentication Username/password pairs
kubernetes.io/ssh-auth SSH private key Git clone over SSH
kubernetes.io/service-account-token ServiceAccount token API authentication
Kubernetes provides several built-in types that add validation and structure. Opaque is the default and most commonly used. The type primarily affects what fields are expected and validated.

Creating Secrets

# From literal values
kubectl create secret generic db-creds \
  --from-literal=username=admin \
  --from-literal=password='S3cur3P@ss!'

# From files
kubectl create secret tls app-tls \
  --cert=tls.crt --key=tls.key

# From a YAML manifest (values must be base64-encoded)
kubectl apply -f secret.yaml
# secret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-creds
type: Opaque
data:
  username: YWRtaW4=        # echo -n 'admin' | base64
  password: UzNjdXIzUEBzcyE=  # echo -n 'S3cur3P@ss!' | base64
There are several ways to create secrets. The imperative approach with kubectl is convenient for quick creation, but for reproducibility you'll want YAML manifests. Note that YAML-based secrets require base64-encoded values.

⚠️ Base64 is NOT Encryption

Critical Misconception: Many assume base64 encoding provides security. It does not — it is trivially reversible. 
# "Decrypting" a Kubernetes secret takes one command
$ echo 'UzNjdXIzUEBzcyE=' | base64 -d
S3cur3P@ss!

# Anyone with GET access to secrets can read them
$ kubectl get secret db-creds -o jsonpath='{.data.password}' | base64 -d
S3cur3P@ss!
Key Point: Base64 is an encoding scheme, not a security mechanism. Without encryption at rest, secrets are stored in plaintext in etcd.
This is one of the most important points in this entire module. Base64 encoding exists to handle binary data in JSON/YAML, not to provide confidentiality. Anyone with API access to read secrets can decode them instantly.

How Secrets Are Stored in etcd

Default Behavior

  • Stored as base64-encoded plaintext in etcd
  • Anyone with etcd access can read all secrets
  • etcd backups contain all secrets in the clear
  • No encryption layer between API server and etcd

Verification

# Read secret directly from etcd
ETCDCTL_API=3 etcdctl get \
  /registry/secrets/default/db-creds \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  | hexdump -C | head -20

Without encryption at rest, the secret value is visible in the hexdump output.

This demonstration shows that without encryption at rest, etcd stores secrets in a way that is trivially readable. This is why controlling access to etcd is critical, and why encryption at rest is essential for production clusters.

Secrets Threat Model

🔴 Storage

  • Plaintext in etcd
  • Unencrypted backups
  • etcd compromise

🟠 Transit

  • API server to etcd
  • API server to kubelet
  • Kubelet to Pod

🔵 Consumption

  • Environment variables
  • Volume mounts
  • Application logging
Secrets must be protected at rest, in transit, and during consumption. Each layer requires different controls.
Think of secrets management as a chain: storage in etcd, transmission over the network, and consumption inside pods. A weakness at any point in this chain can lead to exposure. We will address each of these layers in the following sections.

🔒 Section 2

Encryption at Rest

Now that we understand the problem, let's look at how to encrypt secrets when they're stored in etcd. This is a fundamental hardening step for any production cluster.

EncryptionConfiguration Resource

apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets
    providers:
      - aescbc:
          keys:
            - name: key1
              secret: c2VjcmV0LWtleS0xMjM0NTY3ODkwMTIzNDU2  # 32-byte base64
      - identity: {}  # Fallback: allows reading unencrypted secrets
How it works: The first provider in the list is used for writing. All providers are tried in order for reading. The identity provider at the end ensures existing unencrypted secrets remain readable during migration.
The EncryptionConfiguration tells the API server how to encrypt and decrypt resources before storing them in etcd. The order of providers matters: the first one is used for encryption, while all are tried for decryption. This allows gradual migration from unencrypted to encrypted storage.

Encryption Providers Compared

Provider Strength Speed Key Management
identity None (plaintext) Fastest N/A
aescbc Strong (AES-256-CBC) Fast Local keys
aesgcm Strong (AES-256-GCM) Fastest encrypted Must rotate frequently
secretbox Strong (XSalsa20+Poly1305) Fast Local keys
kms v2 Strongest (envelope encryption) Slight latency External KMS
Note: aesgcm requires key rotation every 200,000 writes due to nonce reuse risk. For most deployments, aescbc or kms v2 is recommended.
The choice of provider depends on your security requirements and operational capabilities. KMS v2 with envelope encryption is the gold standard because the data encryption key is itself encrypted by an external KMS, adding defense in depth. For clusters without external KMS access, aescbc is a solid choice.

Configuring the API Server

# 1. Generate a 32-byte encryption key
head -c 32 /dev/urandom | base64

# 2. Create the EncryptionConfiguration file
sudo vi /etc/kubernetes/enc/encryption-config.yaml

# 3. Update the kube-apiserver manifest
sudo vi /etc/kubernetes/manifests/kube-apiserver.yaml
# Add to kube-apiserver.yaml spec.containers.command:
- --encryption-provider-config=/etc/kubernetes/enc/encryption-config.yaml

# Add volume mount:
volumeMounts:
  - name: enc-config
    mountPath: /etc/kubernetes/enc
    readOnly: true

# Add volume:
volumes:
  - name: enc-config
    hostPath:
      path: /etc/kubernetes/enc
      type: DirectoryOrCreate
For kubeadm-based clusters, the API server runs as a static pod. Modifying the manifest in /etc/kubernetes/manifests will trigger an automatic restart. Make sure the encryption config file is in place before restarting the API server, or it will fail to start.

Verifying Encryption Works

# Create a test secret
kubectl create secret generic encryption-test \
  --from-literal=mykey=mydata

# Read it directly from etcd
ETCDCTL_API=3 etcdctl get \
  /registry/secrets/default/encryption-test \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key | hexdump -C

Before Encryption

Hexdump shows mydata in plaintext

After Encryption

Hexdump shows k8s:enc:aescbc:v1:key1 prefix followed by ciphertext

The encrypted data will have a prefix indicating the encryption provider and key name used. If you see plaintext data, encryption is not working. The prefix format is k8s:enc:PROVIDER:v1:KEY_NAME followed by the encrypted payload.

Encryption Key Rotation

apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets
    providers:
      - aescbc:
          keys:
            - name: key2              # NEW key first (used for writing)
              secret: bmV3LXNlY3JldC1rZXktMTIzNDU2Nzg5MDEyMzQ1Ng==
            - name: key1              # OLD key second (still used for reading)
              secret: c2VjcmV0LWtleS0xMjM0NTY3ODkwMTIzNDU2
      - identity: {}
# After updating the config and restarting the API server:
# Re-encrypt all existing secrets with the new key
kubectl get secrets --all-namespaces -o json | \
  kubectl replace -f -

# Verify, then remove the old key from the config
Key rotation process: Add new key first → restart API server → re-encrypt all secrets → remove old key → restart again.
Key rotation is essential but often overlooked. The process must be done carefully: add the new key before removing the old one, and re-encrypt all existing secrets. If you remove the old key before re-encrypting, those secrets become unreadable.

📦 Section 3

Accessing Secrets Securely

Now we look at how pods consume secrets. The method of delivery matters significantly for security. We will compare environment variables and volume mounts, and discuss best practices.

Volume Mounts vs Environment Variables

✅ Volume Mounts (Preferred)

volumeMounts:
  - name: db-creds
    mountPath: /etc/secrets
    readOnly: true
volumes:
  - name: db-creds
    secret:
      secretName: db-creds
  • Stored on tmpfs (in-memory)
  • Auto-updated on change
  • Not exposed in logs
  • Not visible via /proc

⚠️ Environment Variables (Risky)

env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: db-creds
        key: password
  • Visible in kubectl describe pod
  • Inherited by child processes
  • Often logged by frameworks
  • Not updated without restart
This is a critical architectural decision. Environment variables are convenient but introduce multiple exposure vectors. Volume mounts are strongly preferred because secrets are stored in tmpfs (never written to disk) and applications can watch for updates.

Why Environment Variables Leak

# Risk 1: Visible via /proc filesystem
$ cat /proc/1/environ | tr '\0' '\n' | grep DB_PASSWORD
DB_PASSWORD=S3cur3P@ss!

# Risk 2: Inherited by child processes
$ docker inspect <container-id> | jq '.[0].Config.Env'

# Risk 3: Application crash dumps and error logs
ERROR: Connection failed with config:
  host=db.example.com, user=admin, password=S3cur3P@ss!

# Risk 4: Third-party libraries may log environment
DEBUG: Environment variables: {DB_PASSWORD=S3cur3P@ss!, ...}
Real-world impact: Environment variables are one of the most common sources of secret leaks in production. Many logging frameworks capture the full environment on startup or error.
These are not theoretical risks. Major security breaches have occurred because environment variables containing credentials were captured in logs, crash dumps, or monitoring systems. The 12-factor app pattern of using env vars for config is valid for non-sensitive values, but secrets deserve better protection.

tmpfs and Projected Volumes

💾 Secret Volume = tmpfs

When a Secret is mounted as a volume, Kubernetes uses a tmpfs (RAM-backed filesystem). The secret data is never written to the node's disk.

# Projected volume: combine multiple sources
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  containers:
    - name: app
      image: myapp:latest
      volumeMounts:
        - name: all-secrets
          mountPath: /etc/secrets
          readOnly: true
  volumes:
    - name: all-secrets
      projected:
        sources:
          - secret:
              name: db-creds
          - secret:
              name: api-keys
          - configMap:
              name: app-config
Projected volumes are powerful because they let you combine secrets, configmaps, and service account tokens into a single mount point. This simplifies application configuration while maintaining security. The tmpfs backing means that even if the node is compromised and its disk is imaged, mounted secrets are not recoverable from the disk.

Immutable Secrets

apiVersion: v1
kind: Secret
metadata:
  name: db-creds
type: Opaque
immutable: true   # Cannot be modified after creation
data:
  username: YWRtaW4=
  password: UzNjdXIzUEBzcyE=

Benefits

  • Protects against accidental updates
  • Reduces API server load (no watches)
  • Prevents tampering by compromised controllers

Trade-offs

  • Must delete & recreate to update
  • Pods must be restarted for new values
  • Requires versioned naming strategy
Immutable secrets are a good security practice for values that should never change at runtime. They also improve cluster performance at scale since the API server does not need to maintain watches for immutable objects. Use a naming convention like db-creds-v2 when rotation is needed.

🏦 Section 4

External Secrets & HashiCorp Vault

While Kubernetes-native encryption at rest is a good first step, production environments benefit from dedicated external secret management systems. Vault is the most widely adopted solution in the Kubernetes ecosystem.

Why External Secret Stores?

🔒 Centralized Control

  • Single source of truth
  • Cross-cluster secrets
  • Unified audit trail

🔄 Dynamic Secrets

  • Generated on demand
  • Automatic expiration
  • Unique per consumer

🛡️ Advanced Features

  • HSM-backed encryption
  • Fine-grained policies
  • Automatic rotation
Kubernetes Secrets solve the delivery problem. External stores solve the management problem: lifecycle, rotation, audit, and access control.
The key insight is that Kubernetes secrets and external secret stores serve different purposes. K8s secrets are a delivery mechanism. External stores like Vault provide the full secret lifecycle management that production systems require.

HashiCorp Vault Architecture

Core Components

  • Secret Engines: Generate/store secrets (KV, PKI, database, AWS, etc.)
  • Auth Methods: Kubernetes, OIDC, AppRole, cloud IAM
  • Policies: Fine-grained path-based access control
  • Audit Devices: Full audit log of all operations

Kubernetes Integration

# Enable Kubernetes auth
vault auth enable kubernetes

# Configure with cluster info
vault write auth/kubernetes/config \
  kubernetes_host="https://$K8S_HOST" \
  token_reviewer_jwt="$SA_JWT" \
  kubernetes_ca_cert="$K8S_CA_CERT"

# Create a role
vault write auth/kubernetes/role/app \
  bound_service_account_names=app-sa \
  bound_service_account_namespaces=prod \
  policies=app-secrets \
  ttl=1h
Vault's Kubernetes auth method uses the TokenReview API to validate service account tokens. This means pods can authenticate to Vault using their service account identity, with no static credentials needed. The role binding ties specific service accounts in specific namespaces to Vault policies.

Vault Agent Injector

apiVersion: v1
kind: Pod
metadata:
  name: app
  annotations:
    vault.hashicorp.com/agent-inject: "true"
    vault.hashicorp.com/role: "app"
    vault.hashicorp.com/agent-inject-secret-db-creds: "secret/data/db/creds"
    vault.hashicorp.com/agent-inject-template-db-creds: |
      {{- with secret "secret/data/db/creds" -}}
      postgresql://{{ .Data.data.username }}:{{ .Data.data.password }}@db:5432/app
      {{- end -}}
spec:
  serviceAccountName: app-sa
  containers:
    - name: app
      image: myapp:latest
      # Secret is available at /vault/secrets/db-creds
How it works: A mutating webhook injects a Vault Agent sidecar that authenticates to Vault, fetches secrets, and renders them as files inside a shared volume — all transparent to the application.
The Vault Agent Injector uses a Kubernetes mutating admission webhook. When it detects the vault annotations, it injects an init container (for initial secret fetch) and a sidecar container (for ongoing rotation). The application just reads files from /vault/secrets/ with no Vault SDK needed.

Vault CSI Provider

apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: vault-db-creds
spec:
  provider: vault
  parameters:
    vaultAddress: "https://vault.example.com:8200"
    roleName: "app"
    objects: |
      - objectName: "db-username"
        secretPath: "secret/data/db/creds"
        secretKey: "username"
      - objectName: "db-password"
        secretPath: "secret/data/db/creds"
        secretKey: "password"

Agent Injector

  • Sidecar-based
  • Annotation-driven
  • Template support
  • Adds resource overhead

CSI Provider

  • DaemonSet-based
  • Volume-driven
  • Lighter per-pod footprint
  • Standard CSI interface
The CSI provider is generally preferred for new deployments because it doesn't add a sidecar to every pod. However, the Agent Injector has richer templating support. Choose based on your needs: CSI for simpler use cases, Agent Injector when you need complex secret rendering.

Cloud Provider Secret Stores

AWS Secrets Manager

  • IRSA authentication
  • Automatic rotation via Lambda
  • Cross-account access
  • CSI driver available

Azure Key Vault

  • Workload Identity
  • HSM-backed keys
  • Managed certificates
  • Native AKS integration

GCP Secret Manager

  • Workload Identity
  • Automatic replication
  • IAM-based access
  • CSI driver available
All three cloud providers offer CSI-based integrations through the Secrets Store CSI Driver, providing a consistent interface regardless of the backing store.
If you are running on a managed Kubernetes service (EKS, AKS, GKE), the native cloud secret store is often the path of least resistance. They integrate well with the cloud IAM model and require less operational overhead than self-hosted Vault.

🔌 Section 5

Secrets Store CSI Driver

The Secrets Store CSI Driver is a Kubernetes SIG project that provides a standardized interface for mounting external secrets as volumes. Let's look at its architecture and usage patterns.

Secrets Store CSI Driver Architecture

🏗️ How It Works

  1. CSI Driver runs as a DaemonSet on every node
  2. Provider plugin communicates with the external secret store
  3. On pod start, the driver calls the provider to fetch secrets
  4. Secrets are mounted as files in the pod's volume
  5. Optionally synced as Kubernetes Secret objects
# Install the CSI driver
helm repo add secrets-store-csi-driver \
  https://kubernetes-sigs.github.io/secrets-store-csi-driver/charts
helm install csi-secrets-store \
  secrets-store-csi-driver/secrets-store-csi-driver \
  --namespace kube-system \
  --set syncSecret.enabled=true \
  --set enableSecretRotation=true
The CSI driver itself is provider-agnostic. It defines the interface, while provider plugins handle the actual communication with Vault, AWS, Azure, or GCP. This separation means you can switch providers without changing your pod specs.

SecretProviderClass Configuration

apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: aws-secrets
spec:
  provider: aws
  parameters:
    objects: |
      - objectName: "arn:aws:secretsmanager:us-east-1:123456:secret:db-creds"
        objectType: "secretsmanager"
        jmesPath:
          - path: username
            objectAlias: db-username
          - path: password
            objectAlias: db-password
  # Optional: sync as a Kubernetes Secret
  secretObjects:
    - secretName: db-creds-k8s
      type: Opaque
      data:
        - objectName: db-username
          key: username
        - objectName: db-password
          key: password
The SecretProviderClass defines which secrets to fetch and how to map them. The jmesPath feature lets you extract individual fields from a JSON secret. The secretObjects section optionally creates a Kubernetes Secret from the fetched data, which is useful for applications that expect K8s-native secrets.

Mounting CSI Secrets in Pods

apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  serviceAccountName: app-sa
  containers:
    - name: app
      image: myapp:latest
      volumeMounts:
        - name: secrets
          mountPath: /mnt/secrets
          readOnly: true
      env:
        # Use synced K8s Secret as env var (if needed)
        - name: DB_USERNAME
          valueFrom:
            secretKeyRef:
              name: db-creds-k8s
              key: username
  volumes:
    - name: secrets
      csi:
        driver: secrets-store.csi.k8s.io
        readOnly: true
        volumeAttributes:
          secretProviderClass: aws-secrets
Rotation: With enableSecretRotation=true, the CSI driver periodically re-fetches secrets and updates the mounted files. The default rotation interval is 2 minutes.
Pods reference the SecretProviderClass through the volume definition. The CSI driver handles all the authentication and fetching. Note that rotation updates the mounted files, but synced Kubernetes Secrets also get updated when the source changes.

📨 Section 6

Sealed Secrets & GitOps

GitOps workflows store everything in Git, but secrets present a challenge: you cannot commit plaintext secrets to a repository. Sealed Secrets and SOPS solve this problem elegantly.

The GitOps Secret Problem

The Dilemma: GitOps requires all manifests in Git, but committing plaintext secrets to Git is a serious security violation.

❌ What Not to Do

# NEVER commit this to Git!
apiVersion: v1
kind: Secret
metadata:
  name: db-creds
data:
  password: UzNjdXIzUEBzcyE=
  • Base64 is not encryption
  • Git history retains secrets forever
  • Anyone with repo access sees them

✅ Solutions

  • Sealed Secrets — encrypt for the cluster
  • SOPS — encrypt individual values
  • External Secrets Operator — reference external stores
  • Vault — inject at runtime
Even if a repository is private, committing secrets to Git creates a permanent record. Git history is difficult to clean, and repository access often has broader permissions than secret access should allow.

Bitnami Sealed Secrets

🔐 How Sealed Secrets Work

  1. A controller runs in the cluster with a private/public key pair
  2. You encrypt secrets locally using the public key via kubeseal
  3. The encrypted SealedSecret resource is committed to Git
  4. The controller decrypts it using the private key and creates a regular Secret
# Install Sealed Secrets controller
helm repo add sealed-secrets https://bitnami-labs.github.io/sealed-secrets
helm install sealed-secrets sealed-secrets/sealed-secrets \
  --namespace kube-system

# Install kubeseal CLI
brew install kubeseal    # macOS
# or download from GitHub releases
Sealed Secrets is one of the most popular solutions for the GitOps secret problem. The key insight is that encryption happens client-side using the public key, so you never need to expose the private key outside the cluster. Only the cluster's controller can decrypt the sealed secrets.

Creating a SealedSecret

# Create a regular secret manifest (do NOT apply it)
kubectl create secret generic db-creds \
  --from-literal=password='S3cur3P@ss!' \
  --dry-run=client -o yaml > secret.yaml

# Seal it with kubeseal
kubeseal --format yaml < secret.yaml > sealed-secret.yaml

# The sealed version is safe to commit
cat sealed-secret.yaml
# sealed-secret.yaml — safe for Git
apiVersion: bitnami.com/v1alpha1
kind: SealedSecret
metadata:
  name: db-creds
  namespace: default
spec:
  encryptedData:
    password: AgBy3i4OJSWK+PiTySYZZA9rO43cGDEq...  # encrypted
  template:
    metadata:
      name: db-creds
      namespace: default
    type: Opaque
The workflow is straightforward: create a normal secret manifest with --dry-run, pipe it through kubeseal, and commit the result. The SealedSecret contains encrypted data that only the cluster controller can decrypt. Note that the sealed secret is scoped to a specific name and namespace by default.

Sealed Secrets Scopes

Scope Flag Behavior
strict (default) --scope strict Bound to exact name + namespace
namespace-wide --scope namespace-wide Can be renamed within the namespace
cluster-wide --scope cluster-wide Can be used in any namespace
Security note: Use strict scope whenever possible. Wider scopes allow sealed secrets to be moved to different namespaces, potentially bypassing RBAC controls.
The strict scope is the most secure because a sealed secret cannot be moved to a different namespace or renamed. If an attacker copies a cluster-wide sealed secret to a namespace they control, the controller will happily decrypt it there. Always prefer strict scope.

SOPS with age/GPG

# Install SOPS and age
brew install sops age

# Generate an age key
age-keygen -o key.txt
# Public key: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p

# Encrypt a secret file
sops --encrypt --age age1ql3z7hjy... secret.yaml > secret.enc.yaml

# Decrypt (requires the private key)
export SOPS_AGE_KEY_FILE=key.txt
sops --decrypt secret.enc.yaml
# secret.enc.yaml — values encrypted, keys readable
apiVersion: v1
kind: Secret
metadata:
    name: db-creds
data:
    password: ENC[AES256_GCM,data:abc123...,iv:xyz...,tag:def...]
sops:
    age:
        - recipient: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
          enc: |
            -----BEGIN AGE ENCRYPTED FILE-----
SOPS (Secrets OPerationS) takes a different approach from Sealed Secrets. It encrypts individual values within a YAML file while leaving keys and structure visible. This makes diffs meaningful in Git. It supports age, GPG, AWS KMS, GCP KMS, and Azure Key Vault as encryption backends. Tools like Flux and ArgoCD have native SOPS integration.

🛡️ Section 7

Secrets Best Practices & RBAC

Our final section covers the operational practices that tie everything together: RBAC, auditing, rotation, and detecting secret sprawl. Even the best encryption is worthless without proper access controls.

RBAC for Secrets

# Restrict secret access to specific namespaces
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
  namespace: production
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get"]            # No list or watch!
    resourceNames: ["db-creds", "api-keys"]  # Named secrets only
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: app-secret-access
  namespace: production
subjects:
  - kind: ServiceAccount
    name: app-sa
    namespace: production
roleRef:
  kind: Role
  name: secret-reader
  apiGroup: rbac.authorization.k8s.io
Principle of least privilege: Grant get on specific secret names. Avoid list and watch on secrets unless absolutely necessary.
The key RBAC controls for secrets are: avoid granting list and watch verbs (which expose all secrets in a namespace), use resourceNames to restrict access to specific secrets, and never grant cluster-wide secret access. Remember that the default service account in each namespace should not have secret access.

⚠️ Dangerous RBAC Patterns

❌ Avoid These

# Too broad: all secrets
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["*"]

# ClusterRole for secrets
kind: ClusterRole
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get", "list"]

✅ Better Approach

# Scoped: named secrets only
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    verbs: ["get"]
    resourceNames:
      - "app-tls"
      - "db-creds"

# Namespace-scoped Role
kind: Role
metadata:
  namespace: production
Watch out: list access returns secret data by default. An attacker with list on secrets can dump every secret in the namespace with a single API call.
One common mistake is granting list access thinking it only returns metadata. In Kubernetes, listing secrets returns the full data field. This means a service account with list permissions can exfiltrate all secrets with kubectl get secrets -o yaml.

Audit Logging for Secrets

# audit-policy.yaml
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
  # Log all secret access at the metadata level
  # (do NOT log request/response bodies for secrets!)
  - level: Metadata
    resources:
      - group: ""
        resources: ["secrets"]
    # Optionally log specific verbs at higher level
    verbs: ["get", "list", "watch", "create", "update", "delete"]

  # Exclude the kube-system service accounts from noisy logs
  - level: None
    users:
      - "system:kube-controller-manager"
      - "system:kube-scheduler"
    resources:
      - group: ""
        resources: ["secrets"]
Never use Request or RequestResponse audit level for secrets — this would log the actual secret values in the audit log, creating a new exposure vector.
Audit logging for secrets is a balancing act. You want to know who is accessing secrets, but you must avoid logging the secret data itself. The Metadata level captures the who, what, when without recording the actual secret values. Send audit logs to a SIEM for alerting on anomalous secret access patterns.

Secret Rotation Strategies

🔄 Manual Rotation

  • Update secret value
  • Rolling restart pods
  • Simple but error-prone
  • Use for small clusters
kubectl create secret \
  generic db-creds \
  --from-literal=password=NEW \
  --dry-run=client -o yaml \
  | kubectl apply -f -

kubectl rollout restart \
  deployment/app

🤖 Automated Rotation

  • External Secrets Operator
  • CSI driver rotation
  • Vault dynamic secrets
  • Cloud-native rotation

🎯 Dual-Secret Pattern

  • Two active credentials
  • Rotate one at a time
  • Zero-downtime rotation
  • Used by cloud providers
The dual-secret pattern (also called key versioning) is the gold standard for zero-downtime rotation. You maintain two valid credentials, rotate one while the other remains active, verify the new one works, then rotate the second. This eliminates the window where no valid credential exists.

Detecting Secret Sprawl

# Find secrets not referenced by any pod
kubectl get secrets -A -o json | jq -r '
  .items[] | select(.type != "kubernetes.io/service-account-token") |
  "\(.metadata.namespace)/\(.metadata.name)"
' > all-secrets.txt

kubectl get pods -A -o json | jq -r '
  .items[].spec |
  (.volumes[]?.secret.secretName // empty),
  (.containers[].env[]?.valueFrom.secretKeyRef.name // empty),
  (.containers[].envFrom[]?.secretRef.name // empty)
' | sort -u > used-secrets.txt

comm -23 <(sort all-secrets.txt) <(sort used-secrets.txt)
Regular hygiene: Unused secrets are an attack surface. Audit and remove orphaned secrets as part of your security routine.
Secret sprawl is a common problem in long-lived clusters. Secrets accumulate as applications are deployed and removed, but the secrets often remain. Regular auditing to find and remove unused secrets reduces the blast radius if the cluster is compromised. Consider automating this check in your CI/CD pipeline.

✅ Secrets Management Checklist

Foundation

  • Enable encryption at rest for etcd
  • Use volume mounts over env vars
  • Restrict RBAC — no broad list/watch
  • Enable audit logging at Metadata level
  • Encrypt etcd traffic (TLS)

Advanced

  • Integrate external secret store (Vault/cloud KMS)
  • Use Sealed Secrets or SOPS for GitOps
  • Implement automated secret rotation
  • Audit for secret sprawl regularly
  • Mark secrets as immutable when possible
This checklist can serve as a maturity model. Start with the foundation items and progressively adopt the advanced practices. Every production cluster should at minimum have encryption at rest and proper RBAC controls for secrets.

Choosing a Secrets Solution

Solution Complexity Best For GitOps?
K8s Secrets + Encryption at Rest Low Small clusters, getting started No
Sealed Secrets Low GitOps workflows Yes
SOPS Medium GitOps + cloud KMS Yes
External Secrets Operator Medium Cloud-native environments Partial
HashiCorp Vault High Enterprise, multi-cluster, dynamic secrets Via ESO
There is no one-size-fits-all solution. The right choice depends on your team's operational maturity, infrastructure, and security requirements. Many organizations use a combination, such as Vault for dynamic secrets plus Sealed Secrets for static configuration.

📝 Key Takeaways

  • Base64 is encoding, not encryption. Always enable encryption at rest for secrets in etcd.
  • Prefer volume mounts over environment variables. Env vars leak through logs, /proc, child processes, and crash dumps.
  • RBAC matters most for secrets. Restrict to get on named secrets; avoid list and watch.
  • External secret stores add defense in depth. Vault and cloud KMS provide lifecycle management, rotation, and audit.
  • Solve the GitOps secret problem. Use Sealed Secrets or SOPS — never commit plaintext secrets to Git.
  • Audit and rotate regularly. Detect sprawl, log access at the Metadata level, and automate rotation where possible.
These six points summarize the entire module. If students remember nothing else, they should remember that base64 is not encryption, volume mounts are safer than env vars, and secrets RBAC requires special attention.

🔒 Disabling Auto-Mounted Service Account Tokens

  • Kubernetes automatically mounts a service account token into every pod at /var/run/secrets/kubernetes.io/serviceaccount
  • A compromised pod can use that token to query the Kubernetes API
  • Most application pods do not need API access — disable the mount by default
  • Disable at the pod level with automountServiceAccountToken: false in the pod spec
  • Disable at the ServiceAccount level to apply to all pods using that SA
  • Re-enable selectively only for pods that genuinely require API access
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
automountServiceAccountToken: false
---
# Or at the pod level:
spec:
  automountServiceAccountToken: false
By default every pod gets a service account token mounted. If an attacker compromises the pod, they can use that token to talk to the Kubernetes API. Disabling the automatic mount is a simple but effective hardening step. Only pods that need to interact with the API — like controllers or operators — should have it enabled.
