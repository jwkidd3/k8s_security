🏗️ Admission Architecture

Where admission fits in the API request flow

Start with the architecture. Understanding where admission controllers sit in the request flow is essential before diving into specific tools.

🔄 Kubernetes API Request Flow

Every request to the API server passes through a pipeline:

  1. Authentication — Who is making the request?
  2. Authorization — Is this identity allowed to perform this action?
  3. Mutating Admission — Modify the request (add defaults, inject sidecars)
  4. Schema Validation — Is the object valid?
  5. Validating Admission — Does the request comply with policies?
  6. Persistence — Write to etcd
🔑 Admission controllers are your last line of defense before an object is persisted to the cluster.
Emphasize that admission is the final gate. Even if RBAC allows an action, an admission controller can deny it based on policy. This is defense in depth.

🔄 Mutating vs Validating Webhooks

✏️ Mutating Admission

  • Runs first in the pipeline
  • Can modify the incoming object
  • Use cases: inject sidecars, add labels, set defaults
  • Multiple mutating webhooks run in sequence
  • Order matters — later mutations see earlier changes

✅ Validating Admission

  • Runs after mutating admission
  • Can only accept or reject — no modifications
  • Use cases: enforce policies, block violations
  • Multiple validating webhooks run in parallel
  • Any rejection = entire request is denied
Key distinction: mutating webhooks can change the object, validating webhooks can only say yes or no. This separation is intentional for security — validation sees the final mutated object.

🛠️ Built-in vs Dynamic Admission

Built-in Controllers

  • Compiled into the API server
  • Enabled/disabled via --enable-admission-plugins
  • No external dependencies
  • Cannot be customized
  • Examples: NamespaceLifecycle, LimitRanger

Dynamic Controllers (Webhooks)

  • External services called via HTTPS
  • Configured via MutatingWebhookConfiguration or ValidatingWebhookConfiguration
  • Fully customizable logic
  • Require running webhook server
  • Examples: Gatekeeper, Kyverno
Built-in controllers handle fundamental cluster operations. Dynamic webhooks are where custom policy enforcement lives. Most policy engines use the dynamic webhook mechanism.

🔧 Built-in Admission Controllers

Essential controllers compiled into the API server

Now let's look at the most important built-in admission controllers. These are enabled by default in most distributions and handle critical cluster functionality.

📋 Essential Built-in Controllers

Controller Type Purpose
NamespaceLifecycle Validating Prevents operations in non-existent or terminating namespaces
LimitRanger Mutating Applies default resource requests/limits from LimitRange objects
ServiceAccount Mutating Automatically mounts service account tokens into pods
DefaultStorageClass Mutating Adds a default StorageClass to PVCs without one
ResourceQuota Validating Enforces resource quota constraints per namespace
NodeRestriction Validating Limits kubelet to modifying only its own node and pods
Walk through each controller. NamespaceLifecycle prevents creating resources in terminating namespaces. LimitRanger sets defaults when developers forget. NodeRestriction is critical for preventing compromised nodes from affecting other nodes.

🛡️ PodSecurity Admission Controller

Replaces PodSecurityPolicy (removed in v1.25)

Enforces Pod Security Standards at the namespace level via labels.

# Enforce restricted profile on a namespace
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted

Privileged

Unrestricted. No policy applied.

Baseline

Minimally restrictive. Prevents known privilege escalations.

Restricted

Heavily restricted. Follows hardening best practices.

PodSecurity is the built-in replacement for PSP. It is simpler but less flexible. For fine-grained control, you still need a policy engine like Gatekeeper or Kyverno. The three profiles are well-defined standards.

🔒 NodeRestriction in Detail

Why it matters: Without NodeRestriction, a compromised kubelet could modify any node or pod in the cluster.

  • Limits kubelets to modifying only their own Node object
  • Restricts kubelets to modifying pods bound to their node
  • Prevents kubelets from adding/modifying labels with node-restriction.kubernetes.io/ prefix
  • Must be combined with the Node authorizer for full effect

⚠️ Check your cluster: kube-apiserver --enable-admission-plugins=...,NodeRestriction

Most managed Kubernetes services enable this by default.

NodeRestriction is one of the most important security-relevant admission controllers. It implements the principle of least privilege for kubelets. Without it, a compromised node is a cluster-wide compromise.

🌐 Webhook Controllers

Dynamic admission via external services

Now let's look at how dynamic admission webhooks work. This is the mechanism that powers Gatekeeper, Kyverno, and other policy engines.

⚙️ ValidatingWebhookConfiguration

apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: my-policy-webhook
webhooks:
  - name: validate.example.com
    admissionReviewVersions: ["v1"]
    clientConfig:
      service:
        name: webhook-service
        namespace: webhook-system
        path: /validate
      caBundle: LS0tLS1CRUd...   # CA cert for TLS
    rules:
      - operations: ["CREATE", "UPDATE"]
        resources: ["pods"]
    failurePolicy: Fail           # Fail or Ignore
    sideEffects: None
Walk through each field. The caBundle provides TLS trust. Rules define which API calls trigger the webhook. failurePolicy determines behavior when the webhook is unreachable. namespaceSelector limits scope.

⚠️ Failure Policies & Timeouts

failurePolicy: Fail

  • Webhook unreachable = request denied
  • Safer from a security perspective
  • Risk: webhook outage blocks all matched operations
  • Can lock out the entire cluster if misconfigured

failurePolicy: Ignore

  • Webhook unreachable = request allowed
  • Safer from an availability perspective
  • Risk: security bypass during outages
  • Useful during initial rollout and testing
🔑 Best practice: Start with Ignore during rollout, switch to Fail once stable. Always set timeoutSeconds (default is 10s, max is 30s).
This is a critical operational decision. failurePolicy: Fail is the secure choice but can cause outages. Many teams start with Ignore in audit mode and move to Fail once confidence is established. Always set reasonable timeouts.

🎯 Matching Rules & Side Effects

Matching rules control which API requests trigger the webhook:

  • operations — CREATE, UPDATE, DELETE, CONNECT
  • apiGroups — Core (""), apps, batch, etc.
  • resources — pods, deployments, services, etc.
  • scope — Cluster, Namespaced, or *
  • namespaceSelector — Label-based namespace filtering
  • objectSelector — Label-based object filtering

sideEffects must be declared: None, NoneOnDryRun, or Some. Webhooks with side effects may fire on dry-run requests — always declare None if your webhook is pure.

Be precise with matching rules. Overly broad rules cause performance issues and can match system-critical resources. Always use namespaceSelector to exclude kube-system.

🚨 Webhook Security Considerations

Common Pitfalls

  • Self-referencing: Webhook that validates its own namespace can deadlock the cluster
  • Missing exclusions: Always exclude kube-system and the webhook's own namespace
  • TLS certificate expiry: Webhook becomes unreachable when certs expire
  • Resource exhaustion: Webhook under heavy load can slow all API operations
# Always exclude system namespaces
namespaceSelector:
  matchExpressions:
    - key: kubernetes.io/metadata.name
      operator: NotIn
      values: ["kube-system", "kube-public", "webhook-system"]
These are real production war stories. Teams have locked themselves out of clusters by having webhooks validate their own deployment namespace. Always test exclusions carefully.

🏛️ OPA Gatekeeper

Policy enforcement with Open Policy Agent

Gatekeeper is the Kubernetes-native integration for Open Policy Agent (OPA). It is one of the most mature and widely adopted policy engines in the ecosystem.

📖 OPA & Gatekeeper Architecture

Open Policy Agent (OPA)

  • General-purpose policy engine
  • Policies written in Rego language
  • Decouples policy from application code
  • Used beyond Kubernetes: APIs, Terraform, Envoy

Gatekeeper (v3)

  • Kubernetes-native OPA integration
  • Uses CRDs for policy management
  • ConstraintTemplates define policy logic
  • Constraints apply templates to resources
  • Supports audit mode for existing resources

Flow: API Request → Webhook → Gatekeeper Controller → Evaluate Rego → Allow/Deny

Gatekeeper v3 is a significant improvement over v1/v2. It uses CRDs, making policies Kubernetes-native. The ConstraintTemplate + Constraint model separates policy definition from policy application.

📝 ConstraintTemplate: Require Labels

apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequiredlabels
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredLabels
      validation:
        openAPIV3Schema:
          type: object
          properties:
            labels:
              type: array
              items:
                type: string
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8srequiredlabels

        violation[{"msg": msg}] {
          provided := {label | input.review.object.metadata.labels[label]}
          required := {label | label := input.parameters.labels[_]}
          missing := required - provided
          count(missing) > 0
          msg := sprintf("Missing required labels: %v", [missing])
        }
Walk through the structure. The CRD spec defines parameters the constraint can accept. The Rego code defines the policy logic. The violation rule returns messages for non-compliant resources.

🔗 Constraint: Apply the Template

apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredLabels
metadata:
  name: require-team-label
spec:
  enforcementAction: deny    # deny | dryrun | warn
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Namespace"]
      - apiGroups: ["apps"]
        kinds: ["Deployment"]
    excludedNamespaces:
      - kube-system
      - gatekeeper-system
  parameters:
    labels:
      - "team"
      - "environment"

enforcementAction options:

  • deny — Block non-compliant resources
  • dryrun — Audit only, log violations
  • warn — Allow but show warnings to the user
The constraint applies the template to specific resources. Always start with dryrun to see the impact before switching to deny. The excludedNamespaces list is critical to avoid breaking system components.

🚫 Gatekeeper: Block Privileged Pods

apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8sblockprivileged
spec:
  crd:
    spec:
      names:
        kind: K8sBlockPrivileged
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8sblockprivileged
        violation[{"msg": msg}] {
          c := input.review.object.spec.containers[_]
          c.securityContext.privileged == true
          msg := sprintf("Privileged container: %v", [c.name])
        }

In production, add a matching rule for initContainers as well.

Note that we need separate rules for containers and initContainers. This is a common pattern — always check both. A corresponding Constraint would apply this to all pods in target namespaces.

📦 Enforce Image Registry — Template

apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8sallowedrepos
spec:
  crd:
    spec:
      names:
        kind: K8sAllowedRepos
      validation:
        openAPIV3Schema:
          type: object
          properties:
            repos:
              type: array
              items: { type: string }
  targets:
    - target: admission.k8s.gatekeeper.sh
      rego: |
        package k8sallowedrepos
        violation[{"msg": msg}] {
          c := input.review.object.spec.containers[_]
          not startswith(c.image, input.parameters.repos[_])
          msg := sprintf("Image '%v' not from allowed registry", [c.image])
        }
This ConstraintTemplate defines the Rego logic. The next slide shows the Constraint that applies it to specific resources with parameters.

📦 Enforce Image Registry — Constraint

apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sAllowedRepos
metadata:
  name: require-trusted-registry
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
  parameters:
    repos:
      - "gcr.io/my-company/"
      - "us-docker.pkg.dev/my-company/"
🔑 One of the most common policies in production. Combine with image signing for stronger guarantees.
The Constraint applies the template to Pods and specifies which registries are allowed. Always start with dryrun enforcement before switching to deny.

📜 Kyverno

Kubernetes-native policy management — no new language required

Kyverno is an alternative to Gatekeeper that uses YAML instead of Rego. For teams already fluent in Kubernetes YAML, Kyverno has a lower learning curve.

🏗️ Kyverno Architecture

Key Features

  • YAML-native — no Rego or new language
  • Validate, mutate, generate, and verify images
  • ClusterPolicy (cluster-wide) and Policy (namespaced)
  • Policy reports via PolicyReport CRDs
  • Built-in JMESPath and CEL support

Architecture

  • Runs as a set of controllers in the cluster
  • Registers mutating and validating webhooks
  • Background scan for existing resources
  • Generates PolicyReport / ClusterPolicyReport
  • Admission controller + reporting engine

Rule types: validate | mutate | generate | verifyImages

Kyverno's YAML-native approach makes it accessible to any Kubernetes operator. The generate capability is unique — it can create resources (like NetworkPolicies) automatically when a namespace is created.

🏷️ Kyverno: Require Labels

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-labels
spec:
  validationFailureAction: Enforce   # Enforce or Audit
  background: true                    # Scan existing resources
  rules:
    - name: require-team-label
      match:
        any:
          - resources:
              kinds:
                - Deployment
                - StatefulSet
              namespaces:
                - "production"
                - "staging"
      exclude:
        any:
          - resources:
              namespaces:
                - "kube-system"
      validate:
        message: "The label 'team' is required."
        pattern:
          metadata:
            labels:
              team: "?*"

Compare to Gatekeeper: No Rego, no ConstraintTemplate. The pattern-matching syntax is pure YAML.

The pattern field uses Kyverno's pattern matching. The ?* wildcard means any non-empty value. This is the same policy we wrote in Gatekeeper but with significantly less code.

🚫 Kyverno: Block Privileged Pods

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: disallow-privileged-containers
spec:
  validationFailureAction: Enforce
  background: true
  rules:
    - name: deny-privileged
      match:
        any:
          - resources:
              kinds:
                - Pod
      validate:
        message: >-
          Privileged mode is not allowed. Set
          securityContext.privileged to false.
        deny:
          conditions:
            any:
              - key: "{{ request.object.spec.[containers, initContainers][].securityContext.privileged }}"
                operator: AnyIn
                value: [true]
Kyverno uses JMESPath expressions for complex lookups. The bracket notation checks both containers and initContainers arrays in one expression. Compare this to the two separate Rego rules in Gatekeeper.

📦 Kyverno: Enforce Image Registry

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: restrict-image-registries
spec:
  validationFailureAction: Enforce
  background: true
  rules:
    - name: validate-registries
      match:
        any:
          - resources:
              kinds:
                - Pod
      validate:
        message: "Images must come from an approved registry."
        pattern:
          spec:
            containers:
              - image: "gcr.io/my-company/* | us-docker.pkg.dev/my-company/*"
            =(initContainers):
              - image: "gcr.io/my-company/* | us-docker.pkg.dev/my-company/*"

The =(...) anchor means "if this field exists, it must match." Init containers are optional, so we use the conditional anchor.

The conditional anchor is a Kyverno-specific feature. Without it, the policy would fail on pods that have no initContainers. The pipe syntax allows multiple allowed patterns.

✨ Kyverno: Mutate & Generate Rules

Mutate: Add Default Labels

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: add-default-labels
spec:
  rules:
    - name: add-managed-by
      match:
        any:
          - resources:
              kinds:
                - Deployment
      mutate:
        patchStrategicMerge:
          metadata:
            labels:
              +(managed-by): "platform-team"

Generate: Create NetworkPolicy

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: default-deny-netpol
spec:
  rules:
    - name: generate-netpol
      match:
        any:
          - resources:
              kinds:
                - Namespace
      generate:
        kind: NetworkPolicy
        apiVersion: networking.k8s.io/v1
        name: default-deny-all
        namespace: "{{request.object.metadata.name}}"
        data:
          spec:
            podSelector: {}
            policyTypes:
              - Ingress
              - Egress

The +(...) anchor means "add if not already present." Generate rules create companion resources automatically.

These are powerful capabilities beyond simple validation. The generate rule automatically creates a default-deny NetworkPolicy whenever a new namespace is created — ensuring security by default. This is unique to Kyverno.

⚙️ ValidatingAdmissionPolicy

Native in-process validation with CEL expressions (K8s 1.28+)

ValidatingAdmissionPolicy is the newest option, built directly into the Kubernetes API server. It uses CEL — Common Expression Language — and requires no external webhook. This is GA as of Kubernetes 1.30.

💡 Why ValidatingAdmissionPolicy?

Advantages

  • No webhook overhead — runs in-process
  • No external dependencies to manage
  • No TLS certificates to rotate
  • Faster evaluation (microseconds vs milliseconds)
  • CEL is deterministic and sandboxed
  • Built into Kubernetes — no installation

Limitations

  • Validation only — no mutation or generation
  • CEL is less expressive than Rego
  • No external data sources
  • Requires Kubernetes 1.28+ (GA in 1.30)
  • Smaller policy library compared to Gatekeeper/Kyverno
  • No background scanning of existing resources
ValidatingAdmissionPolicy is not a replacement for Gatekeeper or Kyverno — it complements them. Use it for simple validation rules where webhook overhead is not justified. Complex policies still benefit from a full engine.

🏷️ CEL: Require Labels

apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicy
metadata:
  name: require-team-label
spec:
  failurePolicy: Fail
  matchConstraints:
    resourceRules:
      - apiGroups: ["apps"]
        apiVersions: ["v1"]
        operations: ["CREATE", "UPDATE"]
        resources: ["deployments"]
  validations:
    - expression: "has(object.metadata.labels) && 'team' in object.metadata.labels"
      message: "All deployments must have a 'team' label."
    - expression: "has(object.metadata.labels) && 'environment' in object.metadata.labels"
      message: "All deployments must have an 'environment' label."
---
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicyBinding
metadata:
  name: require-team-label-binding
spec:
  policyName: require-team-label
  validationActions: [Deny]
  matchResources:
    namespaceSelector:
      matchExpressions:
        - key: kubernetes.io/metadata.name
          operator: NotIn
          values: ["kube-system"]
Two resources are needed: the policy defines the rules, and the binding applies them. This is similar to RBAC's Role + RoleBinding pattern. CEL expressions are concise but readable.

🚫 CEL: Block Privileged Containers

apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicy
metadata:
  name: deny-privileged-containers
spec:
  failurePolicy: Fail
  matchConstraints:
    resourceRules:
      - apiGroups: [""]
        apiVersions: ["v1"]
        operations: ["CREATE", "UPDATE"]
        resources: ["pods"]
  validations:
    - expression: >-
        object.spec.containers.all(c,
          !has(c.securityContext) ||
          !has(c.securityContext.privileged) ||
          c.securityContext.privileged == false)
      message: "Privileged containers are not allowed."
    - expression: >-
        !has(object.spec.initContainers) ||
        object.spec.initContainers.all(c,
          !has(c.securityContext) ||
          !has(c.securityContext.privileged) ||
          c.securityContext.privileged == false)
      message: "Privileged init containers are not allowed."

CEL expressions use familiar syntax: .all(), .exists(), .filter(), .map(), has(), ternary ? :

CEL's .all() function iterates over all containers. The has() checks prevent null reference errors. Compare this to Gatekeeper's two Rego rules and Kyverno's JMESPath approach — all three achieve the same result.

📦 CEL: Enforce Image Registry

apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicy
metadata:
  name: restrict-image-registries
spec:
  failurePolicy: Fail
  matchConstraints:
    resourceRules:
      - apiGroups: [""]
        apiVersions: ["v1"]
        operations: ["CREATE", "UPDATE"]
        resources: ["pods"]
  validations:
    - expression: >-
        object.spec.containers.all(c,
          c.image.startsWith("gcr.io/my-company/") ||
          c.image.startsWith("us-docker.pkg.dev/my-company/"))
      message: "All container images must come from an approved registry."
    - expression: >-
        !has(object.spec.initContainers) ||
        object.spec.initContainers.all(c,
          c.image.startsWith("gcr.io/my-company/") ||
          c.image.startsWith("us-docker.pkg.dev/my-company/"))
      message: "All init container images must come from an approved registry."
🔑 Same policy across three engines — choose based on your team's needs, not the tool's popularity.
We have now seen the same three policies — require labels, block privileged, enforce registry — implemented in all three engines. This gives students a direct comparison to inform their choice.

📊 ValidatingAdmissionPolicy: Actions & Audit

apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingAdmissionPolicyBinding
metadata:
  name: restrict-registries-audit
spec:
  policyName: restrict-image-registries
  validationActions:
    - Audit     # Log violations but don't block
    - Warn      # Show warnings to the user
  matchResources:
    namespaceSelector:
      matchLabels:
        environment: staging

validationActions:

  • Deny — Reject the request
  • Audit — Log the violation in audit logs
  • Warn — Show a warning message to the user

You can combine actions: [Audit, Warn] for non-blocking enforcement.

The combination of Audit and Warn is excellent for gradual rollout. You can see violations in audit logs and users see warnings, but nothing is blocked. Switch to Deny when ready.

🧭 Policy Engine Strategy

Choosing, deploying, and managing policies at scale

Now that students understand all three engines, let's discuss how to choose between them and build a comprehensive policy strategy.

📊 Engine Comparison

Feature Gatekeeper Kyverno CEL (VAP)
Policy Language Rego YAML/JMESPath CEL
Validation ✅ ✅ ✅
Mutation ✅ (v3.10+) ✅ ❌
Generation ❌ ✅ ❌
Image Verification ❌ ✅ ❌
External Data ✅ ✅ (API calls) ❌
Webhook Overhead Yes Yes None
Background Scan ✅ (Audit) ✅ ❌
Policy Library Large Large Growing
Learning Curve Steeper (Rego) Lower (YAML) Moderate
No single engine is universally best. Gatekeeper excels in complex logic. Kyverno excels in usability and breadth of features. CEL excels in performance and simplicity for straightforward rules.

🧩 When to Use Each Engine

🏛️ Gatekeeper

  • Complex cross-resource policies
  • Existing OPA investment
  • Need for external data
  • Multi-platform policy (Terraform, Envoy)

📜 Kyverno

  • Teams preferring YAML
  • Need mutation + generation
  • Image verification
  • Auto-generating companion resources

⚙️ CEL (VAP)

  • Simple validation rules
  • Performance-critical clusters
  • Minimizing dependencies
  • Already on K8s 1.30+
🔑 You can use multiple engines together. CEL for simple rules, Kyverno/Gatekeeper for complex ones.
Many production environments combine engines. Use CEL for simple, high-frequency checks and a full engine for complex policies. The engines do not conflict with each other.

🔄 Policy-as-Code in CI/CD

Shift left: Test policies before they reach the cluster.

Gatekeeper

# Test with gator CLI
gator test \
  --filename policies/ \
  --filename test-resources/

# Verify constraint templates
gator verify suite.yaml

Kyverno

# Test with kyverno CLI
kyverno apply policies/ \
  --resource test-pod.yaml

# Test with expected results
kyverno test tests/

CI/CD Integration Pattern:

  1. Store policies in Git alongside application manifests
  2. Run policy tests on every pull request
  3. Validate application manifests against policies before deploy
  4. Deploy policies through GitOps (Argo CD, Flux)
Policy-as-code is essential. Both Gatekeeper and Kyverno provide CLI tools for testing policies in CI. This catches policy errors before they reach the cluster and prevents policy changes from blocking legitimate workloads.

🧪 Testing Policies

# Kyverno test manifest (tests/require-labels/kyverno-test.yaml)
apiVersion: cli.kyverno.io/v1alpha1
kind: Test
metadata:
  name: require-labels-test
policies:
  - ../../policies/require-labels.yaml
resources:
  - resources.yaml
results:
  - policy: require-labels
    rule: require-team-label
    resource: good-deployment    # Has required labels
    kind: Deployment
    result: pass
  - policy: require-labels
    rule: require-team-label
    resource: bad-deployment     # Missing labels
    kind: Deployment
    result: fail
# Run the tests
$ kyverno test tests/require-labels/
Loading test ( tests/require-labels/kyverno-test.yaml ) ...
  Pass: 2, Fail: 0, Warn: 0, Error: 0, Skip: 0
Writing test cases for policies is just as important as writing the policies themselves. Always test both positive (should pass) and negative (should fail) cases. Include edge cases like pods without securityContext fields.

📚 Policy Libraries

Gatekeeper Library

  • github.com/open-policy-agent/gatekeeper-library
  • Pod Security Standards templates
  • General-purpose templates
  • Well-tested and maintained
  • Community contributions

Kyverno Policies

  • kyverno.io/policies
  • 300+ ready-to-use policies
  • Categorized by Pod Security Standards

Both ecosystems have extensive policy libraries. Start with existing policies and customize them. Writing Rego or complex validation from scratch is error-prone. The community has already solved most common policy needs.

🚀 Policy Rollout Strategy

Phased Rollout Approach

  1. Audit/Warn mode — Deploy policies in non-blocking mode. Monitor violations.
  2. Targeted enforcement — Enable enforcement on non-production namespaces first.
  3. Broad enforcement — Roll out to production namespaces after validation.
  4. Continuous monitoring — Track policy violations and false positives.

⚠️ Never deploy policies directly in enforce mode to production. Always start with audit/warn and review the impact.

🔑 Monitor: kubectl get constraintname -o yaml (Gatekeeper) or kubectl get policyreport -A (Kyverno)
This phased approach prevents policies from breaking production workloads. The audit phase is critical for discovering legitimate workloads that would be blocked. Always have an exception process for approved violations.

🔄 Migration Paths

From PSP to modern policy engines:

Migration From Migration To Strategy
PodSecurityPolicy PodSecurity Admission Map PSP fields to PSS profiles; label namespaces
PodSecurityPolicy Gatekeeper/Kyverno Translate each PSP to equivalent constraint/policy
Gatekeeper Kyverno Rewrite Rego as YAML patterns; test in parallel
Webhook-based CEL (VAP) Migrate simple validation rules; keep complex ones in engine

Migration Best Practice

Run old and new engines in parallel. Deploy the new engine in audit mode alongside the existing one. Once violation counts match, switch enforcement.

PSP migration was a major project for many organizations when PSP was removed in 1.25. The parallel-run strategy is the safest approach for any migration. Never do a direct cutover.

📝 Key Takeaways

  • 🏗️ Admission controllers are the last gate before objects are persisted — use them to enforce security and operational policies.
  • 🔧 Built-in controllers like NodeRestriction and PodSecurity handle fundamental cluster security without external tools.
  • 🌐 Dynamic webhooks enable custom policy enforcement but require careful configuration around failure policies and exclusions.
  • 🏛️ OPA Gatekeeper offers powerful Rego-based policies and a mature ecosystem for complex policy needs.
  • 📜 Kyverno provides YAML-native policies with mutation, generation, and image verification — no new language required.
  • ⚙️ ValidatingAdmissionPolicy (CEL) delivers in-process validation with zero webhook overhead for Kubernetes 1.28+.
  • 🚀 Always roll out policies gradually — audit first, enforce later. Test policies in CI/CD before deployment.
  • 📚 Leverage policy libraries rather than writing from scratch. Combine engines for comprehensive coverage.
Review each takeaway. Emphasize the phased rollout approach and the importance of testing. The choice of engine should be driven by team capabilities and requirements, not hype.
