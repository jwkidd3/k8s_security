📦 Section 1

Container Image Fundamentals

We begin with the internals of container images so students understand why hardening matters at a structural level.

📦 Anatomy of a Container Image

  • Manifest — metadata describing layers and config
  • Config JSON — environment vars, entrypoint, user
  • Layers — filesystem diffs stacked via Union FS
  • Union FS — overlays layers into a single view (OverlayFS)

Layer Stacking

Layer 4: COPY app binary     (app)
Layer 3: RUN apt-get install  (packages)
Layer 2: RUN apt-get update   (index)
Layer 1: FROM ubuntu:22.04    (base OS)
Each Dockerfile instruction creates a layer. Layers are read-only; the container gets a thin writable layer on top. Understanding this helps explain why deleted files still consume space if removed in a later layer.

🔍 Image Layers and Security Implications

  • Each layer is immutable and content-addressable (SHA256)
  • Deleting a file in a later layer does not remove it from the image
  • Secrets added in any layer persist in the image history
  • Every package installed adds potential CVEs to the attack surface
Critical: docker history can reveal secrets embedded in earlier layers, even if a subsequent layer removes the file.
Demo idea: show docker history revealing a secret that was removed in a later layer. This is one of the most common image security mistakes.

🎯 Section 2

Base Image Selection

Choosing the right base image is one of the highest-impact decisions for container security.

🎯 Base Image Comparison

Image Use Case Pros Cons
Ubuntu/Debian Full-featured apps, legacy Familiar, broad compatibility Large, many CVEs
Alpine Minimal, general-purpose Tiny, fast builds musl libc issues
Distroless Production workloads No shell, minimal CVEs Hard to debug
scratch Static binaries (Go, Rust) Zero attack surface No OS utilities at all
Each base image sits on a spectrum from convenience to security. In production, lean toward distroless or scratch whenever possible.

🛡️ Distroless Deep Dive

Google's Distroless images contain only your application and its runtime dependencies. No shell, no package manager, no unnecessary OS utilities.

static

Statically-linked binaries (Go, Rust)

~2 MB

base

glibc + libssl + ca-certs

~20 MB

cc

base + libgcc

~22 MB

java

OpenJDK runtime

python3

Python interpreter

nodejs

Node.js runtime

Distroless images are maintained by Google and published to gcr.io/distroless. The static variant is the smallest and works for statically compiled languages. Language-specific variants include only the runtime needed.

🛡️ Distroless: Nonroot Tags

Every Distroless image ships a :nonroot tag variant that runs as UID 65534:

# Runs as root (default)
FROM gcr.io/distroless/static-debian12

# Runs as nonroot user (UID 65534) - preferred
FROM gcr.io/distroless/static-debian12:nonroot
Best practice: Always use the :nonroot tag unless your application specifically requires root privileges.
The nonroot variant sets the USER directive to 65534 (nobody). This means even if an attacker breaks out of the application, they have minimal OS privileges.

⚠️ Alpine Considerations

Advantages

  • Extremely small (~7 MB)
  • Fast pull and build times
  • Includes shell for debugging
  • Active security team
  • apk is fast and simple

Watch Out For

  • musl vs glibc — some apps expect glibc
  • DNS resolution — musl resolves differently
  • Python wheels — may need compilation
  • Node.js native modules — compatibility issues
  • Performance — musl allocator can be slower
musl DNS resolution is a classic Alpine gotcha. musl does not support the search/ndots directives the same way glibc does, which can cause issues in Kubernetes where DNS relies heavily on search paths.

🧭 When to Use Each Base Image

Use scratch when...

  • Statically compiled Go/Rust binary
  • No runtime dependencies
  • Maximum security posture

Use Distroless when...

  • You need a runtime (Java, Python, Node)
  • You want no shell access in production
  • You need CA certificates or TLS

Use Alpine when...

  • You need shell access for debugging
  • Your app is compatible with musl
  • You need to install system packages

Use Ubuntu/Debian when...

  • Legacy app with specific glibc needs
  • Complex native dependencies
  • Still harden it!
The decision tree should default to scratch/distroless and only fall back to Alpine or full distros when there is a concrete requirement.

🏗️ Section 3

Multi-Stage Builds

Multi-stage builds are one of the most effective techniques for producing small, secure production images.

🏗️ The Problem: Build Tools in Production

Single-Stage Build

  • Compilers (gcc, javac, go)
  • Build tools (make, maven, npm)
  • Development headers
  • Test frameworks
  • Source code

All shipped to production!

What Production Actually Needs

  • Compiled binary or bundled app
  • Runtime dependencies only
  • Configuration files

Minimal attack surface.

Build tools in production are unnecessary attack surface. A Go compiler, for example, could be used by an attacker to compile exploit tools inside the container.

🏗️ Multi-Stage Build: Go Example

FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o /app/server

FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=builder /app/server /server
ENTRYPOINT ["/server"]
Result: Build stage has Go toolchain (~300 MB). Final image has only the static binary (~10-15 MB).
Walk through each stage. The builder stage has the full Go toolchain. The final stage copies only the compiled binary into a distroless image. CGO_ENABLED=0 ensures a fully static binary.

☕ Multi-Stage Build: Java Example

FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

FROM gcr.io/distroless/java21-debian12:nonroot
COPY --from=builder /app/target/myapp.jar /app/myapp.jar
ENTRYPOINT ["java", "-jar", "/app/myapp.jar"]
Build stage: ~800 MB (JDK + Maven + dependencies)
Final image: ~220 MB (JRE only, no build tools)
Java builds are a great multi-stage candidate because the build toolchain (JDK + Maven/Gradle) is very large compared to the runtime (JRE only).

🐍 Multi-Stage Build: Python Example

FROM python:3.12-slim AS builder
WORKDIR /app
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .

FROM gcr.io/distroless/python3-debian12:nonroot
COPY --from=builder /opt/venv /opt/venv
COPY --from=builder /app /app
ENV PATH="/opt/venv/bin:$PATH"
WORKDIR /app
ENTRYPOINT ["python", "main.py"]
Python multi-stage builds use a virtual environment in the builder, then copy the entire venv to the final image. This avoids shipping pip, setuptools, and wheel in production.

📦 Multi-Stage Build: Node.js Example

FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .

FROM gcr.io/distroless/nodejs20-debian12:nonroot
WORKDIR /app
COPY --from=builder /app /app
CMD ["server.js"]
Tip: Use npm ci instead of npm install for deterministic, reproducible builds.
npm ci uses the lockfile exactly and is faster in CI environments. The distroless nodejs image contains only the Node runtime, not npm or any build tools.

📉 Size Reduction with Multi-Stage Builds

Application Single Stage Multi-Stage Reduction
Go API Server 350 MB 12 MB 97%
Java Spring Boot 800 MB 220 MB 73%
Python Flask 450 MB 120 MB 73%
Node.js Express 350 MB 130 MB 63%
Smaller images = faster pulls, less storage, fewer CVEs, and reduced blast radius.
These are approximate real-world numbers. Go achieves the most dramatic reduction because it compiles to a single static binary.

📝 Section 4

Dockerfile Best Practices

Even with the right base image and multi-stage builds, poor Dockerfile practices can introduce vulnerabilities.

📌 Pin Base Image Versions

Tags are mutable. The same tag can point to different images over time.

# Bad - tag can change
FROM python:3.12

# Better - pinned minor version
FROM python:3.12.2-slim

# Best - pinned with SHA256 digest
FROM python:3.12.2-slim@sha256:abcdef1234567890...
Why digest pinning? Even a specific version tag like 3.12.2-slim can be overwritten. A digest guarantees the exact image bytes.
Digest pinning is the gold standard for reproducible and secure builds. Pair it with automated dependency update tools like Renovate or Dependabot to keep images current.

📦 Minimize Layers

RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y wget
RUN rm -rf /var/lib/apt/lists/*
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
      curl \
      wget && \
    rm -rf /var/lib/apt/lists/*
  • Combine related RUN commands with &&
  • Clean up caches in the same layer they are created
  • Use --no-install-recommends to skip optional packages
If you run apt-get update in one layer and rm in another, the cache files still exist in the earlier layer. Combining into a single RUN ensures the cleanup actually reduces image size.

🚫 Use .dockerignore

Prevent sensitive and unnecessary files from entering the build context:

# .dockerignore
.git
.gitignore
.env
*.md
Dockerfile
docker-compose*.yml
node_modules
__pycache__
.vscode
.idea
*.secret
*.key
*.pem
Without .dockerignore: Your entire directory (including .env, .git, secrets, keys) is sent to the Docker daemon as build context.
The .dockerignore file works like .gitignore. Without it, COPY . . copies everything, including secrets and development files. This is both a security risk and slows builds.

🛡️ Don't Run as Root

# Create a non-root user
RUN groupadd -r appuser && useradd -r -g appuser -s /sbin/nologin appuser

# Set ownership
COPY --chown=appuser:appuser . /app

# Switch to non-root user
USER appuser

ENTRYPOINT ["/app/server"]
Running as root: Container compromise = root access to container filesystem, potential host escape.
Running as non-root: Limits blast radius. Attacker gets only unprivileged access.
By default, Docker runs containers as root. Always add a USER directive. For distroless, use the :nonroot tag variant instead.

🔒 No Secrets in Build Args or Layers

# Secrets visible in image history
ARG DB_PASSWORD
ENV DB_PASSWORD=$DB_PASSWORD
COPY credentials.json /app/
RUN curl -H "Token: $API_KEY" ...
# Use BuildKit secrets
RUN --mount=type=secret,id=db_pass \
    cat /run/secrets/db_pass | \
    my-setup-script

# Mount secrets at runtime
# via Kubernetes Secrets or
# external secret managers
Build args and ENV values are stored in image metadata. Use Docker BuildKit --mount=type=secret or inject secrets at runtime via Kubernetes Secrets.
docker build --secret id=db_pass,src=./db_password.txt passes the secret to the build without persisting it in any layer. At runtime, use Kubernetes Secrets mounted as volumes or environment variables.

📋 COPY vs ADD

COPY (Preferred)

  • Simple file/directory copy
  • Predictable behavior
  • No hidden magic
COPY app.py /app/
COPY --chown=appuser:appuser . /app

ADD (Use Sparingly)

  • Auto-extracts tar archives
  • Supports remote URLs
  • Unpredictable side effects
# Only use for tar extraction
ADD app.tar.gz /app/
# Never use for remote URLs
# ADD http://... (use curl instead)
ADD can fetch remote URLs and auto-extract tarballs, which makes it harder to reason about. COPY is explicit and should be the default choice. ADD with URLs is especially risky because the content could change.

⚡ Order Layers for Cache Efficiency

FROM node:20-alpine
WORKDIR /app
# Any code change invalidates npm install
COPY . .
RUN npm ci
USER node
CMD ["node", "server.js"]
FROM node:20-alpine
WORKDIR /app
# Dependencies change rarely
COPY package*.json ./
RUN npm ci
# Code changes frequently
COPY . .
USER node
CMD ["node", "server.js"]
Rule: Place instructions that change least frequently at the top. Dependencies before source code. OS packages before application packages.
Docker cache invalidation cascades: once a layer changes, all subsequent layers must be rebuilt. Ordering layers from least to most frequently changing maximizes cache hits.

🚨 Bad vs Good Dockerfile: Full Comparison

FROM python:latest
COPY . /app
WORKDIR /app
RUN pip install -r requirements.txt
ARG API_KEY
ENV API_KEY=$API_KEY
EXPOSE 8080
CMD python app.py
FROM python:3.12.2-slim@sha256:abc...
RUN groupadd -r app && \
    useradd -r -g app app
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir \
    -r requirements.txt
COPY --chown=app:app . .
USER app
EXPOSE 8080
ENTRYPOINT ["python", "app.py"]
Walk through every difference: pinned version, non-root user, no secrets in args, layer ordering, no-cache-dir, ENTRYPOINT exec form vs CMD shell form, chown on COPY.

🔍 Dockerfile Linting with Hadolint

# Run Hadolint on your Dockerfile
docker run --rm -i hadolint/hadolint < Dockerfile

# Example output:
# DL3007 warning: Using latest is prone to errors
# DL3008 warning: Pin versions in apt-get install
# DL3009 info: Delete apt-get lists after installing
# DL3025 warning: Use arguments JSON notation for CMD
Hadolint checks Dockerfiles against best practices. Integrate it into your CI pipeline to catch issues before they reach production.
Hadolint parses Dockerfiles and applies rules from the Dockerfile best practices documentation. It also shells out to ShellCheck for RUN instruction linting.

🔎 Section 5

Image Scanning

Scanning images for known vulnerabilities is essential. We cover SBOM generation and vulnerability scanning.

🔎 Why Scan Images?

  • Known vulnerabilities (CVEs) in OS packages and application dependencies
  • Malware detection
  • License compliance
  • Configuration issues (running as root, exposed secrets)
  • Compliance requirements (SOC 2, PCI-DSS, HIPAA)

Shift Left

Scan images in the CI/CD pipeline before they reach the registry. Catching a critical CVE in CI costs minutes; catching it in production costs hours or worse.

Image scanning should be automated and continuous. New CVEs are published daily, so images that were clean yesterday may have vulnerabilities today.

📋 SBOM Generation with Syft

A Software Bill of Materials (SBOM) inventories every component in your image.

# Generate SBOM in CycloneDX format
syft packages myimage:latest -o cyclonedx-json > sbom.json

# Generate SBOM in SPDX format
syft packages myimage:latest -o spdx-json > sbom-spdx.json

# Scan a local Dockerfile directory
syft dir:/path/to/project -o cyclonedx-json
Syft (by Anchore) detects packages from OS package managers, language-specific manifests (package.json, go.mod, pom.xml), and binary analysis.
SBOMs are increasingly required by regulation (US Executive Order 14028). CycloneDX and SPDX are the two main standards. Syft supports both.

🛡️ Vulnerability Scanning with Trivy

# Scan image for HIGH and CRITICAL vulnerabilities
trivy image --severity HIGH,CRITICAL myimage:latest

# Scan and fail CI if critical issues found
trivy image --exit-code 1 --severity CRITICAL myimage:latest

# Scan with SBOM output
trivy image --format cyclonedx --output sbom.json myimage:latest

# Scan filesystem (for local development)
trivy fs --severity HIGH,CRITICAL .
Trivy scans container images, filesystems, git repositories, and Kubernetes clusters. It detects vulnerabilities, misconfigurations, and secrets.
Trivy is free, open source, and maintained by Aqua Security. The --exit-code flag makes it easy to use as a CI gate. Demonstrate running trivy against a sample image.

🔍 Vulnerability Scanning with Grype

# Scan an image
grype myimage:latest

# Scan and filter by severity
grype myimage:latest --only-fixed --fail-on critical

# Scan an SBOM (from Syft)
grype sbom:./sbom.json

# Output in table or JSON
grype myimage:latest -o json > results.json
Grype + Syft form a powerful pair: Syft generates the SBOM, Grype scans it for vulnerabilities. Both are open source from Anchore.
Grype can consume SBOMs generated by Syft, so you generate the SBOM once and scan it repeatedly as new vulnerability data becomes available, without needing the original image.

⚙️ Integrating Scanning into CI/CD

# GitHub Actions example
- name: Build image
  run: docker build -t myapp:${{ github.sha }} .

- name: Scan with Trivy
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: myapp:${{ github.sha }}
    format: table
    exit-code: 1
    severity: CRITICAL,HIGH

- name: Generate SBOM
  run: syft packages myapp:${{ github.sha }} -o cyclonedx-json > sbom.json

- name: Upload SBOM
  uses: actions/upload-artifact@v4
  with:
    name: sbom
    path: sbom.json
This GitHub Actions workflow builds, scans, generates an SBOM, and uploads it as an artifact. The exit-code: 1 flag will fail the pipeline if HIGH or CRITICAL vulnerabilities are found.

✍️ Section 6

Image Signing and Verification

Signing images ensures that what you deploy is exactly what you built. This is the foundation of supply chain integrity.

✍️ Why Sign Images?

The Supply Chain Problem

Between building an image and running it in production, many things can go wrong:

  • Registry compromise — attacker replaces your image with a malicious one
  • Man-in-the-middle — image tampered during transit
  • Tag mutation — someone pushes a different image to the same tag
  • Unauthorized builds — images built outside trusted CI pipelines
Image signing creates a cryptographic guarantee: this image was built by a trusted entity and has not been modified.
Image signing is analogous to code signing for software packages. Without it, you are trusting the registry infrastructure completely.

🔑 Cosign: Signing Images

Cosign is part of the Sigstore project for supply chain security.

# Generate a key pair
cosign generate-key-pair

# Sign an image
cosign sign --key cosign.key myregistry/myimage:v1.0

# Sign with annotations
cosign sign --key cosign.key \
  -a "git_sha=$(git rev-parse HEAD)" \
  -a "build_pipeline=github-actions" \
  myregistry/myimage:v1.0
Cosign stores signatures as OCI artifacts alongside the image in the same registry. No separate signature storage needed.
Cosign was designed specifically for containers and OCI artifacts. The signature is stored as a tag in the registry alongside the image, making it easy to distribute.

✅ Cosign: Verifying Images

# Verify an image signature
cosign verify --key cosign.pub myregistry/myimage:v1.0

# Verify with annotation checks
cosign verify --key cosign.pub \
  -a "build_pipeline=github-actions" \
  myregistry/myimage:v1.0

# Output verification details in JSON
cosign verify --key cosign.pub \
  myregistry/myimage:v1.0 | jq .
Kubernetes integration: Use admission controllers (Kyverno, OPA Gatekeeper, Sigstore Policy Controller) to enforce signature verification before pods can run.
Verification should be enforced at the cluster level. The Sigstore Policy Controller can be installed as a Kubernetes admission webhook that rejects unsigned images.

🌐 Keyless Signing with Sigstore

How It Works

  1. Authenticate via OIDC (GitHub, Google, etc.)
  2. Fulcio issues a short-lived certificate
  3. Image signed with ephemeral key
  4. Signature recorded in Rekor transparency log
  5. Key material discarded
# Keyless signing (opens browser)
cosign sign myregistry/myimage:v1.0

# Keyless verification
cosign verify \
  --certificate-identity=user@example.com \
  --certificate-oidc-issuer=https://accounts.google.com \
  myregistry/myimage:v1.0
Keyless signing eliminates key management. Identity is proven via OIDC; provenance is recorded in a public transparency log.
Keyless signing is ideal for CI/CD pipelines where managing long-lived keys is a burden. GitHub Actions, for example, provides OIDC tokens that Fulcio accepts directly.

📜 Notary v2 Overview

  • OCI-native signature specification (part of Notation project)
  • Signatures stored as OCI reference types in the registry
  • Plugin-based architecture for signing providers
  • Supported by Azure Container Registry and AWS Signer
# Sign with Notation (Notary v2 CLI)
notation sign myregistry/myimage@sha256:abc123...

# Verify
notation verify myregistry/myimage@sha256:abc123...
Cosign vs Notation: Cosign is more widely adopted in the open-source ecosystem. Notation has stronger enterprise/cloud provider backing. Both are valid choices.
Notary v2 (Notation) is the CNCF project for OCI artifact signing. It is particularly well-integrated with Azure and AWS. In practice, many teams use Cosign for its simplicity and Sigstore integration.

🏢 Section 7

Registry Security

The registry is the gatekeeper between your CI/CD pipeline and your Kubernetes cluster. Securing it is critical.

🏢 Private Registry Best Practices

  • Authentication required — no anonymous pulls
  • TLS everywhere — encrypt registry traffic
  • RBAC — separate push/pull permissions by team
  • Vulnerability scanning — scan on push (Harbor, ECR, GCR, ACR)
  • Content trust — enforce signed images
  • Image retention policies — clean up old, untagged images
  • Geo-replication — reduce pull latency and improve availability
  • Audit logging — track who pushed/pulled what
Harbor is a popular CNCF-graduated project for self-hosted registries. Cloud providers offer managed registries with built-in scanning and RBAC.

🔄 Image Pull Policies

Policy Behavior Security Impact
Always Pull every time a pod starts Always gets latest (patched) image
IfNotPresent Pull only if not cached locally May use stale/vulnerable cached image
Never Only use pre-loaded images Air-gapped scenarios only 
spec:
  containers:
  - name: app
    image: myregistry/myimage@sha256:abc123...  # Digest pinned
    imagePullPolicy: Always                      # Always verify
Combine Always pull policy with digest-pinned images for maximum security and reproducibility.
imagePullPolicy: Always with a digest-pinned image ensures that every pod start verifies the image exists in the registry. With tags, Always re-pulls the image each time, which can slow pod startup.

📌 Digest Pinning vs Tag Pinning

Tag Pinning

image: myapp:v1.2.3
  • Human-readable
  • Tags are mutable
  • Can be overwritten
  • No guarantee of content

Digest Pinning

image: myapp@sha256:a1b2c3...
  • Immutable reference
  • Cryptographic guarantee
  • Cannot be overwritten
  • Exact content known
Best practice: Use tools like crane digest or skopeo inspect to resolve tags to digests in your deployment pipelines.
In practice, teams often use tags in Dockerfiles for readability and resolve them to digests in Kubernetes manifests via automation.

🚧 ImagePolicyWebhook Admission Controller

Kubernetes can enforce image policies at admission time:

# Example: Kyverno policy to require image signatures
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signatures
spec:
  validationFailureAction: Enforce
  rules:
  - name: verify-cosign-signature
    match:
      any:
      - resources:
          kinds: ["Pod"]
    verifyImages:
    - imageReferences: ["myregistry/*"]
      attestors:
      - entries:
        - keys:
            publicKeys: |-
              -----BEGIN PUBLIC KEY-----
              ...
              -----END PUBLIC KEY-----
Kyverno and OPA Gatekeeper are the two most popular policy engines for Kubernetes. This Kyverno policy rejects any pod that uses an unsigned image from the specified registry.

📦 OCI Registry Security Features

  • Immutable tags — prevent tag overwriting (ECR, Harbor)
  • Vulnerability scanning on push — automatic scan when images are pushed
  • Quarantine — hold images until scan passes (ACR)
  • Token-scoped access — short-lived pull tokens per namespace
  • OCI reference types — attach signatures, SBOMs, attestations to images
  • Webhook notifications — alert on push, scan results, policy violations
Modern registries are more than storage — they are active participants in your security posture.
ECR immutable tags and ACR quarantine are particularly powerful enterprise features. Harbor provides similar capabilities as an open-source alternative.

🔗 Section 8

Supply Chain Security

Supply chain security is the overarching framework that ties together everything we have covered: base images, builds, scanning, signing, and registry controls.

🔗 SLSA Framework (Supply-chain Levels for Software Artifacts)

Level Requirements Protection
SLSA 1 Build process documented Provenance exists
SLSA 2 Hosted build service, authenticated provenance Tamper resistance after build
SLSA 3 Hardened build platform, non-falsifiable provenance Tamper resistance of build process
SLSA 4 Hermetic, reproducible builds, two-person review Highest confidence in integrity
Goal: Progress toward SLSA 3+ for production workloads. Most organizations start at SLSA 1-2 and incrementally improve.
SLSA (pronounced "salsa") is a framework from Google that provides a common vocabulary for supply chain security levels. It helps organizations measure and improve their supply chain posture.

📋 SBOM Requirements and Standards

SPDX (Linux Foundation)

  • ISO/IEC 5962:2021 standard
  • Focus on license compliance
  • Comprehensive package metadata
  • Formats: JSON, RDF, tag-value

CycloneDX (OWASP)

  • Security-focused SBOM standard
  • Vulnerability references (VEX)
  • Dependency graph support
  • Formats: JSON, XML, protobuf
US Executive Order 14028 requires SBOMs for software sold to the federal government. Both SPDX and CycloneDX are accepted formats.
CycloneDX is generally preferred for security use cases because of its VEX (Vulnerability Exploitability eXchange) support. SPDX is more established for license compliance.

📜 Provenance and Attestations

Attestations are signed statements about how an artifact was produced.

# Generate and attach SLSA provenance with cosign
cosign attest --predicate provenance.json \
  --type slsaprovenance \
  --key cosign.key \
  myregistry/myimage:v1.0

# Verify attestation
cosign verify-attestation \
  --type slsaprovenance \
  --key cosign.pub \
  myregistry/myimage:v1.0

What Provenance Captures

  • Builder — what CI system built the image
  • Source — git repo, commit SHA, branch
  • Build config — Dockerfile, build args
  • Materials — base images, dependencies
Provenance tells you the complete story of how an image was created. Combined with signature verification, it provides strong guarantees about image integrity and origin.

🔏 In-Toto Framework

In-Toto defines and verifies the entire supply chain layout, not just individual artifacts.

Key Concepts

  • Layout — defines expected supply chain steps
  • Steps — individual actions (clone, build, scan, sign)
  • Links — signed evidence that each step occurred
  • Functionaries — authorized actors for each step

Supply Chain Layout

Step 1: Developer writes code
Step 2: CI builds image
Step 3: Scanner finds no critical CVEs
Step 4: Signer signs image
Step 5: Deployer pushes to cluster

Each step produces a signed "link" metadata file.

In-Toto is a CNCF project used by companies like Datadog and is integrated into tools like SLSA and Sigstore. It is particularly useful when multiple teams are involved in the software delivery process.

🎯 Putting It All Together

Secure Image Pipeline

  1. Build — Multi-stage, minimal base, pinned dependencies
  2. Lint — Hadolint checks Dockerfile best practices
  3. Scan — Trivy/Grype for vulnerabilities, fail on CRITICAL
  4. Generate SBOM — Syft produces CycloneDX/SPDX
  5. Sign — Cosign signs image and attaches attestations
  6. Push — Image + signature + SBOM to private registry
  7. Admit — Kyverno/OPA verifies signature before deployment
  8. Monitor — Continuous scanning for new CVEs
This is the complete pipeline. Emphasize that each step adds a layer of security, and the combination provides defense in depth for the image supply chain.

📝 Key Takeaways

  • 📦 Smaller images = fewer vulnerabilities. Use distroless or scratch when possible.
  • 🏗️ Multi-stage builds eliminate build tools from production images, often reducing size by 70-97%.
  • 📌 Pin base images by SHA256 digest for reproducible, tamper-resistant builds.
  • 🔒 Never embed secrets in image layers or build args. Use BuildKit secrets or runtime injection.
  • 🔎 Scan every image in CI/CD with Trivy or Grype. Fail builds on critical vulnerabilities.
  • ✍️ Sign images with cosign and verify signatures at admission with policy engines.
  • 🔗 Generate SBOMs for every production image. It is increasingly a regulatory requirement.
  • 🛡️ Enforce policies at the registry and admission controller level for defense in depth.
Review each takeaway. These are the actionable items students should implement in their organizations.

📖 Quick Reference: Essential Commands

# Scan image for vulnerabilities
trivy image --severity HIGH,CRITICAL myimage:latest

# Generate SBOM
syft packages myimage:latest -o cyclonedx-json > sbom.json

# Scan SBOM for vulnerabilities
grype sbom:./sbom.json

# Sign image with cosign
cosign sign --key cosign.key myregistry/myimage:v1.0

# Verify image signature
cosign verify --key cosign.pub myregistry/myimage:v1.0

# Lint Dockerfile
docker run --rm -i hadolint/hadolint < Dockerfile

# Get image digest
crane digest myregistry/myimage:v1.0
