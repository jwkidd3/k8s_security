🛡️ Section 1

Why Kubernetes Security Matters

This section establishes the urgency. We want the audience to understand that Kubernetes security is not optional — it is a critical operational concern backed by real data and real-world incidents.

Kubernetes Adoption Is Massive

96% of organizations use or evaluate Kubernetes
67% have delayed deployments due to security concerns
59% experienced a security incident in their container environment
The implication: Kubernetes is everywhere, but most teams lack the security expertise to protect their clusters effectively.
Source: CNCF Annual Survey and Red Hat State of Kubernetes Security reports. The 67% figure is particularly striking — security is the number one barrier to Kubernetes adoption.

The Stakes Are High

⚠️ What Can Go Wrong

  • Cryptojacking of compute resources
  • Data exfiltration from pods and volumes
  • Lateral movement across namespaces
  • Supply chain compromise
  • Complete cluster takeover

✅ What Security Enables

  • Confident production deployments
  • Regulatory compliance (SOC2, PCI, HIPAA)
  • Reduced blast radius of incidents
  • Faster incident response
  • Trust from customers and stakeholders

Real-World Kubernetes Breaches

Incident Year Attack Vector Impact
Tesla Cryptojacking 2018 Exposed Kubernetes dashboard (no password) AWS resources used for cryptocurrency mining
Capital One 2019 SSRF to cloud metadata service 106 million customer records exposed
SolarWinds 2020 Supply chain — compromised build pipeline 18,000+ organizations affected
Siloscape 2021 Windows container escape to host Backdoor into Kubernetes clusters
Each of these incidents highlights a different class of vulnerability. Tesla shows the danger of exposed dashboards. Capital One demonstrates cloud metadata attacks. SolarWinds is the canonical supply chain attack. Siloscape was the first known malware targeting Kubernetes clusters specifically.

Case Study: Tesla Cryptojacking

What Happened

  1. Kubernetes dashboard exposed without authentication
  2. Attackers discovered AWS credentials in pod environment
  3. Deployed cryptomining containers across the cluster
  4. Used evasion techniques to avoid detection
Root cause: A single misconfiguration — no authentication on the dashboard.

Lessons Learned

  • Never expose dashboards without authentication
  • Use network policies to restrict egress
  • Monitor for unexpected resource consumption
  • Avoid storing secrets in environment variables
  • Implement pod security policies

The Shared Responsibility Model

Key Insight: Cloud providers secure the infrastructure of Kubernetes, but you are responsible for security in Kubernetes.

Cloud Provider

  • Physical infrastructure
  • Control plane availability (managed K8s)
  • Host OS patching
  • Network infrastructure

Shared

  • Node configuration
  • Kubernetes version upgrades
  • Network plugin configuration
  • Encryption at rest

Your Responsibility

  • RBAC policies
  • Network policies
  • Pod security
  • Image security
  • Secrets management
  • Application code
This is crucial context for the entire course. Most of what we will cover falls in the "your responsibility" column. Even with managed Kubernetes services like EKS, GKE, or AKS, the majority of security concerns remain with the customer.

🌍 Section 2

The Kubernetes Threat Landscape

Common Attack Vectors

External Threats

  • Exposed API servers
  • Vulnerable container images
  • Misconfigured RBAC
  • Unprotected dashboards
  • Ingress misconfigurations

Internal Threats

  • Lateral movement between pods
  • Privilege escalation
  • Container escape to host
  • Service account abuse
  • Namespace hopping

Supply Chain

  • Malicious base images
  • Compromised dependencies
  • Tampered CI/CD pipelines
  • Typosquatting in registries
  • Unverified Helm charts
These three categories represent distinct threat models. External threats are what most people think of first, but internal threats (east-west traffic) and supply chain threats are often more dangerous because they bypass perimeter defenses.

Kubernetes Attack Surface

🎯 Control Plane Targets

  • API Server — front door to the cluster; authn/authz critical
  • etcd — stores all cluster state, secrets in plaintext by default
  • Controller Manager — runs with elevated privileges
  • Scheduler — pod placement can be manipulated
  • Cloud Controller — access to cloud provider APIs

💻 Worker Node Targets

  • Kubelet — node agent; anonymous access can be enabled
  • Container Runtime — escape vulnerabilities (CVE-2019-5736)
  • Pods/Containers — application-level vulnerabilities
  • Kube-proxy — network rule manipulation
  • Host Filesystem — hostPath mounts expose node
Every component here is a potential target. The API server is the most critical because it is the single point of entry. etcd is the crown jewel — if an attacker gets access to etcd, they own the entire cluster. On the worker side, kubelet API and container runtime vulnerabilities are the most common attack paths.

Typical Attack Flow

Reconnaissance → Initial Access → Execution → Persistence → Privilege Escalation → Lateral Movement → Exfiltration 

# 1. Attacker discovers exposed kubelet API
curl -sk https://target-node:10250/pods

# 2. Executes commands inside a running container
curl -sk https://target-node:10250/run/default/web-app/web \
  -d "cmd=cat /var/run/secrets/kubernetes.io/serviceaccount/token"

# 3. Uses stolen service account token to access API server
kubectl --token=$STOLEN_TOKEN --server=https://api-server:6443 \
  get secrets --all-namespaces

# 4. Pivots to other namespaces and exfiltrates data
kubectl --token=$STOLEN_TOKEN exec -it db-pod -- pg_dump mydb
Walk through each step. This is a realistic attack scenario that exploits an exposed kubelet API, steals a service account token, and uses it to access secrets cluster-wide. Each step in this chain can be mitigated — and we will cover how throughout the course.

🔓 Section 3

Why Isn't My Cluster Hardened by Default?

Kubernetes Design Philosophy

"Kubernetes prioritizes developer velocity, composability, and ease of adoption — security is intentionally left as a configuration responsibility."

Why This Matters

  • Low barrier to entry accelerated adoption
  • Flexible security allows diverse use cases
  • One-size-fits-all security doesn't work across industries
  • Operators are expected to harden for their environment

The Consequence

  • Default clusters are functionally wide open
  • Most tutorials skip security configuration
  • Development clusters get promoted to production as-is
  • Teams don't know what they don't know

Insecure Defaults in Kubernetes

Default Behavior Security Risk What Should Be Done
No network policies All pods can communicate freely Apply deny-all then whitelist
SA tokens auto-mounted Every pod gets API access Set automountServiceAccountToken: false
No pod security standards Privileged containers allowed Enforce Restricted PSS
No audit logging No visibility into API activity Enable audit policy
etcd not encrypted at rest Secrets stored in plaintext Enable encryption provider
No resource limits DoS via resource exhaustion Set requests and limits
Go through each row. These are the top six insecure defaults that every team should address immediately. We will cover every single one of these in depth in later modules.

Insecure Pod: What NOT To Do

apiVersion: v1
kind: Pod
metadata:
  name: insecure-app
spec:
  containers:
  - name: app
    image: myapp:latest          # Mutable tag, no digest
    ports:
    - containerPort: 8080
    securityContext:
      privileged: true           # Full host access
      runAsUser: 0               # Running as root
    volumeMounts:
    - mountPath: /host
      name: host-fs
  volumes:
  - name: host-fs
    hostPath:
      path: /                    # Entire host filesystem mounted
  # No resource limits
  # Service account token auto-mounted
  # No readOnlyRootFilesystem
Vulnerabilities: privileged mode, root user, hostPath mount of /, mutable image tag, no resource limits, writable filesystem, auto-mounted SA token.

Hardened Pod: Security Best Practices

apiVersion: v1
kind: Pod
metadata:
  name: hardened-app
spec:
  automountServiceAccountToken: false    # No API access
  securityContext:
    runAsNonRoot: true                   # Enforce non-root
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
    seccompProfile:
      type: RuntimeDefault               # Restrict syscalls
  containers:
  - name: app
    image: myapp@sha256:abc123...        # Immutable digest
    ports:
    - containerPort: 8080
    securityContext:
      allowPrivilegeEscalation: false    # No privilege escalation
      readOnlyRootFilesystem: true       # Immutable filesystem
      capabilities:
        drop: ["ALL"]                    # Drop all capabilities
    resources:
      requests:
        memory: "128Mi"
        cpu: "100m"
      limits:
        memory: "256Mi"
        cpu: "500m"
    volumeMounts:
    - mountPath: /tmp
      name: tmp-vol
  volumes:
  - name: tmp-vol
    emptyDir: {}                         # Ephemeral, no host access
Walk through each security control line by line. This is the reference template for a production-ready pod. In the lab, students will compare insecure and hardened configurations.

Before and After: Key Differences

Insecure Defaults

  • privileged: true
  • runAsUser: 0 (root)
  • image: myapp:latest
  • hostPath mount of /
  • No resource limits
  • SA token auto-mounted
  • Writable root filesystem
  • All Linux capabilities

Hardened Configuration

  • privileged: false (implicit)
  • runAsNonRoot: true
  • image: myapp@sha256:...
  • emptyDir for temp storage only
  • CPU and memory limits set
  • automountServiceAccountToken: false
  • readOnlyRootFilesystem: true
  • capabilities.drop: ["ALL"]
Every hardening step reduces the blast radius of a compromise.

📚 Section 4

Security Terminology and Concepts

Core Security Concepts

Attack Surface

The sum of all points where an attacker can try to enter or extract data from a system. In Kubernetes, this includes the API server, kubelet, container runtime, network, and every running workload.

Defense in Depth

Layered security controls so that if one layer fails, others still protect the system. No single control is considered sufficient on its own.

Principle of Least Privilege

Grant only the minimum permissions necessary for a user, service account, or workload to perform its intended function. Nothing more.

Zero Trust

Never trust, always verify. Every request must be authenticated and authorized regardless of its origin — even from inside the cluster network.

Defense in Depth in Kubernetes

Principle: Multiple independent layers of security ensure no single failure leads to a full compromise.

The 4Cs of Cloud Native Security

Cloud
Cluster
Container
Code

Cloud

Infrastructure security: IAM, network, encryption, physical security

Cluster

K8s components: RBAC, network policies, admission control, audit

Container

Image scanning, minimal base images, non-root, read-only FS

Code

Secure coding, dependency scanning, SAST/DAST, secrets handling

The 4Cs model from the Kubernetes documentation shows that each layer depends on the security of the layers beneath it. You cannot have secure code running in an insecure container, on an insecure cluster, on an insecure cloud.

Immutable Infrastructure

Definition: Servers and containers are never modified after deployment. When changes are needed, the entire unit is replaced with a new version.

Mutable (Traditional)

  • SSH into production servers
  • Apply patches in place
  • Configuration drift over time
  • Hard to reproduce state
  • Attacker changes persist

Immutable (Cloud Native)

  • Build new image for every change
  • Deploy as replacement pods
  • Consistent, reproducible state
  • Read-only root filesystem
  • Attacker changes lost on restart
Security benefit: Immutable containers with read-only filesystems make it significantly harder for attackers to persist malware or modify application behavior.

The CIS Benchmark is the most widely adopted standard. It is updated regularly to match new Kubernetes versions. In Lab 1, students will run kube-bench against their lab cluster and interpret the results.

Running kube-bench

# Run kube-bench as a Kubernetes Job
kubectl apply -f https://raw.githubusercontent.com/aquasecurity/kube-bench/main/job.yaml

# View results
kubectl logs job/kube-bench

# Or run directly on a node
kube-bench run --targets=master,node

# Example output:

Pod Security Standards (PSS)

The built-in Kubernetes mechanism for enforcing pod security (replacing the deprecated PodSecurityPolicy).

Privileged

Unrestricted policy. Provides the widest possible level of permissions. Intended for system-level workloads managed by trusted users.

Use case: kube-system namespace, CNI plugins, monitoring agents

Baseline

Minimally restrictive policy. Prevents known privilege escalations while remaining easy to adopt. Good starting point.

Blocks: hostNetwork, hostPID, privileged containers, hostPath

Restricted

Heavily restricted policy. Follows current pod hardening best practices. The target for all production workloads.

Requires: non-root, drop ALL caps, seccomp, read-only FS
PSS replaced PodSecurityPolicy starting in Kubernetes 1.25. There are three enforcement modes: enforce (reject), audit (log), and warn (show warning). Recommended approach is to start with warn/audit and move to enforce once workloads are compliant.

Enforcing Pod Security Standards

# Apply Restricted standard to a namespace
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    # Enforcement modes: enforce, audit, warn
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/audit-version: latest
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: latest
# Test what would be blocked
kubectl label --dry-run=server --overwrite ns production \
  pod-security.kubernetes.io/enforce=restricted

# Warning: would violate PodSecurity "restricted:latest":
# allowPrivilegeEscalation != false, unrestricted capabilities, ...

Key Takeaways

  • Kubernetes adoption is near-universal, but most clusters are inadequately secured due to insecure defaults and knowledge gaps
  • Real-world breaches — from Tesla to Siloscape — demonstrate that Kubernetes security failures have material business impact
  • Defense in depth is essential: no single control is sufficient; security must be layered across the 4Cs (Code, Container, Cluster, Cloud)
  • Kubernetes is intentionally not hardened out of the box — operators must proactively configure security controls
  • Industry frameworks (CIS Benchmark, NSA/CISA Guide, NIST) and tools (kube-bench, Trivy, Falco) provide actionable guidance
  • Pod Security Standards (Privileged, Baseline, Restricted) are the built-in mechanism for enforcing workload security
