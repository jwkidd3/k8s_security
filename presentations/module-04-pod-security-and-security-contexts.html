🛡️ Section 1

Why Pod Hardening Matters

Let's start with the motivation. Why is hardening pods one of the most impactful things you can do for cluster security?

💥 Pods Are the Blast Radius

A compromised pod is the attacker's foothold inside your cluster.
  • Pods are the smallest deployable unit in Kubernetes
  • Every container runs inside a pod — this is where workloads live
  • A pod compromise gives an attacker access to:
    • Network namespace (lateral movement)
    • Service account tokens (API access)
    • Filesystem and environment variables (secrets)
    • Node resources (escape vectors)
When we talk about blast radius, we mean: what can an attacker reach once they compromise a single pod? Without hardening, the answer is often far too much.

🔒 Principle of Least Privilege for Pods

Apply least privilege across three dimensions:

Identity

  • Non-root user
  • Scoped service account
  • No auto-mounted token

Runtime

  • Minimal capabilities
  • Seccomp / AppArmor
  • Read-only filesystem

Resources

  • CPU & memory limits
  • Ephemeral storage limits
  • Namespace quotas
These three categories — identity, runtime, and resources — frame how we think about pod hardening. Every setting we cover today falls into one of these buckets.

⚠️ The Dangerous Default Pod

A pod with no security settings applied:

  • Runs as root (UID 0) inside the container
  • Full default capabilities granted by the container runtime
  • Read-write filesystem — attacker can modify binaries
  • Auto-mounted service account token at /var/run/secrets/
  • No seccomp profile — all system calls available
  • No resource limits — can consume entire node
  • Privilege escalation allowed by default

Every one of these defaults is a security gap we will close in this module.

This is the starting point for any pod you create without explicit security configuration. It's essentially an open door. Our job is to systematically close each one.

🔍 Before vs After Hardening

Default Pod

apiVersion: v1
kind: Pod
metadata:
  name: insecure-app
spec:
  containers:
  - name: app
    image: nginx:latest

Root user, full caps, r/w FS, auto-mounted token, no limits

Hardened Pod

spec:
  automountServiceAccountToken: false
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
On the left, a typical pod definition. On the right, the same pod with hardening applied. We'll build up to that fully hardened spec step by step.

🔧 Section 2

SecurityContext Deep Dive

The SecurityContext is your primary tool for configuring pod and container security settings. Let's explore it in detail.

📊 Pod-Level vs Container-Level SecurityContext

Pod-Level spec.securityContext

  • runAsUser / runAsGroup
  • runAsNonRoot
  • fsGroup
  • seccompProfile
  • sysctls

Applies to all containers unless overridden

Container-Level securityContext

  • Everything from pod-level, plus:
  • allowPrivilegeEscalation
  • readOnlyRootFilesystem
  • capabilities
  • privileged

Overrides pod-level settings; controls specific containers

Rule of thumb: Set shared defaults at the pod level; set container-specific restrictions at the container level.
Container-level settings override pod-level settings when both are specified. Some settings, like capabilities and readOnlyRootFilesystem, only exist at the container level.

📋 Critical SecurityContext Fields

Field Level Purpose Recommended
runAsNonRoot Pod / Container Reject containers running as UID 0 true
runAsUser Pod / Container Explicit UID for container process 1000+
runAsGroup Pod / Container Explicit GID for container process 3000+
fsGroup Pod GID applied to mounted volumes 2000+
readOnlyRootFilesystem Container Prevent writes to container filesystem true
allowPrivilegeEscalation Container Block setuid / setgid binaries false
privileged Container Full host access (never use in prod) false
This table is your cheat sheet. Every production pod should set all of these fields. The "recommended" column represents security best practice.

🔐 Disabling Auto-Mounted Tokens

By default, every pod gets a service account token mounted at:

/var/run/secrets/kubernetes.io/serviceaccount/token
Problem: If a container is compromised, the attacker can use this token to call the Kubernetes API. 
# Pod-level — disable for the entire pod
spec:
  automountServiceAccountToken: false

# Or ServiceAccount-level — disable for all pods using this SA
apiVersion: v1
kind: ServiceAccount
metadata:
  name: my-app
automountServiceAccountToken: false
Only mount tokens when the pod genuinely needs Kubernetes API access.
This is one of the simplest and most impactful hardening steps. Most application pods never need to call the Kubernetes API, so there's no reason to mount a token.

🛡️ Full Hardened Pod Specification

apiVersion: v1
kind: Pod
metadata:
  name: hardened-app
spec:
  automountServiceAccountToken: false
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: gcr.io/distroless/static:nonroot
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
    resources:              # always set limits
      limits: { memory: "128Mi", cpu: "250m" }
      requests: { memory: "64Mi", cpu: "125m" }
    # ... volumeMounts, volumes omitted for brevity
This is the gold standard hardened pod. Let's walk through each setting: non-root user, all capabilities dropped, read-only filesystem with a writable tmp via emptyDir, seccomp enabled, service account token disabled, and resource limits set.

📦 Choosing Secure Base Images

Fewer binaries = smaller attack surface. A shell-less container is harder to exploit.

Recommended

  • gcr.io/distroless/static
  • gcr.io/distroless/base
  • cgr.dev/chainguard/*
  • Scratch-based Go binaries

Avoid in Production

  • ubuntu:latest
  • alpine:latest (better, but still has shell)
  • node:latest (full OS included)
  • Any :latest tag
Distroless images from Google or Chainguard images remove shells, package managers, and other tools that attackers rely on during post-exploitation. Always use specific image tags, never latest.

🏛️ Section 3

Pod Security Standards & Admission

Individual pod hardening is essential, but we also need cluster-wide enforcement. That's where Pod Security Standards and the Pod Security Admission controller come in.

🔄 PSS Replaces PodSecurityPolicy

PodSecurityPolicy (PSP) was deprecated in Kubernetes 1.21 and removed in 1.25.
  • Pod Security Standards (PSS) — the policy definitions (what is allowed)
  • Pod Security Admission (PSA) — the built-in controller that enforces PSS
PSS + PSA is built into Kubernetes — no additional installation needed (since v1.25 GA).

Third-party alternatives exist (OPA/Gatekeeper, Kyverno) for more granular policies.

If you're still using PodSecurityPolicy, you need to migrate. PSA is the built-in replacement. It's simpler but less flexible than PSP — for advanced use cases, consider Kyverno or OPA Gatekeeper.

📊 Three Pod Security Standards Levels

Privileged

Unrestricted

  • No restrictions applied
  • Use for system-level workloads
  • kube-system, monitoring agents

Baseline

Prevents known escalations

  • Blocks privileged containers
  • Blocks hostNetwork/hostPID
  • Blocks dangerous capabilities

Restricted

Heavily locked down

  • Must run as non-root
  • Must drop ALL capabilities
  • Seccomp profile required
Think of these as three levels of strictness. Most production application namespaces should target Restricted. Infrastructure namespaces may need Baseline or Privileged.

🔎 Baseline vs Restricted Comparison

Control Baseline Restricted
Privileged containers Blocked Blocked
Host namespaces Blocked Blocked
Host ports Blocked Blocked
Capabilities Dangerous ones blocked Must drop ALL; only NET_BIND_SERVICE allowed
Volume types Most allowed Only core types allowed
Running as root Allowed Blocked (runAsNonRoot required)
Privilege escalation Allowed Blocked
Seccomp profile Not required Required (RuntimeDefault or Localhost)
The key differences are in the bottom half of this table. Restricted requires non-root, blocks privilege escalation, requires seccomp, and limits capabilities much more aggressively.

⚙️ Pod Security Admission Modes

PSA offers three modes of operation per namespace:

enforce

Violations are rejected — pods are not created

audit

Violations are logged in the audit log but pods are allowed

warn

Violations return a warning to the user but pods are allowed

You can combine modes — e.g., enforce Baseline while warning on Restricted to discover what breaks.
This three-mode approach lets you roll out policies gradually. Start with warn, then add audit, and finally enforce once you've fixed all violations.

🏷️ Namespace Labeling for PSA

Apply Pod Security Standards to a namespace via labels:

apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted

Label format:

pod-security.kubernetes.io/<MODE>: <LEVEL>
pod-security.kubernetes.io/<MODE>-version: <VERSION>  # optional, e.g., v1.28
Tip: You can pin to a Kubernetes version to prevent policy changes when upgrading the cluster.
This is all it takes. One kubectl label command or a few lines in your namespace manifest, and the PSA controller enforces the policy for every pod in that namespace.

📈 Progressive Rollout Strategy

Migrate to Restricted enforcement safely:

Phase 1: Observe

labels:
  pod-security.kubernetes.io/warn: restricted
  pod-security.kubernetes.io/audit: restricted

Discover violations — no workloads are blocked

Phase 2: Baseline + Warn Restricted

labels:
  pod-security.kubernetes.io/enforce: baseline
  pod-security.kubernetes.io/warn: restricted

Block the worst violations; warn on remaining gaps

Phase 3: Full Enforcement

labels:
  pod-security.kubernetes.io/enforce: restricted

All pods must comply with the Restricted standard

This phased approach avoids breaking running workloads. Phase 1 lets you collect data, Phase 2 blocks the most dangerous configurations, and Phase 3 locks everything down.

🧩 Section 4

Linux Capabilities

Linux capabilities split the almighty root privilege into granular units. Understanding them is key to building least-privilege containers.

🔑 What Are Linux Capabilities?

Core Concept

Traditional UNIX has two privilege levels: root (UID 0) and everyone else. Linux capabilities divide root's powers into ~40 distinct privileges that can be independently granted or revoked.

  • CAP_NET_BIND_SERVICE — bind to ports below 1024
  • CAP_CHOWN — change file ownership
  • CAP_KILL — send signals to any process
  • CAP_SYS_ADMIN — a catch-all "super" capability
Containers should have only the capabilities they actually need — not the default set.
Think of capabilities like individual permissions. Instead of giving a process full root access, you can give it just the specific powers it needs. CAP_SYS_ADMIN is the most dangerous because it grants a wide range of root-like abilities.

📦 Default Container Runtime Capabilities

Docker and containerd grant these capabilities by default:

  • CAP_CHOWN
  • CAP_DAC_OVERRIDE
  • CAP_FSETID
  • CAP_FOWNER
  • CAP_MKNOD
  • CAP_NET_RAW
  • CAP_SETGID
  • CAP_SETUID
  • CAP_SETFCAP
  • CAP_SETPCAP
  • CAP_NET_BIND_SERVICE
  • CAP_SYS_CHROOT
  • CAP_KILL
  • CAP_AUDIT_WRITE
That's 14 capabilities by default — most workloads need zero or one of these.
Most application containers never need any of these capabilities. The defaults are generous for compatibility, not for security. Notice CAP_NET_RAW in the list — that allows packet spoofing.

🚨 Dangerous Capabilities

Capability Risk
CAP_SYS_ADMIN Mount filesystems, trace processes, BPF — almost equivalent to root
CAP_NET_ADMIN Modify routing, firewall rules, sniff traffic
CAP_SYS_PTRACE Debug/inject into other processes — container escape vector
CAP_DAC_OVERRIDE Bypass all file permission checks
CAP_NET_RAW Craft raw packets — ARP spoofing, DNS poisoning
CAP_SYS_MODULE Load kernel modules — full host compromise
If a workload requests CAP_SYS_ADMIN, question whether it truly needs it. There is almost always a better approach.
These capabilities appear in real CVEs and container escape techniques. CAP_SYS_ADMIN and CAP_SYS_PTRACE are the most common paths to container escape.

✅ Best Practice: Drop ALL, Add Selectively

containers:
- name: web
  securityContext:
    capabilities:
      drop: ["ALL"]
      add: ["NET_BIND_SERVICE"]  # Only if binding port < 1024

Dangerous

capabilities:
  add: ["SYS_ADMIN"]

Default (too broad)

# No capabilities
# section at all
# = 14 caps granted

Minimal

capabilities:
  drop: ["ALL"]
The Restricted PSS level requires dropping ALL capabilities.
This is the single most important capability configuration. Start with drop ALL and only add back specific capabilities your application genuinely needs. Most apps need zero.

🛡️ Section 5

Seccomp and AppArmor

Beyond capabilities, we can restrict which system calls a container can make and enforce mandatory access control profiles. These are deep-defense layers.

🧱 Seccomp Profiles

Secure Computing Mode (seccomp)

Seccomp filters restrict which Linux system calls a container can make. Blocking unnecessary syscalls shrinks the kernel attack surface.

Profile Type Description Use Case
RuntimeDefault Container runtime's built-in profile (~300 of ~380 calls) Recommended default
Localhost Custom profile loaded on the node Fine-grained restriction
Unconfined No seccomp filtering Avoid in production
RuntimeDefault is the pragmatic choice — it blocks dangerous syscalls like ptrace, mount, and reboot while allowing the hundreds of calls most applications need. Custom profiles are more secure but require profiling your application first.

📝 Specifying Seccomp in Pod Spec

Pod-Level (all containers)

spec:
  securityContext:
    seccompProfile:
      type: RuntimeDefault

Container-Level (specific)

containers:
- name: app
  securityContext:
    seccompProfile:
      type: Localhost
      localhostProfile: profiles/app.json
The Restricted PSS level requires either RuntimeDefault or Localhost seccomp profiles.

Custom profiles are stored on nodes at /var/lib/kubelet/seccomp/ by default.

Setting seccomp at the pod level is the simplest approach. If you need a custom profile for a sidecar but RuntimeDefault for the main container, use container-level overrides.

🛡️ AppArmor and SELinux

AppArmor

  • Mandatory Access Control (MAC) for files, network, capabilities
  • Available on Ubuntu/Debian nodes
  • Profiles: runtime/default, localhost/<profile>, unconfined
metadata:
  annotations:
    container.apparmor.security.beta.kubernetes.io/app: runtime/default

SELinux

  • Label-based MAC — default on RHEL/CentOS nodes
  • Configured via seLinuxOptions
securityContext:
  seLinuxOptions:
    level: "s0:c123,c456"
    type: "container_t"
AppArmor and SELinux provide an additional layer of mandatory access control beyond capabilities and seccomp. Use whichever is native to your node operating system.

📦 RuntimeClass — Sandboxed Containers

For workloads requiring stronger isolation than Linux namespaces:

gVisor (runsc)

  • User-space kernel intercepting syscalls
  • Dramatic reduction in kernel attack surface
  • Some performance overhead

Kata Containers

  • Lightweight VMs per pod
  • Hardware-level isolation
  • Higher overhead, strongest isolation
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: runsc
---
apiVersion: v1
kind: Pod
metadata:
  name: sandboxed-app
spec:
  runtimeClassName: gvisor
  containers:
  - name: app
    image: myapp:1.0
RuntimeClass lets you choose the container runtime on a per-pod basis. gVisor is great for untrusted workloads. Kata provides VM-level isolation. Both are production-ready.

💰 Section 6

Resource Limits as Security

Resource management is not just an operational concern — it is a security control. Without limits, a single compromised or buggy pod can take down an entire node.

⚠️ Missing Limits = Denial of Service

A pod without resource limits can:
  • Consume all CPU — starve co-located workloads
  • Exhaust memory — trigger OOM killer on the node
  • Fill disk — crash the kubelet via ephemeral storage
  • Fork bomb — create unlimited processes
Resource limits are a security control, not just a capacity planning tool.
Attackers who gain code execution in a container often use resource exhaustion as a stepping stone — filling disk to corrupt logs, exhausting CPU to mask other activity, or triggering OOM to cause pod restarts.

📊 Resource Requests vs Limits

Requests

  • Scheduling guarantee — node must have this much available
  • Used by the scheduler to place pods
  • Determines QoS class

Limits

  • Hard ceiling — container cannot exceed this
  • CPU: throttled; Memory: OOM killed
  • Prevents noisy-neighbor problems
resources:
  requests:
    memory: "64Mi"
    cpu: "125m"      # 0.125 CPU cores
  limits:
    memory: "128Mi"
    cpu: "250m"      # 0.25 CPU cores
Requests guarantee minimum resources for scheduling. Limits cap maximum usage. Memory limits are hard — exceed them and the container is killed. CPU limits cause throttling, not termination.

🎯 LimitRange — Namespace Defaults

Automatically apply default limits to pods that don't specify them:

apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
  namespace: production
spec:
  limits:
  - type: Container
    default:          # Default limits if not specified
      cpu: "500m"
      memory: "256Mi"
    defaultRequest:   # Default requests if not specified
      cpu: "100m"
      memory: "128Mi"
    max:              # Maximum allowed limits
      cpu: "2"
      memory: "1Gi"
    min:              # Minimum allowed requests
      cpu: "50m"
      memory: "64Mi"
Safety net: LimitRange ensures no pod in the namespace runs without resource constraints.
LimitRange acts as a safety net. Even if a developer forgets to set limits, the namespace defaults kick in. The max and min fields prevent anyone from requesting extreme values.

📊 ResourceQuota — Namespace Totals

Cap the total resources a namespace can consume:

apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-quota
  namespace: production
spec:
  hard:
    requests.cpu: "4"
    requests.memory: "8Gi"
    limits.cpu: "8"
    limits.memory: "16Gi"
    pods: "20"
    persistentvolumeclaims: "10"
LimitRange = per-pod defaults and bounds
ResourceQuota = total namespace budget
ResourceQuota prevents any single team or namespace from consuming a disproportionate share of cluster resources. Combined with LimitRange, you get both per-pod and per-namespace constraints.

🏆 QoS Classes

Kubernetes assigns a QoS class based on resource settings:

Guaranteed

Requests == Limits for all containers

  • Last to be evicted
  • Most predictable performance

Burstable

Requests < Limits (or partially set)

  • Evicted after BestEffort
  • Can burst above requests

BestEffort

No requests or limits set

  • First to be evicted
  • No resource guarantees
For security-critical workloads, always aim for Guaranteed QoS to prevent eviction.
The QoS class determines eviction order under node pressure. BestEffort pods are killed first, Guaranteed pods last. Security-critical workloads should always be Guaranteed.

🧩 Putting It All Together

A fully hardened pod combines all the controls from this module:

Layer Control Setting
Identity Non-root user runAsNonRoot: true, runAsUser: 1000
Identity Service account token automountServiceAccountToken: false
Runtime Capabilities drop: ["ALL"]
Runtime Privilege escalation allowPrivilegeEscalation: false
Runtime Filesystem readOnlyRootFilesystem: true
Runtime System calls seccompProfile: RuntimeDefault
Resources CPU & memory Requests and limits set
Cluster Namespace policy PSA enforce: restricted
This summary table is your checklist. Every production pod should address each of these rows. The combination of pod-level settings and namespace-level PSA enforcement gives you defense in depth.

Duration: 30 minutes

  • Apply Pod Security Admission (PSA) labels to enforce Baseline and Restricted profiles
  • Configure SecurityContext settings: runAsUser, runAsGroup, capabilities
  • Test capability restrictions and seccomp profiles

📌 Key Takeaways

  • Default pods are insecure — they run as root with broad capabilities and no resource limits
  • SecurityContext is your primary hardening tool — set it at both pod and container levels
  • Always disable service account token auto-mounting unless the pod needs API access
  • Drop ALL capabilities and add back only what is strictly required
  • Pod Security Standards (Restricted level) enforce best practices cluster-wide via namespace labels
  • Seccomp RuntimeDefault should be the minimum for every production workload
  • Resource limits are a security control — use LimitRange and ResourceQuota as safety nets
  • Use distroless images to reduce the attack surface inside containers
These are the essential takeaways. If students remember nothing else, they should remember: run as non-root, drop all capabilities, set a seccomp profile, disable the service account token, and set resource limits.
