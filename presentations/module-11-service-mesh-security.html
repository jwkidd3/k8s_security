🌐 Why Service Mesh?

Understanding the problem space

Before diving into tools, we need to understand what problems service meshes solve and when they are the right choice versus simpler alternatives.

Service-to-Service Communication Challenges

Without a Mesh

  • Traffic is unencrypted by default within the cluster
  • No mutual identity verification between services
  • NetworkPolicy is L3/L4 only — no L7 controls
  • No built-in retries, timeouts, or circuit breaking
  • Limited observability into east-west traffic
⚠️ The Flat Network Problem
By default, any pod can talk to any other pod. A compromised workload can freely probe and attack neighboring services.
NetworkPolicy helps, but cannot inspect HTTP methods, paths, or headers.
Kubernetes networking is flat by default — every pod can reach every other pod. NetworkPolicy adds L3/L4 controls but cannot inspect application-layer traffic. A service mesh closes these gaps.

What a Service Mesh Adds

🔒 Security

  • Automatic mTLS
  • Cryptographic identity
  • L7 authorization policies
  • Certificate rotation

🔍 Observability

  • Golden metrics per route
  • Distributed tracing
  • Service dependency maps
  • Real-time traffic tapping

⚙️ Reliability

  • Retries & timeouts
  • Circuit breaking
  • Traffic splitting
  • Load balancing (L7)
The three pillars of a service mesh: security, observability, and reliability. From a security perspective, mTLS and authorization policies are the key features.

When Do You Need a Service Mesh?

✅ Good Fit

  • Many microservices with complex communication patterns
  • Regulatory requirement for encryption in transit
  • Need L7 authorization between services
  • Detailed observability into service-to-service traffic
  • Zero-trust networking requirements

❌ Probably Not Needed

  • Small number of services (<5-10)
  • Simple request patterns (single API gateway)
  • NetworkPolicy provides sufficient control
  • Team lacks operational capacity for another layer
  • Performance overhead is unacceptable
"A service mesh is powerful, but adds operational complexity. Adopt it when the security and observability benefits outweigh that cost."
Not every cluster needs a service mesh. For smaller deployments, NetworkPolicy combined with application-level TLS may be sufficient. Evaluate the trade-offs.

🏗️ Istio & Linkerd Architecture

Data plane, control plane, and sidecar proxies

The Sidecar Proxy Model

Core Idea: A lightweight proxy is injected alongside every application container. All inbound and outbound traffic flows through this proxy, enabling transparent security and observability.
Control Plane (configuration, certificates, policy)
Data Plane (sidecar proxies intercepting all traffic)
Key Benefit: Applications do not need code changes. The mesh operates at the infrastructure layer, handling mTLS, retries, and authorization transparently.
The sidecar model means every pod gets a proxy container injected. This proxy intercepts all TCP traffic using iptables rules. The application is completely unaware of the mesh.

Istio Architecture

Control Plane: istiod

  • Pilot — service discovery, traffic management
  • Citadel — certificate authority, identity
  • Galley — configuration validation

All consolidated into a single istiod binary since Istio 1.5+

Data Plane: Envoy

  • High-performance C++ proxy
  • Extremely configurable (xDS APIs)
  • Supports HTTP/1.1, HTTP/2, gRPC, TCP
Resource Overhead
Envoy sidecars typically consume 50-100 MB memory each. Istiod requires 1-2 GB for large clusters. Consider this in resource-constrained environments.
Ambient Mode (newer)
Istio's ambient mode removes sidecars entirely, using node-level ztunnel proxies + optional waypoint proxies for L7 features.
Istio is the most feature-rich mesh but has the highest resource overhead. The ambient mode is a significant architectural shift that eliminates sidecars. For our labs, we use Linkerd due to lighter resource requirements.

Linkerd Architecture

Control Plane

  • destination — service discovery, policy
  • identity — certificate authority (mTLS)
  • proxy-injector — mutating webhook for sidecar injection

Data Plane: linkerd-proxy

  • Ultra-lightweight Rust-based proxy
  • Purpose-built for Kubernetes
  • ~10-20 MB memory per sidecar
  • Sub-millisecond p99 latency overhead
Why Linkerd for Labs?
  • 5x less memory than Envoy sidecars
  • Simpler installation and operation
  • Automatic mTLS with zero configuration
  • Well-suited for kind clusters
  • CNCF graduated project
Linkerd's Rust-based proxy is significantly lighter than Envoy. This makes it ideal for development clusters and resource-constrained environments. Linkerd graduated from CNCF in 2024.

Istio vs Linkerd Comparison

Feature Istio Linkerd
Proxy Envoy (C++) linkerd-proxy (Rust)
Memory per sidecar 50-100 MB 10-20 MB
mTLS Configurable (permissive/strict) Automatic, always on
L7 Authorization Rich policy model Server/route-based policies
Traffic Management Very flexible (VirtualService) TrafficSplit, HTTPRoute
Multi-cluster Full support Supported (multi-cluster link)
Complexity Higher Lower
CNCF Status Graduated Graduated
Both are CNCF graduated projects and production-ready. The choice depends on your requirements: Istio for maximum flexibility, Linkerd for simplicity and performance.

🔒 Mutual TLS (mTLS)

Encrypting and authenticating every connection

What is Mutual TLS?

Standard TLS: Client verifies the server's identity.
Mutual TLS: Both client and server verify each other's identity. Every connection is encrypted AND authenticated in both directions.

mTLS Provides

  • Confidentiality — traffic is encrypted
  • Integrity — data cannot be tampered with
  • Authentication — both ends prove identity
  • Authorization foundation — identity enables policy
Zero-Trust Enabler
mTLS is the foundation of zero-trust networking. Every connection is verified, regardless of network location.
mTLS is the cornerstone security feature of any service mesh. It ensures that even if an attacker gains access to the network, they cannot eavesdrop on or impersonate legitimate services.

How mTLS Works in a Mesh

  1. Certificate Issuance — control plane CA issues short-lived certificates to each proxy
  2. Identity Binding — certificates are bound to the pod's ServiceAccount identity
  3. Connection Establishment — proxies perform TLS handshake with mutual verification
  4. Automatic Rotation — certificates are rotated frequently (every 24h by default)
SPIFFE Identity Model: Both Istio and Linkerd use SPIFFE (Secure Production Identity Framework for Everyone). Identity URIs follow the format: spiffe://trust-domain/ns/namespace/sa/service-account
SPIFFE provides a standard identity format across mesh implementations. The identity is derived from the Kubernetes ServiceAccount, creating a cryptographic binding between workload identity and network identity.

Automatic mTLS in Linkerd

Zero Configuration Required! Linkerd enables mTLS automatically for all meshed traffic. There are no modes to configure — it just works.

Verifying mTLS is Active

# Check if mTLS is active on a specific pod
linkerd viz edges deployment -n my-namespace

# View the identity of connected proxies
linkerd identity -n my-namespace

# Check connections with tap
linkerd viz tap deployment/my-app -n my-namespace \
  --to deployment/my-db | grep tls=true
Non-meshed traffic: Traffic from/to pods without the Linkerd proxy is not encrypted. This is visible in edges output as tls=not_provided_by_remote.
Linkerd's approach is opinionated: mTLS is always on for meshed workloads. There is no permissive mode to configure, which reduces misconfiguration risk. The tap command shows TLS status in real time for each connection.

PeerAuthentication in Istio

# Enforce strict mTLS for the entire mesh
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system    # Mesh-wide when in istio-system
spec:
  mtls:
    mode: STRICT             # Reject non-mTLS connections

---
# Per-namespace override
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: payment-service
spec:
  mtls:
    mode: STRICT
  portLevelMtls:
    8080:
      mode: PERMISSIVE       # Allow non-mTLS on health check port
Permissive vs Strict: Istio defaults to PERMISSIVE mode, which accepts both mTLS and plaintext. Always switch to STRICT after verifying all clients are meshed.
Istio's PeerAuthentication has three modes: DISABLE, PERMISSIVE, and STRICT. PERMISSIVE is useful during migration but should never remain in production. Port-level overrides are useful for health check endpoints that receive non-meshed traffic from kubelet.

Certificate Management

Linkerd Certificates

  • Trust anchor — root CA (bring your own or auto-generated)
  • Issuer certificate — intermediate CA used by identity controller
  • Workload certificates — issued per-proxy, 24h expiry
# Check certificate expiry
linkerd check --proxy

# Rotate issuer cert
# (using cert-manager for
# automatic rotation)

Production Best Practices

Never use auto-generated trust anchors in production! Always bring your own CA certificates and manage rotation with cert-manager or Vault. 
# Generate trust anchor
step certificate create \
  root.linkerd.cluster.local \
  ca.crt ca.key \
  --profile root-ca \
  --no-password \
  --not-after=87600h
Certificate management is critical for production mesh deployments. Auto-generated certificates are fine for development but create operational risk in production. Use cert-manager to automate rotation.

🛡️ Authorization Policies

Controlling workload-to-workload communication

Linkerd Authorization Policies

Linkerd uses a default-deny model with Server and ServerAuthorization resources (and the newer AuthorizationPolicy / HTTPRoute resources) to control which workloads can communicate. 
# Define a Server (what to protect)
apiVersion: policy.linkerd.io/v1beta1
kind: Server
metadata:
  name: api-server
  namespace: my-app
spec:
  podSelector:
    matchLabels:
      app: api
  port: 8080
  proxyProtocol: HTTP/2

A Server resource identifies the port and pods that should receive authorization enforcement.

The Server resource is the foundational policy primitive in Linkerd. It tells Linkerd which ports on which pods should have authorization enforced. Without a Server resource, traffic flows freely between meshed pods.

Linkerd ServerAuthorization

# Allow only the frontend to call the API
apiVersion: policy.linkerd.io/v1beta1
kind: ServerAuthorization
metadata:
  name: allow-frontend-to-api
  namespace: my-app
spec:
  server:
    name: api-server
  client:
    meshTLS:
      serviceAccounts:
        - name: frontend
          namespace: my-app
Identity-Based: Authorization is based on the client's mTLS identity (ServiceAccount), not IP addresses. This is resilient to pod rescheduling.
ServerAuthorization binds to a Server and specifies which clients can access it. Using ServiceAccount identity means policies survive pod restarts and IP changes. You can also use unauthenticatedTLS and authorizatedIdentities for more flexibility.

Linkerd AuthorizationPolicy (Newer API)

# Fine-grained HTTP route authorization
apiVersion: policy.linkerd.io/v1alpha1
kind: AuthorizationPolicy
metadata:
  name: allow-get-orders
  namespace: my-app
spec:
  targetRef:
    group: policy.linkerd.io
    kind: HTTPRoute
    name: orders-get-route
  requiredAuthenticationRefs:
    - name: frontend-identity
      kind: MeshTLSAuthentication
      group: policy.linkerd.io

---
apiVersion: policy.linkerd.io/v1alpha1
kind: HTTPRoute
metadata:
  name: orders-get-route
  namespace: my-app
spec:
  parentRefs:
    - name: api-server
      kind: Server
      group: policy.linkerd.io
  rules:
    - matches:
        - path:
            value: /orders
          method: GET
The newer AuthorizationPolicy API works with Gateway API HTTPRoute resources for fine-grained L7 authorization. You can control access by HTTP method and path, not just by workload identity.

MeshTLSAuthentication

# Define who the "frontend" identity is
apiVersion: policy.linkerd.io/v1alpha1
kind: MeshTLSAuthentication
metadata:
  name: frontend-identity
  namespace: my-app
spec:
  identities:
    - "frontend.my-app.serviceaccount.identity.linkerd.cluster.local"

---
# Network-level authentication (for non-meshed clients)
apiVersion: policy.linkerd.io/v1alpha1
kind: NetworkAuthentication
metadata:
  name: allow-monitoring
  namespace: my-app
spec:
  networks:
    - cidr: 10.0.0.0/8   # Allow from cluster network
Non-meshed clients: Use NetworkAuthentication for traffic that does not have mTLS identity (e.g., probes from kubelet, external load balancers).
MeshTLSAuthentication identifies clients by their SPIFFE identity string. NetworkAuthentication is a fallback for non-meshed traffic — use sparingly as it relies on IP ranges which are less secure than cryptographic identity.

Istio AuthorizationPolicy

# Allow only specific services to access the payment API
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: payment-api-policy
  namespace: payment
spec:
  selector:
    matchLabels:
      app: payment-api
  action: ALLOW
  rules:
    - from:
        - source:
            principals:
              - "cluster.local/ns/checkout/sa/checkout-service"
      to:
        - operation:
            methods: ["POST"]
            paths: ["/api/v1/charge"]
      when:
        - key: request.headers[x-request-id]
          notValues: [""]
Istio's model: Combines source identity, destination operations, and conditional rules in a single resource. Supports ALLOW, DENY, and CUSTOM actions.
Istio's AuthorizationPolicy is more feature-rich than Linkerd's, supporting header matching, JWT claims, IP ranges, and custom external authorization. The 'when' conditions enable very fine-grained access control.

Default Deny Pattern

Linkerd

# Creating a Server resource
# without any ServerAuthorization
# automatically denies all traffic
# to that port

apiVersion: policy.linkerd.io/v1beta1
kind: Server
metadata:
  name: api-server
  namespace: my-app
spec:
  podSelector:
    matchLabels:
      app: api
  port: 8080
  proxyProtocol: HTTP/2
# No ServerAuthorization =
# default deny!

Istio

# Explicit deny-all policy
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: deny-all
  namespace: my-app
spec:
  {}
# Empty spec = deny all traffic
# to the namespace

# Then add specific ALLOW rules
# for permitted communication
"Start with default-deny, then explicitly allow only the communication paths your application needs."
Default deny is a fundamental zero-trust principle. In Linkerd, merely defining a Server resource without authorization triggers deny. In Istio, an empty AuthorizationPolicy spec denies everything.

🧑‍💻 Linkerd Deep Dive

Installation, injection, and observability

Installing Linkerd on a kind Cluster

# 1. Install the Linkerd CLI
curl --proto '=https' --tlsv1.2 -sSfL https://run.linkerd.io/install | sh
export PATH=$HOME/.linkerd2/bin:$PATH

# 2. Validate the cluster is ready
linkerd check --pre

# 3. Install the control plane
linkerd install --crds | kubectl apply -f -
linkerd install | kubectl apply -f -

# 4. Verify installation
linkerd check

# 5. Install the viz extension (observability)
linkerd viz install | kubectl apply -f -
linkerd viz check
That's it! Linkerd control plane is running with automatic mTLS. No additional configuration needed for encryption.
The linkerd check command validates over 40 different conditions. It is an excellent troubleshooting tool when something goes wrong. The viz extension adds Prometheus, Grafana, and the dashboard.

Injecting Linkerd Proxies

Annotation-Based Injection

# Per-namespace injection
apiVersion: v1
kind: Namespace
metadata:
  name: my-app
  annotations:
    linkerd.io/inject: enabled

---
# Per-deployment injection
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  template:
    metadata:
      annotations:
        linkerd.io/inject: enabled

CLI-Based Injection

# Inject into existing resources
kubectl get deploy -n my-app \
  -o yaml | linkerd inject - | \
  kubectl apply -f -

# Verify injection
linkerd check --proxy -n my-app

# Check proxy status
kubectl get pods -n my-app \
  -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.containers[*].name}{"\n"}{end}'
Injected pods will have 2 containers: the app + linkerd-proxy
Namespace-level annotation is the recommended approach for production. The CLI injection method is useful for testing or when you cannot modify the namespace. Both result in the same proxy being added.

Traffic Splitting & Retries

Traffic Splitting (Canary)

# Split traffic between versions
apiVersion: split.smi-spec.io/v1alpha2
kind: TrafficSplit
metadata:
  name: api-canary
  namespace: my-app
spec:
  service: api
  backends:
    - service: api-stable
      weight: 900     # 90%
    - service: api-canary
      weight: 100     # 10%

Retry Configuration

# Configure retries via
# ServiceProfile
apiVersion: linkerd.io/v1alpha2
kind: ServiceProfile
metadata:
  name: api.my-app.svc.cluster.local
  namespace: my-app
spec:
  routes:
    - name: GET /api/v1/orders
      condition:
        method: GET
        pathRegex: /api/v1/orders
      isRetryable: true
      timeout: 5s
Security Note: Traffic splitting enables safe rollout of security policy changes. Apply new authorization rules to the canary first and verify before full rollout.
Traffic splitting uses the SMI TrafficSplit spec. This is useful for canary deployments and A/B testing. From a security perspective, it allows gradual rollout of policy changes.

📊 Observability for Security

Detecting threats through metrics and traffic analysis

Golden Metrics for Security

Metric Security Signal
Latency Unusual spikes may indicate data exfiltration or DoS
Traffic Volume Unexpected surges suggest scanning or brute-force attacks
Error Rate 403/401 spikes indicate unauthorized access attempts
Saturation Resource exhaustion may signal a DoS attack
Security Anomalies to Watch
  • Service A suddenly talking to Service C (never before)
  • 4xx error rate spike on internal APIs
  • Request volume 10x normal at unusual hours
  • Non-TLS connections appearing in a meshed namespace
The golden metrics — latency, traffic, errors, saturation — are well-known for SRE purposes, but they are equally valuable security signals. A service mesh gives you these metrics for free on every connection.

Distributed Tracing for Security

Distributed tracing follows a request across multiple services, providing a complete picture of the request path. This is invaluable for security forensics — tracing the path of a suspicious request through your system.

Integration Options

  • Jaeger — popular open-source tracing backend
  • Tempo (Grafana) — scalable trace storage
  • OpenTelemetry — vendor-neutral collection

Linkerd propagates trace headers (b3, w3c) automatically but requires application instrumentation for full traces.

Security Use Cases

  • Trace the full path of a malicious request
  • Identify which services a compromised workload contacted
  • Correlate authorization failures across services
  • Verify request routing matches policy expectations
Linkerd does not perform distributed tracing natively — it propagates headers. Applications must forward trace context headers for end-to-end traces. For security forensics, combining mesh metrics with traces gives a complete picture.

Detecting Anomalous Traffic Patterns

# Check for unexpected connections between services
linkerd viz edges deployment -n my-app -o json | \
  jq '.[] | select(.noTLS == true)'

# Find services with high error rates
linkerd viz stat deploy -n my-app -o json | \
  jq '.[] | select(.successRate < 0.99)'

# Watch for traffic to services that should not receive external requests
linkerd viz tap deploy/internal-api -n my-app \
  --to-resource=deploy/database \
  --method=DELETE
Red Flags:
  • DELETE or PUT requests to services that should only receive GET
  • Traffic from unexpected namespaces or ServiceAccounts
  • Connections without mTLS in a fully-meshed namespace
Combine tap, stat, and edges commands to build a picture of normal traffic. Any deviation from the baseline should trigger investigation. Consider automating these checks and feeding them into alerting systems.

🚀 Practical Adoption

From evaluation to production

Incremental Adoption Strategy

  1. Evaluate — install mesh in a dev/staging cluster, measure overhead
  2. Observe — mesh one namespace, monitor metrics without enforcing policy
  3. Encrypt — verify mTLS is active on all meshed connections
  4. Authorize — add Server resources and ServerAuthorizations incrementally
  5. Enforce — enable default-deny, ensure all communication is explicitly authorized
  6. Expand — roll out namespace by namespace to production
Key Principle: Never enable authorization enforcement on all namespaces at once. Roll out one namespace at a time, validate, then move to the next.
The incremental approach reduces risk significantly. Observation before enforcement lets you discover all legitimate communication patterns before creating policies that might break them.

Namespace-by-Namespace Rollout

# Phase 1: Inject proxy (observe only)
kubectl annotate namespace staging linkerd.io/inject=enabled
kubectl rollout restart deploy -n staging

# Phase 2: Verify mTLS
linkerd viz edges deployment -n staging
# Confirm all edges show tls=true

# Phase 3: Add Server resources (enables authorization)
kubectl apply -f policies/staging-servers.yaml

# Phase 4: Add ServerAuthorizations (allow known traffic)
kubectl apply -f policies/staging-authz.yaml

# Phase 5: Verify no unexpected denials
linkerd viz stat deploy -n staging
# Watch for success rate drops

# Phase 6: Move to next namespace
kubectl annotate namespace production linkerd.io/inject=enabled
Each phase should run for a sufficient period — typically a few days at minimum. This allows you to observe the full range of traffic patterns including batch jobs, scheduled tasks, and less-frequent communication paths.

Handling Non-Meshed Traffic

Common Sources

  • Kubelet — health probes (liveness, readiness)
  • Ingress controllers — may not be meshed
  • Jobs/CronJobs — short-lived pods
  • External services — outside the cluster
  • Monitoring tools — Prometheus scraping
# Allow kubelet health probes
apiVersion: policy.linkerd.io/v1alpha1
kind: AuthorizationPolicy
metadata:
  name: allow-health-probes
  namespace: my-app
spec:
  targetRef:
    group: policy.linkerd.io
    kind: Server
    name: health-server
  requiredAuthenticationRefs:
    - name: kubelet
      kind: NetworkAuthentication
      group: policy.linkerd.io
---
apiVersion: policy.linkerd.io/v1alpha1
kind: NetworkAuthentication
metadata:
  name: kubelet
  namespace: my-app
spec:
  networks:
    - cidr: 0.0.0.0/0
Kubelet health probes are the most common source of non-meshed traffic. The kubelet runs on the host network and does not have a mesh proxy. You must explicitly allow this traffic or your pods will be killed when they fail health checks.

Performance Overhead

Metric Linkerd Istio (sidecar)
Latency overhead (p99) <1ms 2-5ms
Memory per sidecar 10-20 MB 50-100 MB
CPU per sidecar (idle) ~10m ~50m
Control plane memory ~500 MB ~1-2 GB
Pod startup overhead ~100ms ~1-2s

These numbers are approximate and depend on traffic volume. For kind clusters with limited resources, Linkerd's lighter footprint is a significant advantage. In production, always benchmark with your actual workload before committing to a mesh.

Proxy Not Injected

kubectl get ns my-app \
  -o jsonpath='{.metadata.annotations}'

# Verify webhook
kubectl get \
  mutatingwebhookconfigurations

# Force re-injection
kubectl rollout restart \

Connection Refused (Policy)

# Check which Server
# resources exist
kubectl get servers -n my-app

# Check authorizations
kubectl get \

The Ultimate Debugging Tool: linkerd check validates 40+ conditions and provides actionable remediation steps.
The most common issues are: proxy not injected (missing annotation), connections refused after adding Server resources (forgot to add authorization), and certificate expiry (forgot to set up rotation). Always start troubleshooting with linkerd check.

Production Readiness Checklist

  • Trust anchor certificates are externally managed (not auto-generated)
  • Certificate rotation is automated via cert-manager or Vault
  • All namespaces have injection annotations
  • Server resources defined for all security-critical ports
  • ServerAuthorizations enforce least-privilege communication
  • Health probe ports have NetworkAuthentication for kubelet
  • Grafana alerts configured for success rate drops and non-TLS connections
  • Access logging enabled on sensitive services
  • Resource requests/limits set on proxy containers
  • Linkerd control plane is monitored and has HA configuration
This checklist covers the essential items for production mesh deployments. Each item should be verified and documented before going live. The HA configuration requires running multiple replicas of control plane components and using PodDisruptionBudgets.

📝 Module 11 Key Takeaways

  • Service meshes add mTLS, L7 authorization, and observability to Kubernetes networking
  • Linkerd provides automatic mTLS with zero configuration — encryption is always on
  • Authorization policies should follow a default-deny pattern using Server resources
  • SPIFFE identities bind cryptographic identity to ServiceAccounts
  • Golden metrics from the mesh are powerful security signals for anomaly detection
  • Adopt incrementally: observe first, enforce later, one namespace at a time
  • Production deployments require externally managed certificates and HA control planes
Reinforce these key points before moving to the next module. The most important takeaway is the incremental adoption strategy — rushing mesh adoption causes outages.
