Why Audit Logging Matters

Security Benefits

Section 1

Audit Logging Configuration

We begin with the mechanics of enabling and configuring audit logging on the API server.

API Server Audit Flags

The API server accepts several flags to control audit logging:

# Core audit flags
--audit-policy-file=/etc/kubernetes/audit-policy.yaml
--audit-log-path=/var/log/kubernetes/audit.log

# Log rotation settings
--audit-log-maxage=30        # Days to retain old logs
--audit-log-maxbackup=10     # Number of old log files to keep
--audit-log-maxsize=100      # Max size in MB before rotation

# Batch mode (performance tuning)
--audit-log-batch-buffer-size=10000
--audit-log-batch-max-wait=5s
Important: Both --audit-policy-file and --audit-log-path are required. Without a policy file, no events are logged.
The audit-policy-file defines WHAT to log, while audit-log-path defines WHERE to log it. Log rotation settings are critical for production — without them, audit logs can consume all available disk space.

Enabling Audit Logging in kind

kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
    kubeadmConfigPatches:
      - |
        kind: ClusterConfiguration
        apiServer:
          extraArgs:
            audit-policy-file: /etc/kubernetes/audit-policy.yaml
            audit-log-path: /var/log/kubernetes/audit.log
            audit-log-maxage: "30"
            audit-log-maxbackup: "3"
            audit-log-maxsize: "100"
          extraVolumes:
            - name: audit-policy
              hostPath: /etc/kubernetes/audit-policy.yaml
              mountPath: /etc/kubernetes/audit-policy.yaml
              readOnly: true
            - name: audit-logs
              hostPath: /var/log/kubernetes/
              mountPath: /var/log/kubernetes/
              readOnly: false
In kind, we use kubeadmConfigPatches to configure the API server. The audit policy file must be mounted into the control-plane container. The log directory also needs a volume mount so logs persist and are accessible from the host.

Deploying the Audit Policy

Step-by-step for kind clusters

# 1. Create the kind cluster with the configuration above
kind create cluster --config kind-audit-config.yaml --name audit-lab

# 2. Copy the audit policy into the control-plane container
docker cp audit-policy.yaml \
  audit-lab-control-plane:/etc/kubernetes/audit-policy.yaml

# 3. Restart the API server (kind automatically picks it up)
docker exec audit-lab-control-plane \
  kill -HUP $(docker exec audit-lab-control-plane \
    pgrep kube-apiserver)

# 4. Verify audit logs are being written
docker exec audit-lab-control-plane \
  ls -la /var/log/kubernetes/audit.log
For kind clusters, the audit policy must be placed inside the Docker container running the control plane. After updating the configuration, the API server needs to be restarted to pick up the new settings.

Section 2

Audit Policies

The audit policy is the most important part of audit logging — it determines what gets recorded and at what level of detail.

Audit Policy Structure

apiVersion: audit.k8s.io/v1
kind: Policy
# Stages where events can be omitted
omitStages:
  - "RequestReceived"
rules:
  # Rules are evaluated in order; first match wins
  - level: None
    resources:
      - group: ""
        resources: ["endpoints", "services", "services/status"]

  - level: Metadata
    resources:
      - group: ""
        resources: ["secrets", "configmaps"]

  - level: Request
    verbs: ["create", "update", "patch"]
    resources:
      - group: ""
        resources: ["pods"]

  - level: RequestResponse
    resources:
      - group: "rbac.authorization.k8s.io"
Audit policies follow a first-match-wins model. Rules are evaluated top-to-bottom, and the first rule that matches determines the audit level. The omitStages field filters out events at specific processing stages — RequestReceived is often omitted since it duplicates information from later stages.

Audit Levels

Level What is Logged Use Case
None Nothing at all High-volume, low-risk endpoints (health checks)
Metadata Request metadata (user, timestamp, resource, verb) General activity tracking
Request Metadata + request body Write operations on sensitive resources
RequestResponse Metadata + request body + response body Critical resources (RBAC, secrets access)
Performance trade-off: Higher levels capture more detail but generate significantly more log volume. RequestResponse on all resources can produce GBs of logs per day.
Choosing the right level is a balance between visibility and resource consumption. Most production deployments use a tiered approach — None for health checks, Metadata for most resources, Request for write operations, and RequestResponse only for the most sensitive resources like RBAC changes and secret access.

Rules Matching in Detail

Match Criteria

  • users — specific user names
  • userGroups — group membership
  • verbs — API verbs (get, list, create, etc.)
  • resources — API resources
  • namespaces — target namespaces
  • nonResourceURLs — non-resource paths

Example: Targeted Rules

rules:
  # Ignore system:nodes reads
  - level: None
    users: ["system:nodes"]
    verbs: ["get", "list", "watch"]

  # Log admin actions fully
  - level: RequestResponse
    userGroups:
      - "system:masters"

  # Log namespace changes
  - level: Request
    verbs: ["create", "delete"]
    resources:
      - group: ""
        resources: ["namespaces"]
Rules can be very granular. You can combine multiple match criteria in a single rule — all specified criteria must match for the rule to apply. If a field is omitted, it matches everything for that criterion.

Production-Ready Audit Policy

apiVersion: audit.k8s.io/v1
kind: Policy
omitStages: ["RequestReceived"]
rules:
  # 1. Skip noisy endpoints
  - level: None
    resources:
      - group: ""
        resources: ["endpoints", "services", "services/status"]
    users: ["system:kube-proxy"]

  # 2. Skip health checks and discovery
  - level: None
    nonResourceURLs: ["/healthz*", "/livez*", "/readyz*", "/version"]

  # 3. Log secret access at Metadata level (avoid logging secret values)
  - level: Metadata
    resources:
      - group: ""
        resources: ["secrets"]

  # 4. Log RBAC changes with full detail
  - level: RequestResponse
    resources:
      - group: "rbac.authorization.k8s.io"

  # 5. Log workload changes at Request level
  - level: Request
    verbs: ["create", "update", "patch", "delete"]
    resources:
      - group: ""
        resources: ["pods", "deployments", "services"]
      - group: "apps"
        resources: ["deployments", "daemonsets", "statefulsets"]

  # 6. Catch-all: metadata for everything else
  - level: Metadata
This is a well-balanced policy for production. Note the important detail on secrets — we use Metadata level, NOT Request or RequestResponse, because those would log the actual secret values in the audit log, creating a security risk. The catch-all rule at the bottom ensures nothing goes completely unlogged.

"Never log secrets at Request or RequestResponse level — your audit logs would become a treasure trove for attackers."

This is a common mistake. Logging secrets at the Request level writes the base64-encoded secret data directly into the audit log. If the audit log is compromised, all secrets that were accessed during that period are exposed.

Section 3

Audit Backends

Kubernetes supports multiple backends for storing and forwarding audit events. Understanding the options helps you build a robust audit pipeline.

Audit Backend Options

Log Backend

  • Writes JSON to local files
  • Straightforward to configure
  • Built-in log rotation
  • Low latency impact
  • Requires log shipping for centralization
--audit-log-path=/var/log/audit.log
--audit-log-maxage=30
--audit-log-maxbackup=10
--audit-log-maxsize=100

Webhook Backend

  • Sends events via HTTP POST
  • Direct integration with SIEM
  • No local storage needed
  • Potential latency impact
  • Requires reliable endpoint
--audit-webhook-config-file=webhook.yaml
--audit-webhook-batch-max-wait=5s
--audit-webhook-batch-max-size=100
Most production clusters use both — the log backend as a reliable local store and the webhook backend for real-time forwarding to a SIEM. The webhook backend uses batching to minimize performance impact.

Audit Log JSON Structure

{
  "kind": "Event",
  "apiVersion": "audit.k8s.io/v1",
  "level": "RequestResponse",
  "auditID": "a]b12c34-d567-e890-f012-345678901234",
  "stage": "ResponseComplete",
  "requestURI": "/api/v1/namespaces/default/pods",
  "verb": "create",
  "user": {
    "username": "jane@example.com",
    "groups": ["developers", "system:authenticated"]
  },
  "sourceIPs": ["192.168.1.100"],
  "userAgent": "kubectl/v1.28.0",
  "objectRef": {
    "resource": "pods",
    "namespace": "default",
    "name": "web-app-7d9f8b6c5-xk2lm",
    "apiVersion": "v1"
  },
  "responseStatus": {
    "metadata": {},
    "code": 201
  },
  "requestReceivedTimestamp": "2025-01-15T14:30:22.123456Z",
  "stageTimestamp": "2025-01-15T14:30:22.234567Z"
}
Each audit event is a self-contained JSON object. The key fields for security analysis are: user (who), verb (what action), objectRef (on what resource), responseStatus (was it allowed), and sourceIPs (from where). The auditID links events across stages.

Webhook Backend Configuration

# webhook-config.yaml
apiVersion: v1
kind: Config
clusters:
  - name: audit-webhook
    cluster:
      server: https://siem.example.com/k8s-audit
      certificate-authority: /etc/kubernetes/pki/webhook-ca.crt
contexts:
  - name: audit-webhook
    context:
      cluster: audit-webhook
      user: audit-webhook
current-context: audit-webhook
users:
  - name: audit-webhook
    user:
      client-certificate: /etc/kubernetes/pki/webhook-client.crt
      client-key: /etc/kubernetes/pki/webhook-client.key
Performance tip: Use batch mode with --audit-webhook-batch-max-wait=5s to reduce the number of HTTP requests. Ensure the webhook endpoint has high availability — a down endpoint can cause API server throttling.
The webhook configuration uses the standard kubeconfig format. mTLS is recommended for securing the webhook endpoint. In production, the webhook endpoint should be highly available — consider running it as a cluster service or using a managed SIEM endpoint.

Section 4

Analyzing Audit Logs

Collecting audit logs is only half the story. You need to know how to query and analyze them to extract actionable security insights.

Key Audit Log Fields

Field Description Security Relevance
user.username Identity of the requester Track who performed actions
verb API operation Identify write vs read operations
objectRef.resource Target resource type Focus on sensitive resources
responseStatus.code HTTP status code 403s indicate unauthorized attempts
sourceIPs Client IP addresses Detect access from unexpected locations
userAgent Client tool identifier Spot unusual tooling
When investigating incidents, start with these fields. The combination of user + verb + resource tells you WHO did WHAT to WHICH resource. The responseStatus tells you if it succeeded. sourceIPs and userAgent provide context about HOW and FROM WHERE.

Querying Audit Logs with jq

# Find all failed requests (403 Forbidden)
cat audit.log | jq 'select(.responseStatus.code == 403)' \
  | jq '{user: .user.username, verb: .verb,
         resource: .objectRef.resource, ns: .objectRef.namespace}'

# List all users who accessed secrets
cat audit.log | jq 'select(.objectRef.resource == "secrets")' \
  | jq -r '.user.username' | sort | uniq -c | sort -rn

# Find all pod deletions
cat audit.log | jq 'select(.verb == "delete" and
  .objectRef.resource == "pods")' \
  | jq '{user: .user.username, pod: .objectRef.name,
         ns: .objectRef.namespace, time: .requestReceivedTimestamp}'

# Track RBAC changes
cat audit.log | jq 'select(
  .objectRef.apiGroup == "rbac.authorization.k8s.io" and
  (.verb == "create" or .verb == "update" or .verb == "delete"))' \
  | jq '{user: .user.username, verb: .verb,
         resource: .objectRef.resource, name: .objectRef.name}'
jq is the essential tool for working with Kubernetes audit logs. Each line in the audit log file is a separate JSON object, so you can pipe it through jq for filtering and field extraction. These queries form the foundation of manual incident investigation.

Detecting Suspicious Activity Patterns

Red Flags to Watch For

  • Excessive 403 errors from a single user
  • Secret access from unexpected service accounts
  • RBAC modifications outside change windows
  • Pod exec/attach from unknown sources
  • Namespace creation by non-admin users
  • Unusual API access patterns at odd hours

Detection Queries

# Brute-force detection:
# many 403s from one user
cat audit.log | jq -r \
  'select(.responseStatus.code == 403)
   | .user.username' \
  | sort | uniq -c | sort -rn \
  | head -10

# Pod exec tracking
cat audit.log | jq \
  'select(.objectRef.subresource
    == "exec")' \
  | jq '{user: .user.username,
    pod: .objectRef.name,
    ns: .objectRef.namespace}'
These patterns are common indicators of compromise or attempted compromise. In production, these queries should be automated and tied to alerting systems. A sudden spike in 403 errors often indicates credential stuffing or privilege escalation attempts.

Advanced Audit Log Queries

# Timeline of events for a specific user (incident response)
cat audit.log | jq 'select(.user.username == "suspect-user")' \
  | jq -r '[.requestReceivedTimestamp, .verb,
    .objectRef.resource, .objectRef.name,
    .responseStatus.code] | @tsv' | sort

# Find all actions from a specific source IP
cat audit.log | jq 'select(.sourceIPs[] == "10.0.1.55")' \
  | jq '{time: .requestReceivedTimestamp, user: .user.username,
         verb: .verb, resource: .objectRef.resource}'

# Aggregate activity by user and resource (heatmap data)
cat audit.log | jq -r '[.user.username, .objectRef.resource, .verb]
  | @tsv' | sort | uniq -c | sort -rn | head -20

# Find requests with unusually large response bodies
cat audit.log | jq 'select(.responseObject != null)' \
  | jq '{user: .user.username, resource: .objectRef.resource,
         size: (.responseObject | tostring | length)}'  \
  | jq 'select(.size > 10000)'
These advanced queries are useful during incident response. The timeline query helps reconstruct an attacker's actions. The source IP query helps determine the origin of suspicious activity. The aggregation query helps build a profile of normal vs abnormal behavior.

Section 5

CIS Kubernetes Benchmarks

The CIS Kubernetes Benchmark is the industry-standard security configuration guide for Kubernetes clusters. It provides prescriptive guidance for hardening every component of a Kubernetes deployment.

CIS Kubernetes Benchmark

The Center for Internet Security (CIS) Kubernetes Benchmark provides a comprehensive set of security configuration recommendations organized by cluster component.

Control Plane

  • API server configuration
  • Controller manager
  • Scheduler settings
  • etcd security

Worker Nodes

  • Kubelet configuration
  • Kubelet file permissions
  • Kube-proxy settings
  • Container runtime

Policies

  • RBAC and service accounts
  • Pod security
  • Network policies
  • Secrets management
The CIS Benchmark is regularly updated to keep pace with new Kubernetes releases. It provides two profile levels: Level 1 (practical security settings) and Level 2 (defense-in-depth settings that may impact functionality). Most organizations target Level 1 compliance at minimum.

Benchmark Structure

Section Focus Area Example Controls
1.x Control Plane Components API server flags, auth modes, TLS
2.x etcd Encryption, peer auth, file permissions
3.x Control Plane Configuration Authentication, admission controllers
4.x Worker Nodes Kubelet auth, file permissions
5.x Policies RBAC, PSA, network policies, secrets
Scoring: Each control is either Scored (automated pass/fail) or Not Scored (manual verification required).
Understanding the benchmark structure helps you prioritize remediation. Section 1 (Control Plane) is the most critical because a misconfigured API server can undermine all other security controls. Section 5 (Policies) often has the most findings because it covers operational practices.

kube-bench: Automated CIS Scanning

kube-bench is an open-source tool by Aqua Security that automates CIS Kubernetes Benchmark checks against your cluster. 
# Run kube-bench as a Job in the cluster
kubectl apply -f https://raw.githubusercontent.com/aquasecurity/\
kube-bench/main/job.yaml

# View results
kubectl logs job/kube-bench

# Run specific sections
kubectl run kube-bench --image=aquasec/kube-bench:latest \
  --restart=Never -- run --targets=master

# Run on worker nodes
kubectl run kube-bench-node --image=aquasec/kube-bench:latest \
  --restart=Never -- run --targets=node
kube-bench runs the CIS checks as a Kubernetes Job, so it has access to the node's filesystem and process information. It automatically detects the Kubernetes version and selects the appropriate benchmark version. Results are output to stdout and can be collected from pod logs.

kube-bench in kind Clusters

# kube-bench-job.yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: kube-bench
spec:
  template:
    spec:
      hostPID: true
      containers:
        - name: kube-bench
          image: aquasec/kube-bench:latest
          command: ["kube-bench", "run", "--targets=master,node"]
          volumeMounts:
            - name: var-lib-kubelet
              mountPath: /var/lib/kubelet
              readOnly: true
            - name: etc-kubernetes
              mountPath: /etc/kubernetes
              readOnly: true
      restartPolicy: Never
      volumes:
        - name: var-lib-kubelet
          hostPath:
            path: /var/lib/kubelet
        - name: etc-kubernetes
          hostPath:
            path: /etc/kubernetes
Note: kube-bench requires hostPID: true and host filesystem mounts to inspect the running API server process and configuration files.
Running kube-bench in kind requires special permissions because it needs to inspect the host process table and configuration files. This is why hostPID and hostPath mounts are required. In production, kube-bench is typically run as a DaemonSet to check every node.

Interpreting kube-bench Results

CIS Benchmark Remediation

Common Findings

# Fix file permissions
chmod 644 /etc/kubernetes/manifests/*.yaml
chmod 600 /etc/kubernetes/pki/*.key
chown root:root /etc/kubernetes/admin.conf

# Enable audit logging (1.2.22-1.2.25)
# Add to API server manifest:
--audit-policy-file=/etc/kubernetes/audit.yaml
--audit-log-path=/var/log/audit.log
--audit-log-maxage=30
--audit-log-maxsize=100

# Disable profiling (1.2.21)
--profiling=false

API Server Hardening

# Disable anonymous auth (1.2.1)
--anonymous-auth=false

# Enable RBAC (1.2.8)
--authorization-mode=Node,RBAC

# Enable admission controllers
--enable-admission-plugins=\
  NodeRestriction,\
  PodSecurity,\
  ServiceAccount

# Set secure TLS ciphers (1.2.31)
--tls-cipher-suites=\
  TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256,\
  TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
Remediation should be applied carefully in production. Always test changes in a staging environment first. Some changes (like disabling anonymous auth) can break monitoring systems that rely on unauthenticated health check endpoints. Follow your change management process for control plane modifications.

Section 6

Compliance Frameworks

With CIS benchmarks understood, we now map Kubernetes security controls to industry compliance frameworks. Organizations running Kubernetes must often demonstrate compliance with one or more of these standards.

Compliance Frameworks & Kubernetes

SOC 2

  • Trust Service Criteria: Security, Availability, Confidentiality
  • K8s requirements: Access control (RBAC), audit logging, encryption, change management
  • Evidence: Audit logs, RBAC configurations, network policies

PCI-DSS

  • Payment card industry data security
  • K8s requirements: Network segmentation, access control, log monitoring, vulnerability scanning
  • Evidence: Network policies, scan reports, audit trails

HIPAA

  • Healthcare data protection
  • K8s requirements: PHI encryption at rest/transit, access controls, audit trails, integrity controls
  • Evidence: Encryption configs, RBAC, audit logs

ISO 27001

  • Information security management
  • K8s requirements: Risk assessment, access control, cryptography, operations security
  • Evidence: Security policies, configs, monitoring data
Each framework has different requirements but they overlap significantly in the Kubernetes context. RBAC, audit logging, encryption, and network segmentation are common across all four. Understanding these mappings lets you implement controls once and satisfy multiple frameworks.

CIS Controls to Compliance Mapping

CIS Control SOC 2 PCI-DSS HIPAA ISO 27001
RBAC enabled (1.2.8) CC6.1 7.1 164.312(a) A.9.2
Audit logging (1.2.22) CC7.2 10.1 164.312(b) A.12.4
Encryption at rest (1.2.29) CC6.7 3.4 164.312(a)(2)(iv) A.10.1
Network policies (5.3) CC6.6 1.2 164.312(e) A.13.1
Pod security (5.2) CC6.1 6.2 164.312(a) A.14.2
This mapping table is valuable for compliance teams. It shows how a single Kubernetes security control can satisfy requirements across multiple frameworks. This makes it more efficient to implement — you configure the control once and can provide evidence for multiple audits.

Evidence Collection for Auditors

# RBAC evidence: export current roles and bindings
kubectl get clusterroles,clusterrolebindings -o yaml > rbac-evidence.yaml
kubectl get roles,rolebindings --all-namespaces -o yaml >> rbac-evidence.yaml

# Network policy evidence
kubectl get networkpolicies --all-namespaces -o yaml > netpol-evidence.yaml

# Pod security evidence
kubectl get pods --all-namespaces -o json | jq '.items[] | {
  name: .metadata.name, namespace: .metadata.namespace,
  securityContext: .spec.securityContext,
  containerSecurity: [.spec.containers[].securityContext]
}' > pod-security-evidence.json

# Encryption configuration evidence
kubectl get secrets --all-namespaces --field-selector type=Opaque \
  -o custom-columns=NS:.metadata.namespace,NAME:.metadata.name

# Audit log sample (last 24 hours)
cat audit.log | jq 'select(.requestReceivedTimestamp > "2025-01-14T00:00:00Z")' \
  > audit-sample-evidence.json
Auditors typically want to see both the configuration (what should happen) and the evidence (what did happen). Collecting this evidence regularly and storing it in a secure location makes audit preparation much smoother. Consider automating evidence collection as a CronJob.

Section 7

Continuous Compliance

Point-in-time audits are necessary but not sufficient. Continuous compliance ensures that your cluster stays compliant between audits and catches drift immediately.

Polaris: Workload Compliance

Polaris by Fairwinds validates Kubernetes workload configurations against security and operational best practices.

What Polaris Checks

  • Resource requests and limits
  • Security context settings
  • Health check probes
  • Image pull policies
  • Privileged containers
  • Host namespace usage

Running Polaris

# Install Polaris CLI
brew install FairwindsOps/tap/polaris

# Run audit against cluster
polaris audit \
  --format=pretty

# Run against a single file
polaris audit \
  --audit-path deployment.yaml

# Run as dashboard
polaris dashboard \
  --port 8080
Polaris provides three modes: CLI audit for CI/CD integration, a dashboard for visual monitoring, and a webhook for admission control. The CLI mode is ideal for automated compliance scanning, while the dashboard gives teams visibility into their compliance posture.

Kubescape: Comprehensive Security Scanning

Kubescape is a CNCF project that scans clusters against multiple frameworks including CIS, NSA-CISA, MITRE ATT&CK, and custom frameworks.

# Install Kubescape
curl -s https://raw.githubusercontent.com/kubescape/kubescape/master/\
install.sh | /bin/bash

# Scan against all frameworks
kubescape scan --verbose

# Scan against specific framework
kubescape scan framework cis-v1.23-t1.0.1

# Scan against NSA-CISA hardening guide
kubescape scan framework nsa

# Scan a specific namespace
kubescape scan --include-namespaces production

# Export results as JSON
kubescape scan --format json --output results.json
Kubescape covers more ground than kube-bench because it examines workload configurations in addition to cluster infrastructure. The MITRE ATT&CK framework mapping is particularly valuable for understanding how misconfigurations could be exploited by attackers.

Policy-as-Code for Compliance

Approach

  • Define compliance requirements as code
  • Version control policies alongside infrastructure
  • Enforce at admission time (preventive)
  • Scan continuously (detective)
  • Automate remediation where possible
Shift left: Catch violations before they reach the cluster, not after.

Tools Stack

# CI/CD Pipeline
stages:
  - lint:
      - polaris audit --audit-path .
      - kubescape scan *.yaml

  - admission:
      # OPA Gatekeeper / Kyverno
      - enforce pod security
      - enforce network policies
      - enforce resource limits

  - runtime:
      # Continuous scanning

Policy-as-code ensures that compliance requirements are treated with the same rigor as application code — they are reviewed, tested, versioned, and deployed through CI/CD. This eliminates configuration drift and makes compliance reproducible.

Automating Evidence Collection

# compliance-evidence-cronjob.yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: compliance-evidence
  namespace: compliance
spec:
  schedule: "0 2 * * *"  # Daily at 2 AM
  jobTemplate:
    spec:
      template:
        spec:
          serviceAccountName: compliance-collector
          containers:
            - name: collector
              image: bitnami/kubectl:latest
              command: ["/bin/sh", "-c"]
              args:
                - |
                  # Collect RBAC state
                  kubectl get clusterroles,clusterrolebindings -o yaml \
                    > /evidence/rbac-$(date +%Y%m%d).yaml
                  # Collect network policies
                  kubectl get networkpolicies -A -o yaml \
                    > /evidence/netpol-$(date +%Y%m%d).yaml
                  # Collect pod security contexts
                  kubectl get pods -A -o json | jq '.items[] | {
                    name: .metadata.name, ns: .metadata.namespace,
                    sc: .spec.securityContext}' \
                    > /evidence/podsec-$(date +%Y%m%d).json
              volumeMounts:
                - name: evidence
                  mountPath: /evidence
          volumes:
            - name: evidence
              persistentVolumeClaim:
                claimName: compliance-evidence-pvc
          restartPolicy: OnFailure
Automated evidence collection runs daily and stores snapshots of your security configuration. This creates a timeline that auditors can review and is invaluable for detecting when and how configurations changed. Store evidence in a tamper-evident system for maximum credibility.

Configuration Drift Detection

What is Drift?

Drift occurs when the actual cluster state diverges from the desired secure baseline — often due to manual changes, emergency patches, or misconfigured automation.

Detection Methods

  • GitOps comparison (desired vs actual)
  • Scheduled kube-bench scans
  • Admission controller alerts
  • Audit log monitoring for config changes

Drift Detection Query

# Detect RBAC changes via audit logs
cat audit.log | jq 'select(
  .objectRef.apiGroup ==
    "rbac.authorization.k8s.io" and
  .verb != "get" and
  .verb != "list" and
  .verb != "watch")' | jq '{
    time: .requestReceivedTimestamp,
    user: .user.username,
    action: .verb,
    resource: .objectRef.resource,
    name: .objectRef.name
}'

# Compare kube-bench runs
diff <(cat baseline-results.json | \
  jq '.Controls[].Tests[].Results[] |
  select(.status == "FAIL") | .test_number') \
  <(cat current-results.json | \
  jq '.Controls[].Tests[].Results[] |
  select(.status == "FAIL") | .test_number')
Drift detection is the bridge between point-in-time audits and continuous compliance. By comparing current state against a known-good baseline, you can detect and remediate changes before they become audit findings. GitOps-based deployments naturally resist drift because all changes must go through version control.

"Compliance is not a destination — it is a continuous process. Automate everything you can, and monitor everything you cannot automate."

This mindset shift is critical. Compliance is often treated as a periodic event, but Kubernetes environments change constantly. Continuous compliance tooling ensures that you maintain your security posture every day, not just on audit day.
