⚠️ Why IR is Different in Kubernetes

Traditional IR

  • Long-lived servers
  • Persistent filesystems
  • Well-known host forensic tools
  • Static network topology
  • Evidence persists on disk

Kubernetes IR

  • Ephemeral pods and containers
  • Layered, read-only filesystems
  • Minimal container images lack tools
  • Dynamic service mesh networking
  • Evidence vanishes on pod restart
Critical challenge: A crashed or restarted pod destroys all in-memory and filesystem evidence instantly.
The ephemeral nature of containers is the single biggest challenge for Kubernetes IR. Traditional forensics assumes you can image a disk — in Kubernetes, there may be no disk to image. Speed of evidence collection is paramount.

🛡️ Section 1

IR Framework for Kubernetes

Let's start with the foundational framework. We will adapt the NIST incident response lifecycle specifically for Kubernetes environments.

🔄 NIST IR Lifecycle

The Five Phases

  1. Preparation — tools, runbooks, team training
  2. Detection & Analysis — identify and validate the incident
  3. Containment, Eradication & Recovery — stop the bleeding, remove the threat, restore
  4. Post-Incident Activity — lessons learned, process improvements
This lifecycle is iterative — you may cycle between phases as new evidence emerges.
NIST SP 800-61 provides the standard framework. While the phases are the same for Kubernetes, the specific actions within each phase differ significantly due to the container orchestration layer.

📋 Preparation Phase

Tools & Infrastructure

  • Runtime detection (Falco, Tetragon)
  • Centralized logging (EFK/PLG stack)
  • Audit log pipeline configured
  • Forensic namespace pre-created
  • Ephemeral debug container images ready
  • Network capture tools available

People & Process

  • IR team roles defined
  • Runbooks for common scenarios
  • Escalation paths documented
  • Communication channels established
  • Regular tabletop exercises
  • Access to cluster pre-provisioned
Preparation is everything. When an incident hits, you do not want to be figuring out tooling or permissions. Pre-staging a forensic namespace with the right tools and RBAC is a game-changer for response time.

👥 IR Team Roles for Kubernetes

Role Responsibility K8s-Specific Skills
Incident Commander Coordinates response, makes decisions Cluster architecture understanding
K8s Platform Engineer Executes containment actions kubectl, RBAC, NetworkPolicies
Forensic Analyst Collects and analyzes evidence Container runtimes, log analysis
Security Analyst Threat analysis, IOC identification Audit logs, Falco rules, MITRE ATT&CK
Communications Lead Stakeholder updates Translating K8s concepts for leadership
In smaller organizations, individuals may wear multiple hats. The key is that these responsibilities are assigned before an incident occurs. During an incident is not the time to figure out who does what.

📖 Kubernetes IR Runbook Template

Runbook Structure

# incident-runbook-compromised-pod.yaml
scenario: "Suspected Compromised Pod"
severity: P1
detection_sources:
  - Falco alert: "Terminal shell in container"
  - Audit log: unexpected exec into pod

immediate_actions:
  1. Identify affected pod, namespace, node
  2. Capture pod state (kubectl describe, logs)
  3. Apply deny-all NetworkPolicy
  4. Snapshot container filesystem
  5. Notify Incident Commander

escalation:
  - If lateral movement detected → P0
  - If secrets accessed → rotate immediately
  - If node compromise suspected → cordon node
Runbooks should be version-controlled, reviewed regularly, and tested during tabletop exercises. Each common scenario should have its own runbook with clear, step-by-step actions.

🔍 Section 2

Detection & Triage

Detection is the first real test of your security posture. The faster you detect, the smaller the blast radius. Let's look at what signals to monitor and how to triage effectively.

🔔 Alert Sources in Kubernetes

Runtime Detection

  • Falco alerts
  • Tetragon events
  • Seccomp violations
  • AppArmor denials
  • OPA/Gatekeeper violations

Platform Signals

  • K8s audit logs
  • API server anomalies
  • etcd access patterns
  • Node-level syslog
  • Cloud provider alerts

Application Signals

  • Pod crash loops
  • Unexpected outbound traffic
  • Resource usage spikes
  • Service mesh telemetry
  • Application error rates
No single source catches everything. A mature detection capability correlates signals across all three categories. For example, a Falco alert about a shell in a container, combined with unusual outbound traffic and an audit log showing exec, paints a clear picture of compromise.

✅ Initial Triage Checklist

First 5 Minutes

  1. ☐ What triggered the alert? — Identify source and rule name
  2. ☐ Which pod/container? — kubectl get pod -o wide
  3. ☐ Which namespace/node? — Determine scope
  4. ☐ What ServiceAccount? — Check RBAC permissions
  5. ☐ What Secrets are mounted? — Assess credential exposure
  6. ☐ Network access? — Check existing NetworkPolicies
  7. ☐ Is the pod still running? — Preserve before it is gone
Time is critical: Every second the pod runs, the attacker may be exfiltrating data or moving laterally.
This checklist should be memorized by anyone in the IR rotation. The answers to these questions determine your next actions and severity classification. If the pod has already been restarted, your forensic options are significantly limited.

🚨 Severity Classification

Severity Criteria Response Time Example
P0 — Critical Active exploitation, data exfil < 15 minutes Reverse shell from prod pod
P1 — High Confirmed compromise, contained < 1 hour Crypto miner in staging
P2 — Medium Suspicious activity, unconfirmed < 4 hours Unusual API calls from SA
P3 — Low Policy violation, no exploitation < 24 hours Pod running as root in dev
Severity classification drives response urgency and resource allocation. Over-classification leads to alert fatigue; under-classification leads to missed incidents. Calibrate through regular review and post-incident analysis.

💥 Determining Blast Radius

# 1. What can this ServiceAccount access?
kubectl auth can-i --list --as=system:serviceaccount:app:compromised-sa

# 2. What secrets are available in this namespace?
kubectl get secrets -n app-namespace

# 3. What other pods share this node?
kubectl get pods --field-selector spec.nodeName=node-01 --all-namespaces

# 4. What NetworkPolicies are in effect?
kubectl get networkpolicies -n app-namespace -o yaml

# 5. Are there any cluster-wide resources accessible?
kubectl auth can-i get clusterroles \
  --as=system:serviceaccount:app:compromised-sa
Blast radius = RBAC permissions + network access + mounted secrets + node co-tenants
Think of blast radius as the total damage potential. A pod with cluster-admin privileges on a node with other sensitive workloads has a massive blast radius. A pod with no RBAC, deny-all network policy, and no secrets has a minimal one.

🕓 Timeline Construction

Correlating Events Across Sources

# Pull audit logs for the affected ServiceAccount
kubectl logs -n kube-system kube-apiserver-master \
  | jq 'select(.user.username == "system:serviceaccount:app:web-sa")'

# Check Falco alerts in the time window
kubectl logs -n falco -l app=falco --since=2h \
  | grep "app-namespace"

# Correlate with pod events
kubectl get events -n app-namespace --sort-by='.lastTimestamp'

# Check node-level logs
journalctl -u kubelet --since "2 hours ago" | grep "suspect-pod"

Tip: Synchronize timestamps across all sources using UTC to build an accurate timeline.

Timeline construction is one of the most important forensic activities. Having centralized, timestamped logs makes this dramatically easier. If your logs are scattered across different systems with different time zones, correlation becomes extremely difficult.

🛑 Section 3

Containment Strategies

Once you have triaged and assessed, the next priority is containment — stopping the bleeding while preserving evidence. This is the most time-sensitive phase.

🌐 Network Isolation with NetworkPolicies

# deny-all-quarantine.yaml
# Apply immediately to isolate compromised pod
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: quarantine-deny-all
  namespace: app-namespace
spec:
  podSelector:
    matchLabels:
      quarantine: "true"     # Applied to suspect pods
  policyTypes:
    - Ingress
    - Egress
  # No ingress or egress rules = deny all traffic
Important: Apply the NetworkPolicy before relabeling the pod. The policy must exist first to take effect when labels match.
Network isolation is usually the first containment action. It cuts off communication while keeping the pod running for evidence collection. The order matters: create the policy, then label the pod. If you reverse the order, there is a window where the pod is labeled but not yet isolated.

🏷️ Pod Quarantine Procedure

# Step 1: Ensure deny-all NetworkPolicy exists (from previous slide)
kubectl apply -f deny-all-quarantine.yaml

# Step 2: Label the pod for quarantine (triggers NetworkPolicy)
kubectl label pod suspect-pod -n app-namespace quarantine=true

# Step 3: Remove the pod from any Service selectors
kubectl label pod suspect-pod -n app-namespace app- role-

# Step 4: Cordon the node to prevent new workloads
kubectl cordon node-01

# Step 5: Prevent the controller from replacing the pod
kubectl scale deployment compromised-app -n app-namespace --replicas=0

# NOTE: Do NOT delete the pod yet — collect evidence first!
Quarantine keeps the pod alive but isolated. This is essential for evidence collection.
Notice we scale the deployment to zero replicas rather than deleting the pod. If we just delete the pod, the ReplicaSet controller will create a new one. Scaling to zero prevents replacement while keeping the current pod running thanks to the quarantine label that no longer matches the deployment selector.

🔒 RBAC Lockdown

Immediate Actions

# Revoke ServiceAccount permissions
kubectl delete rolebinding \
  web-sa-binding -n app-namespace

# Invalidate tokens
kubectl delete secret \
  web-sa-token -n app-namespace

# Delete and recreate the SA
kubectl delete serviceaccount \
  web-sa -n app-namespace

Credential Rotation

  • Rotate all Secrets in the namespace
  • Rotate cloud provider credentials
  • Invalidate any JWT/OAuth tokens
  • Rotate TLS certificates if exposed
  • Update external service passwords
  • Rotate etcd encryption keys
Assume all mounted secrets are compromised. Rotate every credential the pod had access to.
Credential rotation is often the most impactful containment action but also the most disruptive. Coordinate with application teams because rotating secrets will cause outages for services using those credentials. Better a planned brief outage than ongoing unauthorized access.

💾 Preserving Evidence During Containment

Evidence Preservation Checklist

  1. ☐ Capture full kubectl describe pod output
  2. ☐ Export pod logs (kubectl logs --all-containers)
  3. ☐ Save pod YAML spec (kubectl get pod -o yaml)
  4. ☐ Snapshot container filesystem (before deletion)
  5. ☐ Record network connections at time of detection
  6. ☐ Screenshot any running processes in the container
  7. ☐ Save the container image hash for later analysis
  8. ☐ Preserve node-level logs for the affected node

All evidence should be timestamped, hashed (SHA-256), and stored in a secure, tamper-evident location.

Chain of custody matters, even for internal investigations. Hash every piece of evidence you collect. Store it in an immutable storage location such as an S3 bucket with object locking enabled.

🔎 Section 4

Evidence Collection

Now that we have contained the threat, let's look at the specific techniques and commands for collecting forensic evidence from Kubernetes environments.

📸 Capturing Pod State

#!/bin/bash
# forensic-capture.sh — Run immediately upon detection
POD="suspect-pod"
NS="app-namespace"
EVIDENCE_DIR="./evidence/$(date +%Y%m%d-%H%M%S)"
mkdir -p "$EVIDENCE_DIR"

# Full pod description
kubectl describe pod "$POD" -n "$NS" > "$EVIDENCE_DIR/describe.txt"

# Pod YAML specification
kubectl get pod "$POD" -n "$NS" -o yaml > "$EVIDENCE_DIR/pod-spec.yaml"

# All container logs (current and previous)
kubectl logs "$POD" -n "$NS" --all-containers > "$EVIDENCE_DIR/logs-current.txt"
kubectl logs "$POD" -n "$NS" --all-containers --previous > "$EVIDENCE_DIR/logs-previous.txt" 2>/dev/null

# Events for the namespace
kubectl get events -n "$NS" --sort-by='.lastTimestamp' > "$EVIDENCE_DIR/events.txt"

# Hash all evidence files
sha256sum "$EVIDENCE_DIR"/* > "$EVIDENCE_DIR/checksums.sha256"
echo "Evidence captured in $EVIDENCE_DIR"
This script should be pre-written, tested, and available in your IR toolkit. When an incident occurs, you should not be writing scripts from scratch. Run it immediately, even before containment if possible, because pods can terminate at any time.

📂 Container Filesystem Snapshots

# Method 1: Copy files from a running container
kubectl cp "$NS/$POD:/etc/passwd" "$EVIDENCE_DIR/passwd"
kubectl cp "$NS/$POD:/tmp/" "$EVIDENCE_DIR/tmp-contents/"

# Method 2: Export the entire container filesystem
# Get the container ID from the node
CONTAINER_ID=$(kubectl get pod "$POD" -n "$NS" \
  -o jsonpath='{.status.containerStatuses[0].containerID}' \
  | sed 's|containerd://||')

# On the node — export the full filesystem
ctr -n k8s.io snapshot view forensic-snap "$CONTAINER_ID"
ctr -n k8s.io content export "$EVIDENCE_DIR/container-fs.tar" "$CONTAINER_ID"

# Method 3: Use crictl on the node
crictl inspect "$CONTAINER_ID" > "$EVIDENCE_DIR/container-inspect.json"
Note: Methods 2 and 3 require SSH or privileged access to the underlying node.
kubectl cp is the simplest method but requires a shell in the container. For containers with minimal images and no shell, you need to go to the node level and use container runtime tools like crictl or ctr directly. This is why having node access pre-configured in your IR plan is essential.

📜 Comprehensive Log Collection

Kubernetes Logs

# Pod logs with timestamps
kubectl logs "$POD" -n "$NS" \
  --timestamps --all-containers

# Previous instance logs
kubectl logs "$POD" -n "$NS" \
  --previous

# Logs from all pods with a label
kubectl logs -n "$NS" \
  -l app=web --all-containers

# Audit logs (API server)
kubectl logs -n kube-system \
  kube-apiserver-master \
  | jq '.| select(.verb != "get")'

Node & System Logs

# kubelet logs
journalctl -u kubelet \
  --since "4h ago"

# Container runtime logs
journalctl -u containerd \
  --since "4h ago"

# Kernel/security logs
journalctl -k --since "4h ago"

# Falco alerts
kubectl logs -n falco \
  -l app=falco --since=4h \
  | jq '.'
Ideally, all these logs are already flowing into a centralized system. If they are not, you need to collect them manually and quickly. Remember that log rotation and pod restarts can destroy logs, so capture early and capture broadly.

🐛 Ephemeral Debug Containers

# Attach a debug container to a running pod
# Shares the process namespace — can see all processes
kubectl debug -it "$POD" -n "$NS" \
  --image=busybox:latest \
  --target=app-container \
  -- sh

# Inside the debug container:
# List processes (sees target container's processes)
ps aux

# Check network connections
netstat -tlnp

# Examine filesystem of the target container
ls /proc/1/root/

# Check environment variables (may contain secrets)
cat /proc/1/environ | tr '\0' '\n'
Advantage: Debug containers work even with distroless images that have no shell or forensic tools.
Ephemeral containers, introduced as stable in Kubernetes 1.25, are a critical forensic tool. They let you attach diagnostic tooling to a running pod without modifying the pod spec. Use a purpose-built forensic image that includes tools like strace, tcpdump, and lsof.

🔧 Essential Forensic kubectl Commands

# Check who/what accessed the API recently
kubectl get events --all-namespaces --sort-by='.metadata.creationTimestamp' \
  | tail -50

# Inspect ServiceAccount token mount
kubectl get pod "$POD" -n "$NS" -o jsonpath='{.spec.volumes[*].name}'

# Check environment variables (may reveal injected secrets)
kubectl set env pod/"$POD" -n "$NS" --list

# View resource consumption (detect crypto miners)
kubectl top pod -n "$NS" --containers

# Check image pull history
kubectl get pod "$POD" -n "$NS" \
  -o jsonpath='{.status.containerStatuses[*].imageID}'

# Inspect node conditions
kubectl describe node "$(kubectl get pod $POD -n $NS \
  -o jsonpath='{.spec.nodeName}')"
Keep a cheat sheet of these commands in your runbook. During a high-pressure incident, you do not want to be looking up jsonpath syntax. Consider creating aliases or scripts for the most commonly used forensic commands.

📊 Section 5

Post-Incident Analysis

After containment and evidence collection, we shift to analysis. This is where we understand what happened, how it happened, and how to prevent it from happening again.

🎯 Root Cause Analysis

The Five Whys — Kubernetes Example

  1. Why did the attacker gain access? — Container had a known CVE.
  2. Why was a vulnerable image deployed? — No image scanning in CI/CD.
  3. Why was there no image scanning? — Admission controller was in audit mode.
  4. Why was it still in audit mode? — Migration plan was never completed.
  5. Why was the migration not completed? — No owner assigned after initial setup.
Root cause is usually a process or organizational gap, not a single technical failure.
The Five Whys technique helps peel back the layers. Notice how we moved from a technical vulnerability to an organizational accountability gap. Addressing only the technical issue means you will have a similar incident in the future.

📅 Attack Timeline Reconstruction

Time (UTC) Source Event
14:22:01 Audit Log Pod web-app-7d4f created in prod
14:35:17 Falco Shell spawned in container web-app
14:35:42 Falco Sensitive file read: /etc/shadow
14:36:05 Audit Log SA token used to list secrets in namespace
14:37:12 Network Outbound connection to 45.33.x.x:443
14:38:00 Alert SOC analyst acknowledges Falco alert
14:42:30 K8s Quarantine NetworkPolicy applied

Mean Time to Detect (MTTD): 3 min | Mean Time to Contain (MTTC): 7 min

An accurate timeline is the foundation of your post-incident report. It shows the dwell time, the speed of response, and any gaps in detection. MTTD and MTTC are key metrics to track and improve over time.

🚩 Indicators of Compromise (IOCs)

Kubernetes-Specific IOCs

  • Unexpected exec into pods
  • New ServiceAccounts or RoleBindings
  • Pods with privileged: true
  • DaemonSets in non-system namespaces
  • CronJobs with base64-encoded commands
  • Pods mounting hostPath volumes
  • Unknown images or tags

Network & System IOCs

  • Outbound connections to C2 IPs
  • DNS queries to crypto mining pools
  • Unusual port scanning from pods
  • High CPU from non-critical pods
  • Binary downloads via curl/wget
  • Modified system binaries in containers
  • Unexpected processes (miners, shells)
Document every IOC you discover and share them with the team. These IOCs should be fed back into your detection rules — for example, creating Falco rules for the specific techniques used by the attacker.

📝 Blameless Postmortems

Postmortem Document Template

  • Incident Summary: One-paragraph description
  • Timeline: Chronological event log with sources
  • Impact: What was affected, for how long, data exposure
  • Root Cause: Technical and process/organizational causes
  • What Went Well: Detection, response, tooling that worked
  • What Needs Improvement: Gaps, delays, missing capabilities
  • Action Items: Specific, owned, time-bound improvements
  • Lessons Learned: Insights for the broader organization
Blameless culture: Focus on systems and processes, not individuals. "How did the system allow this?" not "Who caused this?"
Blameless postmortems are critical for organizational learning. If people fear blame, they will hide information. The goal is to make the system better, not to punish individuals. Every incident is a learning opportunity.

🔄 Section 6

Recovery & Hardening

With the incident analyzed and documented, we move to recovery — restoring services and hardening the environment to prevent recurrence.

🛠️ Remediation Steps

Immediate

  • Delete compromised pods
  • Deploy patched images
  • Complete credential rotation
  • Uncordon nodes after verification

Short-Term

  • Patch vulnerable images
  • Tighten RBAC policies
  • Add missing NetworkPolicies
  • Enable admission controllers

Long-Term

  • Implement image scanning in CI
  • Automate policy enforcement
  • Improve monitoring coverage
  • Schedule regular exercises
Remediation has three timescales. Immediate actions restore service. Short-term actions close the specific gap that was exploited. Long-term actions address the systemic issues identified in the root cause analysis.

✅ Validating Clean State

# Verify no unauthorized workloads remain
kubectl get pods --all-namespaces -o wide | grep -v kube-system

# Check for rogue ServiceAccounts and RoleBindings
kubectl get sa --all-namespaces | grep -v default
kubectl get rolebindings,clusterrolebindings --all-namespaces

# Scan all running images for vulnerabilities
kubectl get pods --all-namespaces \
  -o jsonpath='{.items[*].spec.containers[*].image}' | tr ' ' '\n' | sort -u

# Verify NetworkPolicies are enforced
kubectl get networkpolicies --all-namespaces

# Check for hostPath mounts or privileged pods
kubectl get pods --all-namespaces -o json | \
  jq '.items[] | select(.spec.volumes[]?.hostPath != null) |
  {namespace: .metadata.namespace, name: .metadata.name}'

# Verify no unexpected CronJobs or DaemonSets
kubectl get cronjobs,daemonsets --all-namespaces
After remediation, you must validate that the cluster is clean. An attacker may have established persistence mechanisms — DaemonSets, CronJobs, or modified RBAC — that survive the initial cleanup. Check everything systematically.

📄 Policy Updates Based on Findings

Detection Rule Updates

# New Falco rule from IOCs
- rule: Detect C2 Communication
  desc: Outbound connection to
        known C2 infrastructure
  condition: >
    evt.type = connect and
    fd.sip in (c2_ip_list) and
    container
  output: >
    C2 connection detected
    (pod=%k8s.pod.name
     ip=%fd.sip)
  priority: CRITICAL

Admission Policy Updates

# OPA/Gatekeeper constraint
apiVersion: constraints
         .gatekeeper.sh/v1beta1
kind: K8sBlockHostPath
metadata:
  name: block-host-path
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
    excludedNamespaces:
      - kube-system
  parameters:
    allowedPaths: []
Every incident should result in updated detection rules and policies. Feed the IOCs directly into your Falco rules. Add new admission controller constraints to prevent the attack vector from being exploitable in the future.

📢 Communication Plan

Stakeholder Communication Matrix

Audience When Content
IR Team Immediately Technical details, actions needed
Engineering Leadership Within 1 hour Impact summary, ETA for resolution
Executive Team Within 4 hours Business impact, risk assessment
Legal / Compliance Within 24 hours Data exposure, regulatory implications
Customers (if applicable) Per regulation Impact, remediation, next steps
Communication is often the most under-practiced aspect of IR. Technical teams focus on the technical response, but stakeholder communication is equally important. Pre-draft communication templates for each audience and severity level.

🎮 Section 7

Tabletop Exercises

Tabletop exercises are how you test your IR plan without the pressure of a real incident. They build muscle memory, expose gaps in your process, and improve team coordination.

🎯 Why Tabletop Exercises?

Benefits

  • Test IR procedures without risk
  • Identify gaps in runbooks
  • Build team communication skills
  • Validate tooling and access
  • Train new team members
  • Meet compliance requirements

Frequency & Format

  • Quarterly: Full team tabletop (2-3 hours)
  • Monthly: Mini scenarios (30-60 min)
  • After incidents: Replay the real event
  • Rotate facilitators for fresh perspectives
  • Include non-technical stakeholders periodically
  • Document outcomes and action items
Organizations that run regular tabletop exercises consistently outperform those that do not in actual incident response. It is like a fire drill — you practice so the real event is not the first time you execute the plan.

📝 Designing Effective Scenarios

Scenario Design Principles

  • Realistic: Based on actual attack patterns (MITRE ATT&CK for Containers)
  • Progressive: Start simple, introduce complications via "injects"
  • Role-specific: Each participant has defined responsibilities
  • Time-pressured: Include realistic constraints and deadlines
  • Decision-focused: Force trade-off decisions (availability vs. security)
Avoid: Scenarios that are impossible to solve or that have only one "correct" answer. The goal is to exercise the process, not to trick participants.
Good scenarios have multiple valid response paths. The discussion about which path to take is where the real learning happens. Include realistic complications like tool failures, key personnel being unavailable, or a second incident occurring during response.

💡 Example Scenario: Compromised Container

Scenario: Lateral Movement from Compromised Pod

Setting: Production Kubernetes cluster running an e-commerce platform

Inject 1 (T+0): Falco alert — "Shell spawned in container checkout-svc in namespace prod."
Inject 2 (T+10 min): Audit log shows the pod's ServiceAccount listing secrets across three namespaces.
Inject 3 (T+20 min): Network monitoring detects outbound data transfer to an external IP. Database credentials were among the listed secrets.
Inject 4 (T+30 min): A new DaemonSet appears in the kube-system namespace running an unknown image.
This scenario escalates progressively. Inject 1 is detection. Inject 2 reveals lateral movement. Inject 3 introduces data exfiltration. Inject 4 shows persistence. Each inject forces the team to reassess and make decisions about priorities.

❓ Scenario Discussion Points

Decision Points

  • Do you kill the pod immediately or preserve it for evidence?
  • When do you escalate to P0?
  • Do you shut down the entire namespace or isolate surgically?
  • When do you notify customers?
  • How do you handle the rogue DaemonSet in kube-system?

Evaluation Criteria

  • Time to first containment action
  • Evidence preserved before deletion
  • Correct severity escalation timing
  • Credential rotation completeness
  • Communication cadence and clarity
  • Clean-state validation thoroughness
There are trade-offs at every step. Killing the pod stops the attacker but destroys evidence. Isolating surgically is slower but preserves more. These discussions build judgment that you cannot get from reading a runbook alone.

🏃 Running the Exercise

Exercise Flow

  1. Briefing (10 min): Set the scene, assign roles, review ground rules
  2. Inject Phase (40 min): Deliver injects, observe and record team responses
  3. Hot Wash (15 min): Immediate reactions — what went well, what was confusing
  4. Debrief (20 min): Structured discussion of decision points and alternatives
  5. Action Items (5 min): Assign owners for identified improvements
The facilitator's job is to create pressure, not to judge. Let the team make decisions, then explore alternatives in the debrief.
The hot wash immediately after is critical — capture reactions while the experience is fresh. The debrief is where the deeper learning happens. Always end with concrete, owned action items that improve the IR process.

📈 IR Maturity Model

Level Description Characteristics
1 — Reactive No formal process Ad-hoc response, no runbooks, hero-dependent
2 — Defined Documented procedures Runbooks exist, roles assigned, basic tools
3 — Practiced Regular exercises Tabletops, metrics tracked, continuous improvement
4 — Automated Automated response Auto-containment, SOAR integration, ML-driven detection
5 — Optimized Proactive threat hunting Red team exercises, threat intelligence-driven, minimal MTTD
Most organizations are between Level 1 and 2. This course aims to get you to Level 3.
Be honest about where your organization sits on this maturity model. It is more useful to be solidly at Level 2 than to claim Level 4 with untested automation. Each level builds on the one before it.

🧪 Lab 12: Incident Response Tabletop

Duration: 60 minutes

  • Investigate a pre-staged compromise scenario
  • Collect forensic evidence from running pods
  • Perform containment using NetworkPolicies and RBAC
  • Document findings and create an incident report
This is the capstone lab. Students will work through a realistic incident scenario where a compromised pod is running in the cluster. They need to triage, collect evidence, contain the threat, and write up their findings. Emphasize the importance of evidence collection before containment actions that might destroy data.

📌 Key Takeaways

  • Preparation is paramount — have runbooks, tools, and roles defined before an incident occurs
  • Speed of evidence collection is the biggest K8s IR challenge — ephemeral pods destroy evidence on restart
  • Contain surgically — use NetworkPolicies and label changes to isolate while preserving evidence
  • Collect evidence before containment when possible — you cannot capture what no longer exists
  • Ephemeral debug containers are essential for forensics on minimal or distroless images
  • Blameless postmortems drive organizational learning and improve future response
  • Tabletop exercises build muscle memory — practice regularly so the real event is not the first time
  • Feed IOCs back into detection rules and admission policies to prevent recurrence
These are the essential takeaways for incident response in Kubernetes. The overarching theme is: prepare, detect fast, preserve evidence, contain carefully, learn thoroughly, and practice regularly.

📋 IR Quick Reference Card

Phase 1: Detect & Triage

  • ☐ Identify alert source and affected resources
  • ☐ Classify severity (P0-P3)
  • ☐ Assess blast radius (RBAC + network + secrets)
  • ☐ Notify Incident Commander

Phase 2: Collect Evidence

  • ☐ Run forensic capture script
  • ☐ Attach ephemeral debug container
  • ☐ Snapshot container filesystem
  • ☐ Hash and store all evidence

Phase 3: Contain

  • ☐ Apply deny-all NetworkPolicy
  • ☐ Quarantine pod (relabel)
  • ☐ Revoke RBAC and rotate credentials
  • ☐ Cordon affected node

Phase 4: Recover & Improve

  • ☐ Deploy patched workloads
  • ☐ Validate clean state
  • ☐ Conduct blameless postmortem
  • ☐ Update detection rules and policies
Print this quick reference card and keep it accessible to the IR team. It serves as a rapid checklist during high-pressure incidents when detailed runbooks may be too verbose to follow.
