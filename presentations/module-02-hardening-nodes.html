🛡️ Section 1

Node Attack Surface

We start by understanding what we need to protect before deciding how to protect it.

What Is a Node?

Nodes are the physical or virtual machines that run your Kubernetes workloads.

Every node runs:

  • kubelet — the primary node agent
  • Container runtime — containerd, CRI-O, etc.
  • kube-proxy — network rules management
  • OS services — systemd, kernel, networking stack
Establish baseline understanding. Even experienced operators sometimes overlook the breadth of software running on each node.

Three Dimensions of Attack Surface

🌐 Network

  • Open ports
  • Kubelet API (10250)
  • Read-only port (10255)
  • NodePort services
  • SSH access

📁 Filesystem

  • Config files
  • Secrets on disk
  • Container images
  • etcd data directory
  • PKI certificates

⚙️ Process

  • kubelet process
  • Container runtime
  • System services
  • Kernel modules
  • Cron jobs
Each dimension represents a different category of attack vectors. An adversary may target any of these to gain initial access or escalate privileges.

Why Node Security Matters

Node compromise = cluster compromise path

A compromised node gives an attacker access to:

  • kubeconfig — credentials for API server communication
  • Container runtime socket — ability to run arbitrary containers
  • Service account tokens — mounted in every pod by default
  • Network access — lateral movement to other nodes and pods
  • Secrets in memory — environment variables and volume-mounted secrets
This is the core motivation for node hardening. A single compromised node can be the starting point for full cluster takeover.

Real-World Attack Path

Scenario: Attacker exploits an application vulnerability

  1. Gain code execution inside a container
  2. Escape container via runtime vulnerability or misconfiguration
  3. Access kubelet credentials on the node filesystem
  4. Use kubelet identity to list secrets across namespaces
  5. Pivot to control plane or other workloads
🔑 Node hardening blocks steps 2–5 of this attack chain.
Walk through a concrete attack scenario to illustrate the importance of defense-in-depth at the node level.

💻 Section 2

OS-Level Hardening

The operating system is the first layer of defense. Choosing the right OS and configuring it properly eliminates entire categories of attack.

Container-Optimized OS Philosophy

Traditional Server OS

  • General-purpose package set
  • Mutable filesystem
  • Manual updates via SSH
  • SSH access by default
  • Wide attack surface

Container-Optimized OS

  • Minimal package footprint
  • Immutable root filesystem
  • Transactional auto-updates
  • No SSH (or disabled)
  • Reduced attack surface
💡 Fewer packages = fewer CVEs = smaller attack surface.
The philosophy is simple: include only what is needed to run containers, nothing more. Every additional package is a potential vulnerability.

Minimal OS Images Comparison

Feature Flatcar Container Linux Bottlerocket (AWS) Talos Linux Ubuntu Minimal
SSH Access Optional Admin container only None (API only) Yes
Filesystem Immutable rootfs Immutable rootfs Fully immutable Mutable
Auto-Updates Yes (Nebraska) Yes (orchestrator) Yes (rolling) Manual / unattended-upgrades
Package Count ~350 ~200 ~150 ~600+
Update Mechanism A/B partition A/B partition In-place via API apt / dpkg
Each of these OSes takes a different approach, but they all share the philosophy of minimalism. Talos Linux is the most extreme — no SSH at all, managed entirely via API.

Choosing the Right OS

✅ Recommended For

  • Talos / Bottlerocket: Production clusters with automation
  • Flatcar: Multi-cloud deployments
  • Ubuntu Minimal: Teams needing SSH troubleshooting access

⚠️ Consider

  • Team familiarity and operational tooling
  • Cloud provider integration
  • Compliance requirements
  • Debugging and incident response needs
The most secure option may not be the most practical for every team. Balance security with operational reality.

Kernel Hardening with sysctl

Tune kernel parameters to reduce attack surface:

# /etc/sysctl.d/99-kubernetes-hardening.conf

# Disable ICMP redirects (prevent MITM attacks)
net.ipv4.conf.all.send_redirects = 0
net.ipv4.conf.default.accept_redirects = 0
net.ipv4.conf.all.accept_redirects = 0
net.ipv6.conf.all.accept_redirects = 0

# Disable IP source routing
net.ipv4.conf.all.accept_source_route = 0
net.ipv6.conf.all.accept_source_route = 0

# Enable reverse path filtering
net.ipv4.conf.all.rp_filter = 1
net.ipv4.conf.default.rp_filter = 1

# Kernel panic behavior
kernel.panic = 10
kernel.panic_on_oops = 1

# Prevent OOM killer from crashing the node
vm.panic_on_oom = 0
vm.overcommit_memory = 1
These sysctl settings are widely recommended by CIS benchmarks and security frameworks. Apply them via configuration management and validate with kube-bench.

Applying sysctl Settings

# Apply immediately
sudo sysctl --system

# Verify settings
sysctl net.ipv4.conf.all.send_redirects
# Expected output: net.ipv4.conf.all.send_redirects = 0

# Verify at boot with systemd
sudo systemctl status systemd-sysctl.service
Note: Some sysctl parameters can also be set per-pod using Kubernetes securityContext.sysctls, but node-level settings provide a baseline that applies regardless of pod configuration.
Demonstrate both immediate application and persistence across reboots. Node-level sysctls are defense-in-depth below what pods can configure.

Filesystem Hardening

Critical Paths to Protect

# Kubelet configuration
chmod 644 /etc/kubernetes/kubelet.conf
chmod 600 /var/lib/kubelet/config.yaml

# PKI certificates
chmod 644 /etc/kubernetes/pki/*.crt
chmod 600 /etc/kubernetes/pki/*.key

# Kubeconfig files
chmod 600 /etc/kubernetes/admin.conf
chmod 644 /etc/kubernetes/kubelet.conf

Ownership Requirements

# All k8s configs owned by root
chown root:root /etc/kubernetes/*.conf
chown root:root /etc/kubernetes/pki/*

# Kubelet data directory
chown root:root /var/lib/kubelet/
chmod 750 /var/lib/kubelet/

# Container runtime socket
chmod 660 /var/run/containerd/
                            containerd.sock
File permissions are a CIS Benchmark requirement. Incorrect permissions on certificate private keys or kubeconfigs can allow local privilege escalation.

⚙️ Section 3

Kubelet Security Configuration

The kubelet is the most critical process on the node. Its configuration directly determines how secure the node is within the cluster.

Kubelet Attack Surface

The kubelet exposes powerful APIs on each node:

  • Port 10250 — Full kubelet API (exec, logs, port-forward)
  • Port 10255 — Read-only API (pod info, metrics)
  • Port 10248 — Health check endpoint
An unauthenticated kubelet API lets any network attacker exec into pods, read logs, and access node metadata.
In many default installations, the kubelet API on port 10250 allows anonymous access. This is one of the most common misconfigurations found in cluster assessments.

Critical Kubelet Flags

# Disable anonymous authentication to kubelet API
--anonymous-auth=false

# Use Webhook authorization (delegate to API server)
--authorization-mode=Webhook

# Disable the read-only port entirely
--read-only-port=0

# Protect kernel defaults (fail if sysctl params don't match)
--protect-kernel-defaults=true

# Ensure iptables rules are managed by kubelet
--make-iptables-util-chains=true

# Disable event throttling for audit completeness
--event-qps=0
🔑 Each flag closes a specific attack vector. Apply all of them together.
Go through each flag and explain what attack it prevents. anonymous-auth=false is the single most impactful change you can make.

Kubelet Flags Explained

--anonymous-auth=false

Prevents unauthenticated requests to the kubelet API. Without this, anyone who can reach port 10250 can exec into pods.

--authorization-mode=Webhook

Delegates authorization decisions to the API server instead of always allowing authenticated requests.

--read-only-port=0

Disables the unauthenticated read-only kubelet endpoint on port 10255 that exposes pod and node information.

--protect-kernel-defaults

Causes kubelet to error if kernel parameters differ from expected hardened values, enforcing sysctl compliance.

Spend time on anonymous-auth and authorization-mode — these are the two most critical settings and the ones most often misconfigured.

KubeletConfiguration YAML

apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
authentication:
  anonymous:
    enabled: false
  webhook:
    enabled: true
    cacheTTL: 2m0s
  x509:
    clientCAFile: /etc/kubernetes/pki/ca.crt
authorization:
  mode: Webhook
  webhook:
    cacheAuthorizedTTL: 5m0s
    cacheUnauthorizedTTL: 30s
readOnlyPort: 0
protectKernelDefaults: true
makeIPTablesUtilChains: true
eventRecordQPS: 0
tlsCertFile: /etc/kubernetes/pki/kubelet.crt
tlsPrivateKeyFile: /etc/kubernetes/pki/kubelet.key
rotateCertificates: true
serverTLSBootstrap: true
featureGates:
  RotateKubeletServerCertificate: true
This is the modern, declarative way to configure kubelet. The YAML config file replaces command-line flags and is easier to manage with configuration management tools.

Kubelet Certificate Rotation

Problem: Static kubelet certificates can be stolen and reused indefinitely.

Solution: Automatic certificate rotation limits the window of compromise.

# In KubeletConfiguration
rotateCertificates: true         # Client cert rotation
serverTLSBootstrap: true         # Server cert rotation

# Feature gate (if not default)
featureGates:
  RotateKubeletServerCertificate: true
Tip: You must approve CSRs for server certificate rotation. Automate this with a controller or configure auto-approval for kubelet-serving CSRs.
Certificate rotation is enabled by default in recent Kubernetes versions for client certificates, but server certificate rotation requires explicit opt-in and CSR approval.

Verifying Kubelet Security

# Test anonymous access (should return 401)
curl -sk https://localhost:10250/pods
# Expected: Unauthorized

# Verify read-only port is closed
curl -s http://localhost:10255/healthz
# Expected: Connection refused

# Check kubelet process flags
ps aux | grep kubelet | grep -o '\-\-[a-z-]*'

# Inspect active kubelet configuration
kubectl get --raw "/api/v1/nodes/$(hostname)/proxy/configz" \
  | jq '.kubeletconfig'
Warning: If curl https://localhost:10250/pods returns pod data, your kubelet is accepting anonymous requests!
Always verify your configuration. These commands can be incorporated into automated security tests or CI/CD pipelines.

🛡️ Section 4

AppArmor and SELinux

Mandatory Access Control provides an additional layer of defense that limits what processes can do, even if they are running as root.

Mandatory Access Control (MAC)

MAC restricts what processes can access based on security policies defined by administrators, regardless of the process's user ID or capabilities.

AppArmor

  • Path-based access control
  • Easier to write and audit
  • Default on Ubuntu/Debian
  • Profile per application

SELinux

  • Label-based access control
  • More granular control
  • Default on RHEL/CentOS
  • Context per file and process
Most teams should use whichever MAC system ships with their chosen OS rather than switching. Both provide strong security when configured properly.

When to Use Each

Criteria AppArmor SELinux
OS Ubuntu, Debian, SUSE RHEL, CentOS, Fedora
Learning Curve Lower Higher
Policy Model Path-based Label-based (Type Enforcement)
Kubernetes Support GA since 1.31 GA since 1.27
Best For Quick adoption, readable policies High-security, compliance-driven
💡 Use whichever MAC system is native to your chosen OS. Do not disable it.
The most common mistake is disabling SELinux or AppArmor because it interferes with application deployment. This is a significant security regression.

AppArmor Profile for Containers

# /etc/apparmor.d/k8s-deny-write-etc
#include <tunables/global>

profile k8s-deny-write-etc flags=(attach_disconnected,mediate_deleted) {
  #include <abstractions/base>

  # Allow read access everywhere
  file,

  # Deny write access to sensitive paths
  deny /etc/** w,
  deny /proc/sys/** w,
  deny /sys/** w,

  # Allow container runtime needs
  /dev/null rw,
  /dev/zero rw,
  /dev/urandom r,

  # Allow tmp writes
  /tmp/** rw,
  /var/tmp/** rw,
}
# Load the profile
sudo apparmor_parser -r /etc/apparmor.d/k8s-deny-write-etc
sudo aa-status | grep k8s-deny-write-etc
This is a practical profile that allows normal container operation while preventing writes to sensitive system directories. The attach_disconnected flag is needed for container runtimes.

Applying AppArmor to Pods

apiVersion: v1
kind: Pod
metadata:
  name: apparmor-demo
spec:
  containers:
  - name: app
    image: nginx:1.25
    securityContext:
      appArmorProfile:
        type: Localhost
        localhostProfile: k8s-deny-write-etc
    resources:
      limits:
        memory: "128Mi"
        cpu: "250m"
Kubernetes 1.30+: Use the securityContext.appArmorProfile field (GA). Older clusters may require the annotation container.apparmor.security.beta.kubernetes.io/<name>.
The securityContext field is the modern approach. The annotation-based method is deprecated but still supported for backward compatibility.

SELinux Context Configuration

apiVersion: v1
kind: Pod
metadata:
  name: selinux-demo
spec:
  securityContext:
    seLinuxOptions:
      level: "s0:c123,c456"
  containers:
  - name: app
    image: nginx:1.25
    securityContext:
      seLinuxOptions:
        type: "container_t"
        level: "s0:c123,c456"
    resources:
      limits:
        memory: "128Mi"
        cpu: "250m"
# Verify SELinux mode on the node
getenforce
# Expected: Enforcing

# Check container process labels
ps -eZ | grep container_t
SELinux uses Multi-Category Security (MCS) labels to isolate containers from each other. Each container gets a unique category pair by default.

CIS Benchmark for Nodes

The CIS Kubernetes Benchmark provides a comprehensive checklist for securing your cluster. Section 4 specifically addresses worker node security.

CIS Kubernetes Benchmark — Section 4

The CIS Benchmark for Kubernetes Worker Nodes covers:

4.1 Worker Node Config Files

  • kubelet service file permissions
  • kubelet.conf file permissions
  • kubelet config.yaml permissions
  • Certificate file ownership

4.2 Kubelet Configuration

  • Anonymous auth disabled
  • Webhook authorization
  • Read-only port disabled
  • TLS configuration
  • Certificate rotation
Section 4 has two subsections. 4.1 is about file permissions and ownership. 4.2 is about runtime configuration. Both are equally important.

Key CIS Checks for Nodes

Check ID Description Expected Value
4.1.1 kubelet service file permissions 644 or more restrictive
4.1.5 kubelet.conf ownership root:root
4.2.1 Anonymous authentication false
4.2.2 Authorization mode Webhook
4.2.4 Read-only port 0
4.2.6 Protect kernel defaults true
4.2.10 TLS cert and key configured Present
4.2.11 Certificate rotation true
These are the most critical checks. Many compliance frameworks reference CIS Benchmarks directly, making them a practical compliance tool.

Running kube-bench

kube-bench automates CIS Benchmark checks:

# Run node-specific checks
kube-bench run --targets node

# Run as a Kubernetes Job
kubectl apply -f https://raw.githubusercontent.com/aquasecurity/\
  kube-bench/main/job-node.yaml

# View results
kubectl logs job/kube-bench-node

# Run specific checks only
kube-bench run --targets node --check 4.2.1,4.2.2,4.2.4
kube-bench outputs results as PASS, FAIL, WARN, or INFO for each check, with remediation instructions for failures.
kube-bench is the de facto standard tool for CIS compliance checking. Run it as a DaemonSet for continuous monitoring or as a one-off Job for audits.

Interpreting kube-bench Results

[INFO] 4 Worker Node Security Configuration
[INFO] 4.1 Worker Node Configuration Files
[PASS] 4.1.1 Ensure kubelet service file permissions are set to 644
[PASS] 4.1.2 Ensure kubelet service file ownership is root:root
[FAIL] 4.2.1 Ensure anonymous-auth is set to false

== Summary ==
15 checks PASS
2 checks FAIL
1 checks WARN
0 checks INFO

== Remediations node ==
4.2.1 Set the --anonymous-auth argument to false.
If using a Kubelet config file, edit the file to set:
  authentication:
    anonymous:
      enabled: false

🔄 Section 6

Node Maintenance and Updates

Hardening is not a one-time activity. Nodes need ongoing maintenance to stay secure as new vulnerabilities are discovered.

Patch Management Strategy

Critical (24h)

  • Actively exploited CVEs
  • Container runtime escapes
  • Kernel privilege escalation

High (7 days)

  • Remote code execution
  • Authentication bypass
  • Kubelet vulnerabilities

Standard (30 days)

  • Non-critical kernel CVEs
  • Library updates
  • Minor version upgrades
📅 Define SLAs for each severity level and automate wherever possible.
These are example SLAs — adjust based on your organization's risk tolerance and compliance requirements. The key is having defined timelines rather than ad-hoc patching.

Node Draining and Cordoning

# Mark node as unschedulable (no new pods)
kubectl cordon worker-node-01

# Drain node (evict all pods gracefully)
kubectl drain worker-node-01 \
  --ignore-daemonsets \
  --delete-emptydir-data \
  --grace-period=120 \
  --timeout=300s

# Perform maintenance (OS updates, kubelet upgrade)
ssh worker-node-01 'sudo apt update && sudo apt upgrade -y'

# Return node to service
kubectl uncordon worker-node-01
Always ensure PodDisruptionBudgets (PDBs) are in place before draining to prevent service disruption.
Draining nodes safely requires PDBs to be configured on critical workloads. Without PDBs, a drain can take down all replicas of a service simultaneously.

Automated Node Updates

kured (Kubernetes Reboot Daemon)

  • Watches for /var/run/reboot-required
  • Cordons, drains, reboots, uncordons
  • One node at a time (lock-based)
  • Configurable maintenance windows
helm install kured \
  kured/kured \
  --set configuration.period=1h \
  --set configuration.startTime="02:00" \
  --set configuration.endTime="06:00"

node-problem-detector

  • Detects node health issues
  • Kernel deadlocks, OOM events
  • Container runtime failures
  • Reports via node conditions
kubectl get nodes \
  -o jsonpath='{range .items[*]}
  {.metadata.name}:
  {.status.conditions[*].type}
  {"\n"}{end}'
kured automates the reboot cycle after OS updates. node-problem-detector identifies issues that may require node replacement. Together they provide automated node lifecycle management.

Immutable Node Replacement Pattern

Instead of patching nodes in place, replace them entirely.

  1. Build a new hardened node image (Packer, image builder)
  2. Launch replacement node with the new image
  3. Wait for the node to join the cluster and pass health checks
  4. Cordon and drain the old node
  5. Terminate the old node
Benefits: No configuration drift, guaranteed clean state, auditable image history, easy rollback by reverting to previous image version.
This is the gold standard for node management in cloud environments. It treats nodes as cattle, not pets. Works well with auto-scaling groups and managed node pools.

Key Takeaways

  • 🛡️ Nodes are the foundation — A compromised node provides a direct path to cluster compromise via kubelet credentials, runtime sockets, and service account tokens.
  • 💻 Use container-optimized OSes — Minimal, immutable OS images dramatically reduce the attack surface and simplify patching.
  • ⚙️ Lock down the kubelet — Disable anonymous auth, enable webhook authorization, close the read-only port, and rotate certificates.
  • 🔒 Enable MAC enforcement — AppArmor or SELinux provides mandatory access controls that limit damage from container escapes.
  • 📋 Validate with CIS Benchmarks — Use kube-bench to continuously verify node compliance and catch configuration drift.
  • 🔄 Automate node lifecycle — Implement immutable node replacement and automated updates to maintain security posture over time.
