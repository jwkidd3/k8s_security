Runtime security complements all the preventive controls we've studied. Even with perfect policies, you need visibility into what's actually happening inside your cluster.

Runtime Threats

Understanding What Attackers Do Post-Compromise

Before building detection, we need to understand what we're detecting. Let's map the common runtime attack patterns.

Container Escape Techniques

Privileged Container Escape

  • Mount host filesystem via /dev
  • Access host PID namespace
  • Load kernel modules
  • Exploit CAP_SYS_ADMIN

Kernel Exploit Escape

  • CVE-2022-0185 (filesystem context)
  • CVE-2022-0492 (cgroups escape)
  • Dirty Pipe (CVE-2022-0847)
  • runc vulnerabilities
Detection signal: Processes accessing /proc/1/root, mounting block devices, or writing to host paths from within containers.
Container escapes are the most critical runtime threat. A successful escape gives the attacker full host access, which typically means access to all containers on that node.

Common Runtime Attack Patterns

Crypto Mining

  • High CPU usage
  • Connections to mining pools
  • Unexpected binaries
  • Stratum protocol traffic

Reverse Shells

  • Outbound connections
  • Shell spawned by web server
  • stdin/stdout redirection
  • Netcat/socat usage

Privilege Escalation

  • SUID binary exploitation
  • Service account token theft
  • kubectl exec abuse
  • Node-to-cluster pivoting
These are the top three categories of post-exploitation activity in Kubernetes environments. Crypto mining is the most common — it's the "canary" that tells you your cluster has been compromised.

Lateral Movement in Kubernetes

Attack chain: Compromise pod → Steal service account token → Query API server → Discover secrets → Pivot to other namespaces

Network-Based

  • Pod-to-pod communication (flat network)
  • Service discovery via DNS
  • Accessing metadata API (169.254.169.254)
  • Scanning internal services

API-Based

  • Listing secrets across namespaces
  • Creating privileged pods
  • Modifying RBAC bindings
  • Accessing etcd directly
Lateral movement is often where runtime detection provides the highest value. Network policies prevent some of this, but monitoring catches the attempts.

MITRE ATT&CK for Containers

Tactic Technique Runtime Detection
Initial Access Exploit public-facing app Unexpected process in container
Execution Exec into container Shell spawned in container
Persistence Backdoor container image New binaries written to container
Privilege Escalation Privileged container Mount of sensitive host paths
Defense Evasion Clear container logs Log file deletion detected
Credential Access Access K8s secrets Sensitive file reads in /etc
Lateral Movement Access K8s API Unexpected API server connections
The MITRE ATT&CK framework for containers gives us a structured way to think about what we need to detect. Each tactic maps to specific runtime signals we can monitor for.

Falco Architecture

Cloud-Native Runtime Security

Falco is the CNCF graduated project for runtime threat detection. It's the de facto standard for Kubernetes runtime security.

How Falco Works

System Call Interception

Falco intercepts system calls at the kernel level, applies rules in userspace, and generates alerts when suspicious activity is detected.

Data Flow

  1. Kernel driver captures syscalls
  2. Events streamed to userspace
  3. Rules engine evaluates conditions
  4. Matching events trigger outputs
  5. Alerts sent to configured channels

What Falco Sees

  • File opens, reads, writes
  • Process creation (execve)
  • Network connections
  • Container lifecycle events
  • Kubernetes audit events
Falco's power comes from operating at the kernel level. It sees everything that happens in every container without requiring any modifications to your workloads.

eBPF Driver vs Kernel Module

eBPF Driver (Recommended)

  • No kernel module loading required
  • Works in kind/minikube clusters
  • Safer — runs in eBPF sandbox
  • Easier to deploy and update
  • Supported on Linux 5.8+

Kernel Module

  • Requires kernel headers
  • Must match kernel version exactly
  • Higher risk of kernel panic
  • Better performance in some cases
  • Legacy — being phased out
For our labs: We use the eBPF driver in kind clusters. Set driver.kind=ebpf in the Helm values.
The eBPF driver is the modern approach and what we'll use in the lab. It's much easier to deploy and doesn't require privileged access to load kernel modules.

Falco Output Channels

Standard

  • stdout / stderr
  • Syslog
  • File output
  • Program output (pipe)

gRPC

  • Streaming API
  • Used by Falcosidekick
  • Structured event data
  • High throughput

HTTP

  • Webhook endpoints
  • JSON payloads
  • Custom integrations
  • Cloud functions
In production, you'll typically use the gRPC output with Falcosidekick as a fan-out mechanism to multiple destinations. We'll cover that in the detection pipeline section.

Deploying Falco in Kubernetes

Helm Installation with eBPF

# Add the Falco Helm repo
helm repo add falcosecurity https://falcosecurity.github.io/charts
helm repo update

# Install Falco with eBPF driver
helm install falco falcosecurity/falco \
  --namespace falco --create-namespace \
  --set driver.kind=ebpf \
  --set falcosidekick.enabled=true \
  --set falcosidekick.webui.enabled=true
DaemonSet deployment: One Falco pod per node ensures complete coverage of all containers.
Resource requirements: Falco typically uses 200-500m CPU and 256-512Mi memory per node.
DaemonSet is the standard deployment model. Sidecar injection is possible but rarely used — DaemonSet gives you full node visibility without modifying workloads.

Custom Falco Rules

Writing Detection Logic for Your Environment

Falco ships with excellent default rules, but custom rules tailored to your environment are where you get the most value.

Falco Rule Anatomy

- rule: Detect Shell in Container
  desc: Alert when a shell is spawned inside a container
  condition: >
    spawned_process and container and
    proc.name in (bash, sh, zsh, csh, ksh) and
    not proc.pname in (health_check, crond)
  output: >
    Shell spawned in container
    (user=%user.name container=%container.name
    shell=%proc.name parent=%proc.pname
    cmdline=%proc.cmdline image=%container.image.repository)
  priority: WARNING
  tags: [container, shell, mitre_execution]
Five required fields: rule (name), desc (description), condition (filter), output (alert message), priority (severity level)
Every Falco rule has these five fields. The condition uses Falco's filter syntax based on Sysdig's, which provides access to hundreds of fields covering processes, files, network, and container metadata.

Macros and Lists

Macros (Reusable Conditions)

# A macro is a reusable condition
- macro: container
  condition: container.id != host

- macro: spawned_process
  condition: >
    evt.type = execve and
    evt.dir = <

- macro: sensitive_files
  condition: >
    fd.name startswith /etc and
    (fd.name in (/etc/shadow,
                 /etc/passwd,
                 /etc/sudoers))

Lists (Reusable Values)

# A list is a named collection
- list: shell_binaries
  items: [bash, sh, zsh, csh, ksh]

- list: mining_pools
  items: [
    "pool.minexmr.com",
    "xmr.pool.minergate.com",
    "pool.hashvault.pro"
  ]

- list: allowed_images
  items: [
    "nginx",
    "my-registry.io/app"
  ]
Macros and lists are critical for keeping your rules DRY and maintainable. The default Falco rules use macros extensively — you can override them to customize behavior.

Example: Detect Sensitive File Reads

- list: sensitive_file_paths
  items:
    - /etc/shadow
    - /etc/kubernetes/pki
    - /var/run/secrets/kubernetes.io/serviceaccount/token
    - /root/.kube/config

- rule: Read Sensitive File in Container
  desc: >
    Detect attempts to read sensitive files that may
    contain credentials or security-critical data
  condition: >
    open_read and container and
    fd.name pmatch (sensitive_file_paths) and
    not proc.name in (kubelet, kube-proxy)
  output: >
    Sensitive file read in container
    (file=%fd.name user=%user.name
    container=%container.name image=%container.image.repository
    command=%proc.cmdline)
  priority: WARNING
  tags: [container, filesystem, mitre_credential_access]
This rule watches for reads of files commonly targeted by attackers — service account tokens, SSH keys, shadow files. The exclusion for kubelet prevents false positives from legitimate system processes.

Example: Detect Crypto Mining

- list: mining_pool_domains
  items: ["pool.minexmr.com", "xmr.pool.minergate.com",
          "pool.hashvault.pro", "stratum+tcp"]

- list: mining_binaries
  items: [xmrig, minerd, cpuminer, cgminer, bfgminer]

- rule: Detect Crypto Mining Activity
  desc: Detect potential cryptocurrency mining activity
  condition: >
    (spawned_process and container and
     proc.name in (mining_binaries))
    or
    (container and evt.type in (connect, sendto) and
     fd.sip.name pmatch (mining_pool_domains))
  output: >
    Crypto mining detected
    (process=%proc.name connection=%fd.name
    container=%container.name namespace=%k8s.ns.name
    image=%container.image.repository)
  priority: CRITICAL
  tags: [container, crypto, mitre_resource_hijacking]
Crypto mining detection combines two signals: known mining binaries and connections to known mining pool domains. In production, also alert on sustained high CPU usage as an additional indicator.

Falco Priority Levels

Priority Use Case Response Time
EMERGENCY System is unusable, active breach Immediate
ALERT Action must be taken immediately Minutes
CRITICAL Container escape, crypto mining Minutes
ERROR Unexpected behavior, policy violation Hours
WARNING Shell in container, sensitive reads Hours
NOTICE Informational security events Daily review
DEBUG Rule development and testing As needed
Mapping priorities correctly is essential for managing alert fatigue. Only use CRITICAL and above for events that truly require immediate response.

Node-Level Monitoring

Securing the Underlying Infrastructure

Containers run on nodes, and node compromise is game over. Let's look at how to monitor the host layer.

Process Monitoring

What to Watch

  • Unexpected processes on worker nodes
  • Processes running as root
  • Modified system binaries
  • Processes with unusual parent chains
  • Orphaned processes outside containers
Key principle: Worker nodes should only run kubelet, container runtime, and kube-proxy. Anything else is suspicious. 
# Monitor processes on node
# Expected: kubelet, containerd,
# kube-proxy, system services

# Suspicious indicators:
ps aux | grep -E \
  'nc |ncat|socat|wget|curl' \
  --color

# Check for processes running
# outside any cgroup/container
systemd-cgls

# Monitor real-time process
# creation
auditctl -a always,exit \
  -F arch=b64 \
  -S execve \
  -k process_creation
Process monitoring on nodes is your last line of defense. If an attacker escapes a container, you want to detect the processes they spawn on the host.

File Integrity Monitoring (FIM)

Critical Paths to Monitor

PathReason
/etc/kubernetes/K8s configs & certs
/var/lib/kubelet/Kubelet data
/etc/cni/Network plugin config
/usr/bin/System binaries
/etc/systemd/Service definitions
/etc/shadowUser credentials

AIDE Configuration

# /etc/aide.conf
/etc/kubernetes CONTENT_EX
/var/lib/kubelet CONTENT_EX
/usr/bin CONTENT_EX

# Initialize database
aide --init

# Check for changes
aide --check

# Automate via cron
0 */4 * * * /usr/bin/aide \
  --check | mail -s \
  "AIDE Report" admin@co.com
File integrity monitoring detects modifications to critical system files. AIDE and OSSEC are common choices. Falco can also serve as a lightweight FIM solution using its file-related rules.

Network Connection Monitoring

Suspicious Network Signals

  • Outbound connections from worker nodes
  • Connections to non-standard ports
  • DNS queries to unusual domains
  • Large data transfers (exfiltration)
  • Connections to known C2 infrastructure
# Track node network connections
ss -tunap | grep ESTAB

# Monitor DNS queries
tcpdump -i any port 53 \
  -nn -l 2>/dev/null

# Falco network rule example
- rule: Unexpected Outbound Connection
  condition: >
    outbound and container and
    not fd.sport in (80, 443, 53) and
    not k8s.ns.name = kube-system
  output: >
    Unexpected outbound (port=%fd.sport
    dest=%fd.sip container=%container.name)
  priority: NOTICE
Network monitoring at the node level complements Kubernetes network policies. Even if a network policy blocks traffic, the attempt itself is a valuable detection signal.

Systemd Service Hardening

Securing Node Services

# /etc/systemd/system/kubelet.service.d/security.conf
[Service]
# Restrict filesystem access
ProtectSystem=full
ProtectHome=yes
ReadOnlyPaths=/etc/kubernetes

# Restrict capabilities
CapabilityBoundingSet=CAP_NET_ADMIN CAP_NET_BIND_SERVICE CAP_SYS_RESOURCE
NoNewPrivileges=yes

# Restrict network
RestrictAddressFamilies=AF_INET AF_INET6 AF_UNIX

# Restrict system calls
SystemCallFilter=@system-service
SystemCallArchitectures=native
Defense in depth: Even if an attacker reaches the node, systemd hardening limits what node-level services can be abused.
Systemd service hardening applies the principle of least privilege to node-level services. These settings are enforced by the kernel and cannot be bypassed by the service process.

Leveraging Built-In Observability

Kubernetes provides several built-in monitoring capabilities that are often overlooked from a security perspective.

Kubernetes Audit Logs

Security-Relevant Events

  • RBAC role/binding changes
  • Pod exec and attach operations

Falco + Audit Logs

# Falco can consume K8s audit logs
- rule: K8s Secret Accessed
  desc: Detect secret access
  condition: >
    ka.verb = get and
    ka.target.resource = secrets
    and not ka.user.name in
    (allowed_secret_readers)
  output: >
    Secret accessed (user=%ka.user.name
    secret=%ka.target.name
    ns=%ka.target.namespace)
  priority: WARNING
  source: k8s_audit
Falco's ability to consume Kubernetes audit logs is incredibly powerful. It means you can use the same rules engine for both syscall-level and API-level detection.

Tetragon & eBPF

Next-Generation Runtime Enforcement

Tetragon takes eBPF-based security beyond detection into real-time enforcement. Let's understand the technology and how it compares to Falco.

What is eBPF?

Extended Berkeley Packet Filter

eBPF allows running sandboxed programs in the Linux kernel without changing kernel source code or loading kernel modules. It provides safe, efficient, and programmable access to kernel events.

Observability

Trace any kernel or user-space function without modifying code

Security

Enforce policies at the kernel level with near-zero overhead

Networking

High-performance packet processing (Cilium, Calico eBPF)

eBPF is one of the most significant Linux kernel innovations in recent years. It's the foundation for Falco's modern driver, Tetragon, Cilium, and many other cloud-native tools.

Tetragon Architecture

Key Components

  • eBPF sensors: Kernel-level hooks for process, file, and network events
  • TracingPolicy CRD: Kubernetes-native policy definition
  • In-kernel enforcement: Block actions before they complete
  • Kubernetes-aware: Enriches events with pod/namespace context
Key difference: Tetragon can prevent actions, not just detect them. It kills the process or blocks the syscall in-kernel. 
# Install Tetragon with Helm
helm repo add cilium \
  https://helm.cilium.io
helm repo update

helm install tetragon \
  cilium/tetragon \
  --namespace kube-system

# Observe events in real-time
kubectl exec -n kube-system \
  ds/tetragon -c tetragon -- \
  tetra getevents -o compact
Tetragon is a CNCF project from Isovalent (the Cilium company). Its killer feature is in-kernel enforcement — it can kill a process at the kernel level before the syscall completes, which is faster and more reliable than userspace enforcement.

TracingPolicy: Tetragon's Policy Language

apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: detect-sensitive-file-access
spec:
  kprobes:
    - call: "fd_install"
      syscall: false
      args:
        - index: 0
          type: "int"
        - index: 1
          type: "file"
      selectors:
        - matchArgs:
            - index: 1
              operator: "Prefix"
              values:
                - "/etc/shadow"
                - "/etc/kubernetes/pki"
          matchActions:
            - action: Sigkill  # Kill the process immediately
Real-time enforcement: The Sigkill action terminates the offending process before it can read the file. This is enforcement, not just detection.
TracingPolicy is a Kubernetes CRD, so it fits naturally into GitOps workflows. The matchActions field is what makes Tetragon special — it can take enforcement actions directly in the kernel.

Falco vs Tetragon

Feature Falco Tetragon
Primary Purpose Detection & alerting Detection & enforcement
Data Source Syscalls + K8s audit eBPF kprobes/tracepoints
Enforcement Alert only (external response) In-kernel kill/block
Rule Language YAML with filter syntax Kubernetes CRD (TracingPolicy)
Community CNCF Graduated, large ecosystem CNCF, growing rapidly
Maturity Production-proven Maturing quickly
Best For Broad detection, audit integration Enforcement, deep kernel visibility
"Use Falco for comprehensive detection and Tetragon for targeted enforcement — they complement each other."
These tools are complementary, not competing. Many organizations run both — Falco for broad detection with its rich rule ecosystem, and Tetragon for targeted enforcement of critical policies.

When to Use Each Tool

Choose Falco When...

  • You need broad threat detection
  • You want to consume K8s audit logs
  • You need a mature rule ecosystem
  • You want Falcosidekick integrations
  • Detection and alerting is the goal
  • You need introductory-level setup

Choose Tetragon When...

  • You need real-time enforcement
  • You want to kill malicious processes
  • You need deep kernel visibility
  • You use Cilium for networking
  • You want Kubernetes-native CRDs
  • Blocking threats is required
In practice, start with Falco for visibility, then layer Tetragon for enforcement on your most critical policies. Running both gives you defense in depth at the runtime layer.

Detection Pipeline

From Alerts to Action

Detection is only useful if it reaches the right people at the right time. Let's build a complete detection pipeline.

Building a Detection Pipeline

Event Flow

Kernel Syscalls → Falco → Falcosidekick → Outputs (Slack, PagerDuty, SIEM, S3, etc.)

Collection

  • Falco DaemonSet on every node
  • K8s audit log webhook
  • Node-level agents

Processing

  • Falcosidekick fan-out
  • Priority-based routing
  • Event enrichment

Response

  • Slack/Teams notifications
  • PagerDuty escalation
  • Automated remediation
The pipeline must handle high-volume events efficiently and route them to the right destination based on severity. Falcosidekick is the key integration layer.

Falcosidekick: Alert Fan-Out

# Falcosidekick Helm values
falcosidekick:
  config:
    slack:
      webhookurl: "https://hooks.slack.com/services/XXX"
      minimumpriority: "warning"
      channel: "#security-alerts"

    pagerduty:
      routingkey: "your-integration-key"
      minimumpriority: "critical"

    elasticsearch:
      hostport: "http://elasticsearch:9200"
      index: "falco-events"
      minimumpriority: "notice"

    webhook:
      address: "http://response-engine:8080"
      minimumpriority: "error"
50+ output targets: Falcosidekick supports Slack, Teams, PagerDuty, Elasticsearch, Loki, AWS SNS/SQS/S3, GCP Pub/Sub, Azure Event Hubs, and many more.
Falcosidekick's minimumpriority setting is critical for alert routing. Critical events go to PagerDuty, warnings go to Slack, and everything goes to your SIEM for forensics.

Log Aggregation & SIEM Integration

Log Aggregation Stack

  • Elasticsearch + Kibana: Full-text search, dashboards
  • Loki + Grafana: Lightweight, label-based queries
  • Splunk: Enterprise SIEM capabilities
  • AWS CloudWatch: Cloud-native option
Retention: Keep security logs for at least 90 days. Compliance may require 1+ years.

What to Aggregate

  • Falco alerts (all priorities)
  • Kubernetes audit logs
  • Container stdout/stderr
  • Node-level syslog
  • Network flow logs
  • Authentication events
Centralized logging is non-negotiable for security. When investigating an incident, you need to correlate events across multiple data sources — container logs, audit logs, Falco alerts, and network flows.

Managing Alert Fatigue

Common Pitfalls

  • Too many low-priority alerts
  • Not tuning default rules
  • No exception mechanism
  • Alerting on expected behavior
  • Same alert to all channels

Best Practices

  • Start with high-priority rules only
  • Add exceptions for known behavior
  • Use priority-based routing
  • Review and tune rules weekly
  • Track alert-to-incident ratio
# Tuning Falco: add exceptions to reduce noise
- rule: Terminal shell in container
  append: true
  condition: and not container.image.repository in (debugging-tools, ci-runner)
  # Suppress alerts for known debugging containers
Alert fatigue is the number one killer of security monitoring programs. If your team ignores alerts because there are too many false positives, you have no runtime security. Start small and expand coverage gradually.

Automated Response

Caution: Automated response can cause outages if misconfigured. Start with detection-only and graduate to automated response carefully.

Response Actions

  • Network isolation: Apply deny-all NetworkPolicy
  • Pod termination: Delete compromised pod
  • Node cordon: Prevent new scheduling
  • Snapshot: Capture forensic evidence
  • Escalation: Page on-call engineer
# Falcosidekick response example
# Webhook triggers a Kubernetes job
# that isolates the pod
apiVersion: batch/v1
kind: Job
metadata:
  name: isolate-pod
spec:
  template:
    spec:
      containers:
        - name: responder
          image: bitnami/kubectl
          command:
            - /bin/sh
            - -c
            - |
              kubectl label pod $POD \
                quarantine=true
              kubectl apply -f \
                deny-all-policy.yaml
Automated response is the endgame, but proceed with caution. A false positive that triggers automatic pod deletion in production is worse than no automation at all. Test thoroughly in staging first.

Key Takeaways

  • Runtime security detects threats that bypass preventive controls — it's essential, not optional
  • Falco provides comprehensive syscall-based detection with a mature rule ecosystem
  • Custom Falco rules tailored to your environment deliver the highest detection value
  • Node-level monitoring (process, file integrity, network) secures the infrastructure layer
  • Kubernetes audit logs and events are free security telemetry — use them
  • Tetragon enables in-kernel enforcement — complementing Falco's detection capabilities
  • A complete detection pipeline routes alerts by severity to prevent alert fatigue
  • Start with detection, tune for accuracy, then graduate to automated response
These takeaways represent the core runtime security strategy. The lab will give hands-on experience with Falco, which is the foundation of most Kubernetes runtime security programs.
