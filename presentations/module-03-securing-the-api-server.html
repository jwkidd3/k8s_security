🏛 API Server Architecture & Request Flow

Understanding the front door to your cluster

The API Server — Cluster Front Door

kube-apiserver is the only component that talks directly to etcd. Every other component — scheduler, controller-manager, kubelet, kubectl — communicates through it.

Responsibilities

  • REST API for all cluster operations
  • Authentication & authorization
  • Admission control
  • API aggregation layer
  • Watch/notification mechanism

Security Implication

  • Single point of policy enforcement
  • Compromise = full cluster control
  • Must be hardened, monitored, and rate-limited
  • Network exposure must be minimized
Emphasize that kube-apiserver is effectively the brain of the cluster. If an attacker reaches it with valid credentials, the game is over.

API Server Ports

Port Purpose Security Notes
6443 (default secure) HTTPS API endpoint TLS required, all authn/authz enforced
8080 (legacy insecure) HTTP — no auth Removed in v1.20+. Never enable.
443 (cloud LBs) Load-balanced endpoint TLS termination at LB or passthrough
⚠ Historical Risk: The insecure port (--insecure-port) bypassed all authentication and authorization. Clusters running Kubernetes < 1.20 must explicitly set --insecure-port=0.

Request Lifecycle

Client Request → Authentication → Authorization → Admission Control → Validation → etcd

1. Authentication

Who are you? Client certs, bearer tokens, OIDC, webhook — at least one must succeed.

2. Authorization

Are you allowed? RBAC, Node, Webhook, ABAC — checked in order until one allows or all deny.

3. Admission

Should we modify or reject? Mutating then validating webhooks, plus built-in plugins.

Walk through a kubectl create pod request. Show how each stage can reject. Emphasize that authorization is not RBAC alone — the chain is configurable.

How kubectl Talks to the API Server

# ~/.kube/config (simplified)
apiVersion: v1
kind: Config
clusters:
- cluster:
    server: https://api.mycluster.example.com:6443
    certificate-authority-data: LS0tLS1CRUdJ...   # CA cert
  name: production
users:
- name: admin
  user:
    client-certificate-data: LS0tLS1CRUdJ...      # Client cert
    client-key-data: LS0tLS1CRUdJ...               # Client key
contexts:
- context:
    cluster: production
    user: admin
    namespace: default
  name: admin@production
current-context: admin@production
Key Point: The kubeconfig establishes mutual TLS — the client verifies the server CA, and the server verifies the client certificate.

🔑 Authentication Configuration

Verifying identity at the cluster gate

Authentication Methods Overview

Method Mechanism Production Use
X.509 Client Certificates TLS mutual auth ✓ Recommended
OIDC Tokens Identity provider integration ✓ Recommended
Webhook Token Auth External token verification ✓ Recommended
Bootstrap Tokens Cluster bootstrapping ⚠ Limited scope only
Static Token File CSV file on disk ✘ Never in production
Static Password File CSV file on disk ✘ Removed in v1.19
Multiple authenticators run simultaneously. The first one that can parse the credential returns the identity. If none succeed, the request is rejected as 401 Unauthorized.

X.509 Client Certificates

The most common method for system components and admin access.

# Generate a client certificate
openssl genrsa -out jane.key 2048

# Create a CSR with user/group information embedded
openssl req -new -key jane.key -out jane.csr \
  -subj "/CN=jane/O=developers/O=qa"
# CN = username, O = groups

# Sign with the cluster CA
openssl x509 -req -in jane.csr -out jane.crt \
  -CA /etc/kubernetes/pki/ca.crt \
  -CAkey /etc/kubernetes/pki/ca.key \
  -CAcreateserial -days 365
⚠ Limitation: Certificates cannot be revoked individually. The only option is to rotate the entire CA — a disruptive operation. Prefer OIDC for human users.
This is why many organizations use certificates only for system components and kubeadm bootstrapping, then switch human users to OIDC or webhook-based auth.

Static Token Files — Why Not

# API server flag
--token-auth-file=/etc/kubernetes/tokens.csv

# tokens.csv format:
# token,user,uid,"groups"
abc123,admin,1,"system:masters"
def456,developer,2,"developers"

Security Risks

  • Tokens stored in plaintext on disk
  • No expiration mechanism
  • Requires API server restart to update
  • No audit trail for token creation
  • Cannot be rotated without downtime
CIS Benchmark 1.2.2: Ensure that the --token-auth-file parameter is not set. This is a scored check.

Webhook Token Authentication

Delegate authentication to an external service.

# webhook-config.yaml
apiVersion: v1
kind: Config
clusters:
- name: authn-service
  cluster:
    server: https://authn.example.com/authenticate
    certificate-authority: /etc/kubernetes/pki/webhook-ca.crt
users:
- name: kube-apiserver
  user:
    client-certificate: /etc/kubernetes/pki/webhook-client.crt
    client-key: /etc/kubernetes/pki/webhook-client.key
current-context: webhook
contexts:
- context:
    cluster: authn-service
    user: kube-apiserver
  name: webhook
# API server flag
--authentication-token-webhook-config-file=/etc/kubernetes/webhook-config.yaml
--authentication-token-webhook-cache-ttl=2m  # Cache valid responses
Webhook auth is powerful for integration with existing enterprise identity systems. The external service receives the bearer token and returns user info or a rejection.

Bootstrap Tokens

Used by kubeadm to join new nodes to the cluster.

# Bootstrap token secret
apiVersion: v1
kind: Secret
metadata:
  name: bootstrap-token-07401b
  namespace: kube-system
type: bootstrap.kubernetes.io/token
stringData:
  token-id: "07401b"
  token-secret: "f395accd246ae52d"
  usage-bootstrap-authentication: "true"
  usage-bootstrap-signing: "true"
  expiration: "2026-04-01T00:00:00Z"

Security Properties

  • Time-limited (TTL-based expiration)
  • Scoped to node bootstrapping
  • Stored as Secrets in kube-system
  • Can be revoked by deleting the Secret
Best Practice: Set short TTLs and delete bootstrap token secrets after node join completes.

🛡 Authorization Modes

Controlling who can do what

Authorization Mode Chain

# API server flag — order matters!
--authorization-mode=Node,RBAC
How It Works: Each authorizer is checked in order. If one allows, the request proceeds. If one denies, the next authorizer is checked. If all deny, the request is rejected.
Mode Description Status
RBAC Role-based access control via Roles/ClusterRoles Standard
Node Restricts kubelet access to its own node resources Standard
Webhook External HTTP(S) authorization service Specialized
ABAC Attribute-based policies from a JSON file Legacy
AlwaysAllow Permits all requests Never in prod
AlwaysDeny Blocks all requests Testing only
The recommended production configuration is Node,RBAC. Node must come first so kubelet requests are handled efficiently before falling through to the full RBAC evaluation.

Node Authorization

Purpose

Restricts kubelets to only access resources related to pods scheduled on their node. Prevents a compromised node from reading secrets belonging to pods on other nodes.

Requires: The NodeRestriction admission plugin must also be enabled to prevent kubelets from modifying their own Node object labels with the node-restriction.kubernetes.io/ prefix.

Kubelet Can Access

  • Read Secrets for pods on its node
  • Read ConfigMaps for pods on its node
  • Read PV/PVCs for pods on its node
  • Write Node status for its own node
  • Create/update pod status

Cannot Access

  • Secrets for pods on other nodes
  • Other nodes' status
  • Cluster-scoped admin operations

RBAC Recap & Webhook Authorization

RBAC (Covered in Module 2)

  • Roles / ClusterRoles define permissions
  • RoleBindings / ClusterRoleBindings grant them
  • Deny-by-default — no implicit permissions
  • Namespace-scoped and cluster-scoped variants

Webhook Authorization

  • Sends SubjectAccessReview to an external URL
  • Enables integration with external policy engines
  • Can implement custom authorization logic
  • Used by OPA/Gatekeeper, Kyverno, etc.
# API server flag
--authorization-mode=Node,RBAC,Webhook
--authorization-webhook-config-file=\
  /etc/kubernetes/webhook-authz.yaml

ABAC — Legacy Mode

// abac-policy.json — one rule per line
{"apiVersion": "abac.authorization.kubernetes.io/v1beta1", "kind": "Policy",
 "spec": {"user": "jane", "namespace": "dev", "resource": "pods", "readonly": true}}
{"apiVersion": "abac.authorization.kubernetes.io/v1beta1", "kind": "Policy",
 "spec": {"user": "bob", "namespace": "*", "resource": "*", "readonly": false}}

Why ABAC Is Discouraged

  • Requires API server restart for policy changes
  • No dynamic management via the API
  • Difficult to audit and version control
  • No namespace-level delegation

Migration Path

  • Convert ABAC policies to RBAC Roles
  • Run both modes during transition: --authorization-mode=ABAC,RBAC
  • Monitor audit logs for ABAC-only grants
  • Remove ABAC once RBAC covers all cases

🔒 Securing the Endpoint

TLS, network restrictions, and certificate management

API Server TLS Configuration

# Essential TLS flags for kube-apiserver
--tls-cert-file=/etc/kubernetes/pki/apiserver.crt
--tls-private-key-file=/etc/kubernetes/pki/apiserver.key
--client-ca-file=/etc/kubernetes/pki/ca.crt

# TLS cipher suite hardening
--tls-min-version=VersionTLS12
--tls-cipher-suites=TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256,\
  TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384,\
  TLS_ECDHE_ECDSA_WITH_AES_128_GCM_SHA256,\
  TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384

# Kubelet communication
--kubelet-client-certificate=/etc/kubernetes/pki/apiserver-kubelet-client.crt
--kubelet-client-key=/etc/kubernetes/pki/apiserver-kubelet-client.key
CIS Benchmark: Verify TLS 1.2+ is enforced. TLS 1.0 and 1.1 have known vulnerabilities and must be disabled.
With kubeadm, these are set by default. On manually bootstrapped clusters, every flag must be verified. Use kube-bench to automate the check.

Certificate Rotation

kubeadm Certificate Management

# Check certificate expiration
kubeadm certs check-expiration

# Renew all certificates
kubeadm certs renew all

# Renew specific certificate
kubeadm certs renew apiserver

# Auto-renewal via kubeadm upgrade
# Certificates renew on every
# control plane upgrade

Default Lifetime

kubeadm certificates expire after 1 year. The CA certificate expires after 10 years.

Common Outage: Expired certificates are one of the most frequent causes of cluster outages. Automate monitoring of certificate expiration dates.

Restricting Network Access

Network Level

  • Firewall rules: only allow known CIDR blocks
  • VPN or bastion host for admin access
  • Separate management network for control plane

Cloud Provider

  • Private clusters (no public endpoint)
  • Authorized networks / IP whitelisting
  • VPC peering for cross-account access

Kubernetes Level

  • --advertise-address on private IP
  • --bind-address restrict listening interface
  • API server audit logging
The API server should never be directly reachable from the public internet. Use authorized network ranges or a private endpoint.

kubeadm Configuration Patch

Harden the API server at cluster creation time:

# kubeadm-config.yaml
apiVersion: kubeadm.k8s.io/v1beta3
kind: ClusterConfiguration
apiServer:
  extraArgs:
    anonymous-auth: "false"
    authorization-mode: "Node,RBAC"
    tls-min-version: "VersionTLS12"
    enable-admission-plugins: >-
      NodeRestriction,PodSecurity,EventRateLimit
    audit-log-path: "/var/log/kubernetes/audit.log"
    audit-policy-file: "/etc/kubernetes/audit-policy.yaml"
    # ... additional flags (log rotation, volumes, etc.)
This is the single most important configuration file for hardening a kubeadm cluster. Every flag here maps directly to a CIS Benchmark check.

Audit Logging Configuration

# /etc/kubernetes/audit-policy.yaml
apiVersion: audit.k8s.io/v1
kind: Policy
rules:
  # Log no requests to the RequestReceived stage
  - level: None
    stage:
      - "RequestReceived"
  # Log secret access at Metadata level (do not log secret data)
  - level: Metadata
    resources:
      - group: ""
        resources: ["secrets", "configmaps"]
  # Log changes to core resources at RequestResponse level
  - level: RequestResponse
    resources:
      - group: ""
        resources: ["pods", "services", "namespaces"]
    verbs: ["create", "update", "patch", "delete"]
  # Log everything else at Metadata level
  - level: Metadata
    omitStages:
      - "RequestReceived"
Best Practice: Never log secret data (use Metadata level for secrets). Ship audit logs to a SIEM for analysis and alerting.

⚙️ Kubelet Security

Hardening the node agent

Kubelet Attack Surface

The kubelet runs on every node and has the ability to create, modify, and delete pods. A compromised kubelet means full control of the node and access to all pod secrets on that node.

Kubelet API Endpoints

  • /pods — list running pods
  • /run — execute commands in containers
  • /exec — stream exec sessions
  • /logs — read container logs
  • /metrics — node and pod metrics

Default Risks

  • Anonymous authentication enabled by default
  • Read-only port (10255) exposes pod info
  • No authorization by default (AlwaysAllow)
  • Kubelet API accessible on port 10250
This is a frequently exploited attack vector. Show a curl to the kubelet API at port 10250 with anonymous auth — it returns all running pods and can execute arbitrary commands.

Hardening the Kubelet

# /var/lib/kubelet/config.yaml
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
# Disable anonymous access
authentication:
  anonymous:
    enabled: false
  webhook:
    enabled: true        # Use API server for authn
    cacheTTL: "2m0s"
  x509:
    clientCAFile: "/etc/kubernetes/pki/ca.crt"
# Enable webhook authorization
authorization:
  mode: Webhook          # NOT AlwaysAllow
  webhook:
    cacheAuthorizedTTL: "5m0s"
    cacheUnauthorizedTTL: "30s"
# Disable the read-only port
readOnlyPort: 0          # Default is 10255
# Protect kernel defaults
protectKernelDefaults: true
# Enable certificate rotation
rotateCertificates: true
serverTLSBootstrap: true
Walk through each setting. The webhook authorization mode means the kubelet asks the API server whether a given request is authorized — enforcing RBAC at the node level.

Kubelet TLS Bootstrapping

Problem

New nodes need TLS certificates to communicate securely with the API server, but they do not have certificates yet when they first join.

Bootstrap Flow

  1. Node uses a bootstrap token for initial auth
  2. Kubelet generates a key pair and CSR
  3. CSR sent to the API server
  4. CSR approved (manually or auto-approved)
  5. Signed certificate returned to kubelet
  6. Kubelet uses the certificate for future requests
# View pending CSRs
kubectl get csr

# Approve a CSR
kubectl certificate approve \
  node-csr-abc123

# kubelet flags
--bootstrap-kubeconfig=\
  /etc/kubernetes/bootstrap-kubelet.conf
--rotate-certificates=true
--rotate-server-certificates=true

Kubelet Certificate Rotation

Client Certificate Rotation

Enabled via rotateCertificates: true in the kubelet config. The kubelet automatically requests new certificates before the current one expires.

Default since v1.19: Enabled automatically by kubeadm.

Server Certificate Rotation

Controlled by RotateKubeletServerCertificate feature gate and serverTLSBootstrap: true.

Note: Server certificate CSRs require manual or automated approval. Consider using a CSR signing controller. 
# Verify kubelet certificate rotation is working
journalctl -u kubelet | grep -i "certificate rotation"

# Check current kubelet serving certificate
openssl x509 -in /var/lib/kubelet/pki/kubelet.crt -noout -dates

Kubelet Security Checklist

  • Disable anonymous authentication (authentication.anonymous.enabled: false)
  • Enable webhook authorization (authorization.mode: Webhook)
  • Disable the read-only port (readOnlyPort: 0)
  • Enable TLS bootstrapping and certificate rotation
  • Enable RotateKubeletServerCertificate
  • Set protectKernelDefaults: true
  • Enable the NodeRestriction admission plugin on the API server
  • Restrict network access to kubelet port 10250

👑 etcd Security

Protecting the crown jewels

Why etcd Is the Crown Jewel

etcd stores all cluster state — every Secret, ConfigMap, RBAC policy, pod spec, and service account token. Direct access to etcd bypasses all Kubernetes authentication and authorization.

Impact of Compromise

  • Read all secrets in plaintext
  • Modify RBAC to grant admin access
  • Inject malicious workloads
  • Destroy cluster state entirely

Common Mistakes

  • etcd exposed without TLS
  • Shared network with worker nodes
  • No encryption at rest configured
  • Backups stored unencrypted

Defense Strategy

  • mTLS for all etcd communication
  • Dedicated network segment
  • Encryption at rest
  • Encrypted, access-controlled backups
If an attacker can reach etcd directly, they own the cluster. No RBAC, no admission control, nothing stands between them and the data. This is why etcd must be on a separate network segment.

etcd TLS Configuration

# etcd configuration (via kubeadm or etcd config file)
# Peer communication (etcd node to etcd node)
peer-transport-security:
  cert-file: /etc/kubernetes/pki/etcd/peer.crt
  key-file: /etc/kubernetes/pki/etcd/peer.key
  client-cert-auth: true
  trusted-ca-file: /etc/kubernetes/pki/etcd/ca.crt

# Client communication (API server to etcd)
client-transport-security:
  cert-file: /etc/kubernetes/pki/etcd/server.crt
  key-file: /etc/kubernetes/pki/etcd/server.key
  client-cert-auth: true
  trusted-ca-file: /etc/kubernetes/pki/etcd/ca.crt
# Corresponding API server flags
--etcd-servers=https://127.0.0.1:2379
--etcd-cafile=/etc/kubernetes/pki/etcd/ca.crt
--etcd-certfile=/etc/kubernetes/pki/apiserver-etcd-client.crt
--etcd-keyfile=/etc/kubernetes/pki/apiserver-etcd-client.key
Note the separate CA for etcd. This is a best practice — the etcd CA is distinct from the Kubernetes cluster CA, providing defense in depth.

Encryption at Rest

# /etc/kubernetes/enc/encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets
      - configmaps
    providers:
      # Preferred provider (used for new writes)
      - aescbc:
          keys:
            - name: key1
              secret: c2VjcmV0LWtleS0xLWJhc2U2NC1lbmNvZGVk  # base64
      # Fallback for reading existing unencrypted data
      - identity: {}
# API server flag
--encryption-provider-config=/etc/kubernetes/enc/encryption-config.yaml

# After enabling, re-encrypt all existing secrets
kubectl get secrets --all-namespaces -o json | \
  kubectl replace -f -
Key Management: The encryption key is stored in a file on the control plane node. For production, use a KMS provider (AWS KMS, Azure Key Vault, GCP Cloud KMS, HashiCorp Vault).

KMS Provider Integration

# EncryptionConfiguration with KMS v2
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets
    providers:
      - kms:
          apiVersion: v2
          name: my-kms-provider
          endpoint: unix:///var/run/kmsplugin/socket.sock
          timeout: 3s
      - identity: {}

KMS Advantages

  • Key never leaves the KMS
  • Key rotation without re-encrypting data
  • Hardware security module (HSM) backing
  • Centralized key management and audit trail

Supported KMS Providers

  • AWS KMS via aws-encryption-provider
  • Azure Key Vault via k8s-azure-kms
  • GCP Cloud KMS (built into GKE)
  • HashiCorp Vault via plugin

etcd Backup Security

# Take an etcd snapshot
ETCDCTL_API=3 etcdctl snapshot save \
  /backup/etcd-$(date +%Y%m%d).db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Encrypt the backup
gpg --symmetric --cipher-algo AES256 \
  /backup/etcd-20260313.db

# Verify snapshot integrity
ETCDCTL_API=3 etcdctl snapshot status \
  /backup/etcd-20260313.db --write-out=table

Backup Risks

  • Unencrypted backups contain all secrets
  • Backups stored with loose file permissions
  • No access controls on backup storage
  • Backup retention without rotation
Best Practice: Encrypt backups at rest, transmit over TLS, store with strict access controls, and test restore procedures regularly.

⚡ API Rate Limiting & Availability

Protecting the API server from overload

API Priority and Fairness (APF)

Replaced --max-requests-inflight: APF provides fine-grained, priority-based request management. Enabled by default since Kubernetes v1.20.

FlowSchema

Matches incoming requests by user, group, namespace, verb, or resource, and assigns them to a PriorityLevelConfiguration.

PriorityLevelConfiguration

Defines the concurrency level and queuing behavior for a class of requests. Higher-priority traffic is served first.

Request → FlowSchema Match → Priority Level → Queue → Execute or Reject (429)

APF replaces the older max-requests-inflight flag with a much more nuanced system. The key insight is that not all API requests are equal — a node heartbeat is more important than a list-all-pods request from a dashboard.

API Priority — FlowSchema

# Protect critical system requests
apiVersion: flowcontrol.apiserver.k8s.io/v1
kind: FlowSchema
metadata:
  name: system-critical
spec:
  priorityLevelConfiguration:
    name: system
  matchingPrecedence: 100
  rules:
    - subjects:
        - kind: Group
          group:
            name: "system:masters"
      resourceRules:
        - verbs: ["*"]
          apiGroups: ["*"]
          resources: ["*"]
          namespaces: ["*"]

Matches incoming requests by user, group, namespace, verb, or resource and assigns them to a PriorityLevelConfiguration.

API Priority — PriorityLevelConfiguration

# Limit noisy controllers
apiVersion: flowcontrol.apiserver.k8s.io/v1
kind: PriorityLevelConfiguration
metadata:
  name: batch-controllers
spec:
  type: Limited
  limited:
    nominalConcurrencyShares: 10
    lendablePercent: 50
    limitResponse:
      type: Queue
      queuing:
        queues: 64
        handSize: 6
        queueLengthLimit: 50

Defines the concurrency level and queuing behavior for a class of requests. Higher-priority traffic is served first.

EventRateLimit Admission Plugin

# /etc/kubernetes/eventconfig.yaml
apiVersion: eventratelimit.admission.k8s.io/v1alpha1
kind: Configuration
limits:
  - type: Namespace
    qps: 50
    burst: 100
    cacheSize: 2000
  - type: User
    qps: 10
    burst: 50
  - type: Server
    qps: 500
    burst: 1500
# Enable the admission plugin
--enable-admission-plugins=...,EventRateLimit
--admission-control-config-file=/etc/kubernetes/eventconfig.yaml
Use Case: Prevents a single namespace, user, or source IP from flooding the API server with events — a common pattern in misconfigured controllers or deliberate DoS attacks.

Protecting Against DoS

API Server Level

  • APF with proper priority levels
  • EventRateLimit admission plugin
  • --request-timeout (default 60s)
  • Audit logging for detection

Network Level

  • Load balancer rate limiting
  • Cloud WAF / DDoS protection
  • Network-level IP blocking
  • Connection limits per source

Operational

  • HA control plane (3+ API servers)
  • Watch-based controllers vs. polling
  • Client-side rate limiting in SDKs
  • Monitor apiserver_request_total metrics
Combine multiple layers. APF alone is not sufficient — you also need network-level controls and application-level best practices in your controllers.

Monitoring API Server Health

# Key Prometheus metrics to monitor
apiserver_request_total                    # Total request count by verb/resource/code
apiserver_request_duration_seconds         # Request latency histogram
apiserver_current_inflight_requests        # Current in-flight requests
apiserver_flowcontrol_rejected_requests_total  # APF rejections
apiserver_flowcontrol_dispatched_requests_total # APF dispatches
etcd_request_duration_seconds              # etcd latency (indicates backend pressure)

# Example alert rule (Prometheus)
# Alert if API server error rate exceeds 1% for 5 minutes
- alert: KubeAPIServerHighErrorRate
  expr: |
    sum(rate(apiserver_request_total{code=~"5.."}[5m]))
    / sum(rate(apiserver_request_total[5m])) > 0.01
  for: 5m
  labels:
    severity: critical
Key Metric: Watch apiserver_flowcontrol_rejected_requests_total to detect when APF is throttling legitimate traffic — this may indicate your priority levels need tuning.

Module 3 — Key Takeaways

  • The API server is the single point of policy enforcement — secure it first
  • Use X.509 certificates or OIDC for authentication; never use static token files
  • Set --authorization-mode=Node,RBAC and enable the NodeRestriction admission plugin
  • Enforce TLS 1.2+ and harden cipher suites
  • Disable kubelet anonymous auth, enable webhook authorization, disable the read-only port
  • Treat etcd as the crown jewel — mTLS, separate network, encryption at rest
  • Use API Priority and Fairness to protect against overload and DoS
  • Monitor API server metrics and ship audit logs to a SIEM
